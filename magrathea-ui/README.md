# Magrathea frontend workspace

This npm workspace separates reusable frontend infrastructure from product-owned code:

- `packages/product-shell`: publishable, product-neutral shell contracts and future shared implementation.
- `packages/object-storage-extension`: Object Storage-owned screens and integrations.
- `apps/object-storage-admin`: the deployable Object Storage composition, emitted to `dist/object-storage`.
- `apps/magrathea-example`: a minimal second deployable composition, emitted to `dist/magrathea-example`.
- `templates/product-app`: a product-neutral consumer template and deterministic build fixture.

Each app owns a `product.manifest.json`. The canonical build verifies different extension, route, and localization registrations, then records the version and content digest of the single built `@magrathea/product-shell` artifact in both distributions. Shell source is imported from the workspace package and is never copied into an app.

## Commands

Run from `magrathea-ui`:

```bash
npm ci
npm run lint
npm run typecheck
npm test
npm run validate:template
npm run build
npm run validate:reproducible-packaging
npm run test:e2e
```

## Real-browser acceptance-test infrastructure

Playwright serves the built Object Storage Admin distribution on the fixed loopback URL `http://127.0.0.1:4173` and runs Chromium projects at 360, 768, and 1440 CSS-pixel widths. The reusable fixtures in `e2e/fixtures/` intercept Admin API requests and can enable a separately configured S3 HeadObject endpoint. The shared `axe` fixture exposes `@axe-core/playwright` to acceptance cases.

- `npm run test:e2e` builds and runs the headless browser suite locally.
- `npm run test:e2e:ci` is the deterministic CI entry point.
- `npm run test:e2e:ui` builds and opens Playwright UI mode.
- `npm run test:e2e:runner -- --project chromium-360` reruns against an existing build and selects one viewport.

The committed smoke is infrastructure evidence only. Browser-dependent EP-7 acceptance requirements remain pending until the frontend acceptance scenarios owned by the frontend test suite validate them.

`npm run validate` runs all frontend checks. `validate:reproducible-packaging` performs two clean product and documentation builds and compares a normalized, path-sorted SHA-256 inventory of every generated byte. Generated `dist` and bootstrap static resources remain ignored and are cleaned after the gate. Docker target `frontend-packaging-validation` is the canonical packaging validation path.
