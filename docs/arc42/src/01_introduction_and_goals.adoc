== Introduction and Goals

Magrathea ObjectStore implements an **AWS S3-compatible object storage** route surface built with Spring Boot 4 WebFlux and Java 21. It is evolving from a reference DDD/reactive implementation toward an enterprise-ready, market-leading S3-compatible object store (see `PLAN.md`, Enterprise Production Readiness Plan EP-0..EP-11 and Killer-App Track KA-1..KA-6).

It provides an S3-compatible REST API for object CRUD, bucket configuration, and multipart upload, plus a separate internal/admin JSON API for operational configuration. Storage Engine is the default and only supported single-node product backend, with filesystem-durable metadata and content. Legacy in-memory adapters are test/retirement scaffolding and must not be presented as a deployable single-node backend. AWS CLI parity has improved for canonical object CRUD basics and metadata durability; remaining semantic and production-hardening scope is tracked explicitly.

=== Goals

* **S3 API compatibility**: map the in-scope AWS S3 REST API route surface and classify semantic implementation separately from route count.
* **Reactive by design**: full WebFlux non-blocking I/O, `DataBuffer` streaming for binary content.
* **Domain-Driven Design**: pure domain with zero framework dependencies, hexagonal architecture, explicit bounded contexts (Object Store, Storage Engine).
* **Durability by default**: the Storage Engine backend persists bucket registry, multipart upload state, per-object configuration, and bucket configuration families as crash-safe filesystem documents (temp-file + atomic-rename commits) that survive a full process restart.
* **Testability**: domain/unit tests, WebTestClient Cucumber, targeted AWS CLI compatibility, explicit gap tags (`@implemented-and-validated`, `@implemented-not-e2e-validated`, `@partial`, `@config-only`, `@absent`, `@not-implemented`) when scenarios remain unsupported, and JaCoCo coverage reporting.
* **Evidence-based completion**: every completion claim is backed by production code plus semantic tests; see `docs/test-report.md` for current validation evidence and the generated Gherkin requirements appendix for per-scenario status.

=== Motivation

The project serves as a reference implementation of Java DDD + Spring Boot 4 reactive patterns with hexagonal architecture, and is evolving toward production-quality, market-differentiated S3-compatible storage through evidence-based semantic coverage. The roadmap (`PLAN.md`, `docs/roadmap.md`) sequences durability (EP-2), streaming completeness (EP-3), security/identity (EP-1), space management (EP-4), operability (EP-5), and further phases before broader production claims, alongside a Killer-App Track (positioning, ecosystem conformance, data-lake readiness, distribution) documented in `docs/positioning.md`.
