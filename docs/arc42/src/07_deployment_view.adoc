== Deployment View

=== Development Environment

**Single node, running locally.**

| Element | Description |
|---|---|
| Runtime JAR | `bootstrap-application-1.0.0-SNAPSHOT.jar` running on the configured HTTP port |
| Static resources | Generated Vue/admin documentation assets, ARC42 JSON, ADR JSON, JaCoCo JSON exports, C4 PNGs, and ARC42 images served from classpath `static/**` |
| Default file store | In-memory (ConcurrentHashMap), allocated in the same JVM heap as the server |
| Java Runtime | JDK 21 |
| Build Tool | Apache Maven 3.9+ |

=== Docker Deployment

A `Dockerfile` and `docker-compose.yml` are present at the project root for containerized execution:

**Dockerfile:**
- Builder stage: Maven 3.9 + Eclipse Temurin 21 with Node.js installed for documentation/frontend generation
- Regenerates web documentation and admin UI static assets inside the builder stage from source-controlled docs and `magrathea-ui` sources
- Does not copy host-generated `bootstrap-application/src/main/resources/static/**` assets; `.dockerignore` excludes those generated resources from the build context
- Runtime stage: Eclipse Temurin 21 JRE, copies the packaged Spring Boot JAR from the builder
- Exposes ports 8080 and 8081
- Entrypoint: `java -jar /app.jar`

**docker-compose.yml:**
- Single service `magrathea-s3` on port `8080:8080`
- Environment variables for configuration

=== Production Deployment (Future)

The two-container architecture (`s3-server` + `file-store`) is intentionally monolithic for single-node operation. In the future:

- `file-store` can be replaced by R2DBC + PostgreSQL for relational persistence
- `file-store` can become a separate container for horizontal scaling
- `s3-server` can be clustered with session affinity
- Storage Engine deployment uses `storage-engine-reactive-application`, `storage-engine-reactive-infrastructure`, and the ACL adapter. Phase 5 verifies policy/catalog planning; end-to-end runtime read/write remains a pending gate before production claims.
