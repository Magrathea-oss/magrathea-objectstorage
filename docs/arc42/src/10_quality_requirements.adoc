=== Quality Scenarios

==== Scenario 1 — Object Upload Performance

- **Context:** User uploads large files (100+ MB)
- **Stimulus:** PUT /{bucket}/{key} with binary body
- **Response:** Upload completes without OutOfMemoryError
- **Metric:** Throughput > 50 MB/s, peak memory < 256 MB

==== Scenario 2 — S3 API Compatibility

- **Context:** AWS CLI interacts with the system
- **Stimulus:** AWS CLI command for each implemented operation
- **Response:** Output identical to real AWS S3 (same XML, same error codes)
- **Metric:** route mapped plus semantic classification (`Mapped`, `Stubbed`, `Stateful`, `AWS CLI compatible`, `Storage-engine compatible`, or `Semantically S3-compatible`) with evidence in the test report

==== Scenario 3 — Test Coverage

- **Context:** A new S3 operation is implemented
- **Stimulus:** Code commit
- **Response:** Maven verification generates JaCoCo coverage as the current baseline
- **Metric:** Coverage trend and thresholds are taken from current JaCoCo reports; Clover/OpenClover is optional/legacy

==== Scenario 4 — Restart Safety (Storage Engine backend)

- **Context:** Storage-engine profile is active; buckets, multipart uploads, and per-object/bucket configuration have been created
- **Stimulus:** The application process is stopped and started again against the same filesystem root
- **Response:** The bucket registry, multipart upload sessions, and per-object/bucket configuration are reloaded from durable filesystem documents; no state is lost or torn
- **Metric:** `MetadataDurabilityRestartTest` (restart-simulation unit tests) and `StorageEngineRestartSafetyTest` (real Spring-context restart) both pass; see `docs/test-report.md` for current run evidence

==== Scenario 5 — Product Extension Isolation and Accessibility

- **Context:** An operator uses the Object Storage Admin Application or a maintainer removes/adds a Product Extension
- **Stimulus:** Navigate by keyboard and direct URL at supported widths, or remove an extension from the product manifest
- **Response:** Navigation and content remain accessible; removal eliminates extension routes, navigation, permissions, localization, and screens without modifying Product Shell source
- **Metric:** 72/72 Vitest tests and 39/39 Playwright/axe tests across 360/768/1440-pixel Chromium viewports; deterministic extension-removal validation passes

==== Scenario 6 — Truthful Operational Unavailability

- **Context:** No real recovery, garbage-collection, scrub, audit, metrics, or traces report provider is configured
- **Stimulus:** The Admin UI requests an operational report
- **Response:** The Admin API returns HTTP 503 `report-provider-not-configured` with `availability: not-configured`; the UI presents an unavailable state and fabricates no records
- **Metric:** EP-7 Admin API Cucumber executes the six provider-unavailable examples within its 18 passed scenarios / 132 steps
