=== Quality Scenarios

==== Scenario 1 — Object Upload Performance

- **Context:** User uploads large files (100+ MB)
- **Stimulus:** PUT /{bucket}/{key} with binary body
- **Response:** Upload completes without OutOfMemoryError
- **Metric:** Throughput > 50 MB/s, peak memory < 256 MB

==== Scenario 2 — S3 API Compatibility

- **Context:** AWS CLI interacts with the system
- **Stimulus:** AWS CLI command for each implemented operation
- **Response:** Output identical to real AWS S3 (same XML, same error codes)
- **Metric:** route mapped plus semantic classification (`Mapped`, `Stubbed`, `Stateful`, `AWS CLI compatible`, `Storage-engine compatible`, or `Semantically S3-compatible`) with evidence in the test report

==== Scenario 3 — Test Coverage

- **Context:** A new S3 operation is implemented
- **Stimulus:** Code commit
- **Response:** Maven verification generates JaCoCo coverage as the current baseline
- **Metric:** Coverage trend and thresholds are taken from current JaCoCo reports; Clover/OpenClover is optional/legacy

==== Scenario 4 — Reactor-wide Coverage Evidence (`REQ-QUAL-001`, `REQ-QUAL-002`)

- **Status:** `implementation-informed`, `@partial`
- **Context:** The production Maven reactor and cross-module Cucumber runners have completed
- **Stimulus:** Run `mvn -B -Pcoverage clean verify --no-transfer-progress`
- **Response:** One aggregate HTML/XML/CSV report and one threshold-free JSON summary preserve covered and missed counts for all production modules
- **Observed metric (2026-07-12, revision `c3fe8bb1`):** 48,191/60,631 instructions (79.48%), 1,957/3,484 branches (56.17%), 8,868/11,367 lines (78.02%), 2,671/3,316 methods (80.55%), and 570/635 classes (89.76%). Lowest instruction coverage: `object-store-reactive-infrastructure` 20.01%, `bootstrap-application` 61.93%, and `object-store-reactive-application` 64.38%. No threshold is enforced and no sufficiency claim is made.

==== Scenario 5 — Fail-closed Dependency Vulnerability Evidence (`REQ-QUAL-003`, `REQ-QUAL-004`)

- **Status:** `implementation-informed`, `@partial`, `needs-review`
- **Context:** OWASP Dependency-Check scans resolved production-reactor dependencies under the CVSS 7.0 policy
- **Stimulus:** Run `NVD_API_KEY="$(< /secure/path/nvd-api-key)" scripts/run-dependency-check.sh`, using an access-restricted, non-repository key file or equivalent CI secret injection
- **Response:** A complete scan publishes `target/dependency-check-report.json` and `.html`; normalized status is written to `target/site/dependency-check-analysis.json`. An incomplete scan fails and emits `target/dependency-check-scan-error.json` plus `target/dependency-check-scan.log` without fabricating clean evidence.
- **Observed metric (2026-07-13, historical EP-8 clean revision `209b3170b64d3311ba9b773fdb7bd5581519e682`):** Dependency-Check 12.2.2 downloaded 190,000/364,891 NVD records before HTTP 429; `scanStatus=error`, `clean=null`, 8 scan errors. The outcome is monitoring-only `unknown/error`: there is no vulnerability-findings, clean-scan, or zero-vulnerability claim and no EP-8 remediation was performed. The later dirty-working-tree root test did not refresh this assessment.

==== Scenario 6 — Restart Safety (Storage Engine backend)

- **Context:** Storage-engine profile is active; buckets, multipart uploads, and per-object/bucket configuration have been created
- **Stimulus:** The application process is stopped and started again against the same filesystem root
- **Response:** The bucket registry, multipart upload sessions, and per-object/bucket configuration are reloaded from durable filesystem documents; no state is lost or torn
- **Metric:** `MetadataDurabilityRestartTest` (restart-simulation unit tests) and `StorageEngineRestartSafetyTest` (real Spring-context restart) both pass; see `docs/test-report.md` for current run evidence

==== Scenario 7 — Product Extension Isolation and Accessibility

- **Context:** An operator uses the Object Storage Admin Application or a maintainer removes/adds a Product Extension
- **Stimulus:** Navigate by keyboard and direct URL at supported widths, or remove an extension from the product manifest
- **Response:** Navigation and content remain accessible; removal eliminates extension routes, navigation, permissions, localization, and screens without modifying Product Shell source
- **Metric (2026-07-12):** 86/86 Vitest tests across 14 files; 57/57 Playwright Chromium tests (19 at each 360/768/1440-pixel viewport), including axe 3/3 with zero violations and 12/12 Linux Chromium visual baselines; template and deterministic extension-removal validation, six-workspace typecheck, ESLint, and the five-package/application build pass

==== Scenario 8 — Truthful Operational Unavailability

- **Context:** No real recovery, garbage-collection, scrub, audit, metrics, or traces report provider is configured
- **Stimulus:** The Admin UI requests an operational report
- **Response:** The Admin API returns HTTP 503 `report-provider-not-configured` with `availability: not-configured`; the UI names the unavailable evidence, explains operational impact, offers only valid documentation/configuration/retry/navigation next steps, and enables no missing operation or fabricated record
- **Metric:** EP-7 Admin API Cucumber executes the six provider-unavailable examples within its 18 passed scenarios / 132 steps; Playwright validates REQ-ADMIN-037 across all three viewports

==== Scenario 9 — Task-oriented Administration UX (`REQ-ADMIN-032..036`)

- **Status:** `implementation-informed`, `@implemented-and-validated`
- **Context:** An operator assesses a degraded service, inspects backend evidence, validates a non-persistent policy proposal, or changes appearance
- **Stimulus:** Use grouped navigation and the priority dashboard, reveal supporting backend detail, submit validation, or select `system`, `light`, or `dark`
- **Response:** Readiness, selected backend, and attention conditions precede supporting detail; a degraded device links to its detail; navigation and quick tasks expose only valid Admin Control Plane work; collapsed detail never hides critical conditions; validation prevents duplicates and preserves truthful pending/success/failure semantics; appearance persists and system mode follows operating-system changes
- **Metric (2026-07-12):** REQ-ADMIN-032..036 pass in the 57-test Playwright matrix and supporting Vitest suite; 12 visual baselines cover dashboard, collapsed backend attention, unavailable reports, and validation success at 360/768/1440 pixels

==== Scenario 10 — Live Admin API Browser Boundary

- **Status:** `implementation-informed`, bounded evidence
- **Context:** The built browser UI is connected through transport-only routing, without JSON fixtures, to a bare single-node `bootstrap-application` Admin API on port 8081
- **Stimulus:** Load dashboard/data-hygiene evidence
- **Response:** The browser observes `200 /admin/health`, `200 /admin/live`, `200 /admin/backend-status`, `503 /admin/ready`, `503 /admin/storage-devices`, `503 /admin/reports/recovery`, `503 /admin/reports/garbage-collection`, and `503 /admin/reports/scrub`; the UI renders selected `in-memory`, unavailable dashboard state, and three provider-not-configured report states
- **Limitation:** This runtime had no YAML catalogs. Configured `storage-engine` dashboard priority and degraded-device linking are deterministic browser-fixture evidence, not live evidence from this run. This scenario does not establish any missing operational provider.

==== Scenario 11 — Fixed-Slice Quorum Write Safety

- **Status:** `implementation-informed`, `@implemented-and-validated` for `REQ-CLUSTER-002..004`; EP-10 remains partial.
- **Context:** Fixed nodes A, B, and C run one voter and one replica server per JVM with policy `N=3/W=2`.
- **Stimulus:** An unconditional whole-object write succeeds, has only one checksum-valid durable acknowledgement, or has two staged replicas but no control quorum.
- **Response:** S3 success occurs only after at least `W=2` checksum-valid durable acknowledgements and consensus commit of the object-reference generation. Otherwise publication fails, staged bytes remain unreachable, and no degraded result is returned.
- **Metric (2026-07-13):** The Java 21 shared real-process WebTestClient/AWS CLI gate passes 14 scenarios / 188 steps for `REQ-CLUSTER-001..005/019/020`; the first five publication scenarios retain their passing results. Focused cross-module `REQ-CLUSTER-009/010` mechanisms and the post-repair data-plane regression also pass. Ordinary root Maven success is supporting evidence only.

==== Scenario 12 — Fixed-Slice Failover and Complete-Restart Integrity

- **Status:** `implementation-informed`, `@implemented-and-validated` for `REQ-CLUSTER-001`, `002`, and `005`.
- **Context:** Bucket and whole-object reference generations are committed in fixed A/B/C; stable UUIDs and each node's identity, Ratis, object, temporary, and runtime roots are distinct.
- **Stimulus:** Stop coordinator A while B and C remain mutually reachable, then retrieve through B; separately stop all A/B/C JVMs and restart them from their original non-empty roots with seed order `C,A,B`.
- **Response:** B resolves the consensus-committed generation and returns the exact uploaded bytes, length, SHA-256, and ETag. Complete restart recovers original identities and committed state without seed order rewriting membership.
- **Metric (2026-07-13):** Both flows pass in the shared WebTestClient and AWS CLI real-process scenarios. The result proves one-node stop and complete fixed-cluster restart, not dynamic replacement, general partitions, healing, or production readiness.

==== Scenario 13 — Bounded Cluster Mechanism and Security Safety

- **Status:** `implementation-informed`, `@implemented-and-validated` for `REQ-CLUSTER-008..013`; `REQ-CLUSTER-014` remains `@partial`.
- **Context:** The fixed profile uses Ratis control, separate grpc-java replica transfer, stable UUIDs, 64 KiB maximum payload frames, and test-local mTLS fixtures.
- **Stimulus:** Start/restart fixed voters, transfer a multi-frame object under slow demand, cancel or expire transfer, attempt an invalid peer, or evaluate publication ordering across modules.
- **Response:** Persisted fixed state recovers; flow control remains finite; cancellation/deadline releases resources without durable acknowledgement; both internal transports reject unauthenticated or identity-mismatched peers; S3 remains the only external object API.
- **Metric (2026-07-13):** Focused real Ratis, gRPC, TLS, and cross-module Cucumber reports for `REQ-CLUSTER-008..013` pass. Clustered multipart, conditional/versioned or chunked writes, erasure coding, dynamic membership/PKI lifecycle, healing/rebalance/orphan cleanup, rolling upgrades, and broader partitions remain not implemented or unvalidated (`006/007`, `015..018`).

==== Scenario 15 — Missing-local Repair Before GET Response (`REQ-CLUSTER-019`)

- **Status:** `implementation-informed`, `@implemented-and-validated`.
- **Context:** Fixed A/B/C has a current committed whole-object generation naming target B, but B's exact artifact is absent and another named replica is checksum-valid.
- **Stimulus:** An S3 client sends GET through B.
- **Response:** Before emitting response payload, B durably ensures/deduplicates the target-specific job, uses a current fenced claim, transfers and completely verifies a different named replica directly, durably publishes it, and opens the repaired local artifact once for the response. No complete local-payload preflight or second filesystem read occurs.
- **Metric (2026-07-13):** `REQ-CLUSTER-019` passes in both Java 21 real-process WebTestClient and AWS CLI modes; together with `020`, the focused repair-only gate passes 4 scenarios / 80 steps.

==== Scenario 16 — Corruption Failure and Later-read Recovery (`REQ-CLUSTER-020`)

- **Status:** `implementation-informed`, `@implemented-and-validated`.
- **Context:** The current reference names B and B has a present artifact whose corruption becomes visible only while the one-pass GET stream is consumed.
- **Stimulus:** A client reads through B, then a later client reads after repair completion.
- **Response:** The first GET fails with an integrity outcome, durably schedules/deduplicates repair, and never transparently retries another replica because bytes may already have been emitted. A fenced worker later verifies and durably replaces the artifact; only the later GET may succeed. Scheduling never converts the failed request into success.
- **Metric (2026-07-13):** `REQ-CLUSTER-020` passes in both real-process validation modes with client/body integrity assertions and no transparent corrupt-stream retry. The combined `001..005/019/020` gate passes 14 scenarios / 188 steps.

==== Scenario 17 — Durable-job Authority, Fencing, and Idempotent Recovery (`REQ-CLUSTER-021..025`)

- **Status:** `implementation-informed`; `021/022/023/025` are `@implemented-and-validated`, while `024` is `@partial`.
- **Context:** One repair job is deduplicated, transitions through consensus, or is interrupted around claim, transfer, durable publication, completion, snapshot, restart, or leader change.
- **Stimulus:** A process-local scheduler queries committed work; a new process session claims/reclaims while a stale worker may resume.
- **Response:** Ratis, not scheduler memory, owns one semantic job and monotonic attempt/claim history. `ClusterNodeRuntime` owns scheduler start/stop; wake signals contain no job state. Reclaim increments `claimGeneration`; already-valid target bytes are idempotent success; stale result commands have no effect; a changed reference makes the job `OBSOLETE`.
- **Metric (2026-07-13):** Repair-control `REQ-CLUSTER-021..026` passes 22 scenarios / 210 steps and the data-plane regression passes. `REQ-CLUSTER-024` remains partial because control persistence, fencing, and some scheduler reconciliation do not constitute the full seven-point real-filesystem/gRPC crash-side-effect matrix.

==== Scenario 18 — Snapshot Migration and Fail-closed Compatibility (`REQ-CLUSTER-026`)

- **Status:** `implementation-informed`, `@implemented-and-validated`.
- **Context:** A node starts from the reference-only version-1 snapshot, version-2 active/terminal repair state, or an unknown future schema.
- **Stimulus:** Load snapshot, replay committed commands, and take the next snapshot.
- **Response:** Version 1 deterministically initializes an empty job map and next writes version 2; version 2 preserves job specification, lifecycle, attempts/dedup history, claim/session/retry fields, terminal reasons, and last-applied metadata; unknown future versions fail closed without discarding work. Payload bytes never enter snapshots.
- **Metric (2026-07-13):** `REQ-CLUSTER-026` passes in the focused 22-scenario / 210-step repair-control gate, including stale-token rejection after restore.

These scenarios deliberately exclude prepared-artifact intents, orphan cleanup, rebalance, broad periodic anti-entropy, dynamic membership, erasure coding, multipart, and broader partitions. Broad `REQ-CLUSTER-017` remains partial, and no production-readiness claim follows.
