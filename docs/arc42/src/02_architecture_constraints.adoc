[[section-architecture-constraints]]
== Architecture Constraints

=== Technical Constraints

- **Java 21**: latest LTS, records, sealed interfaces, pattern matching
- **Spring Boot 4 WebFlux**: reactive non-blocking stack
- **Maven multi-module**: flat sibling modules per bounded context
- **Repository-wide payload bound (`planned / needs-validation`)**: every payload-plane element MUST have a finite byte bound, and every stage MUST also bound demand/prefetch, concurrency, queues, retained residuals, retries, and in-flight transport/storage work. `request(n)` or `limitRate(n)` alone is an element-count control, not a byte budget. Full-payload aggregation/replay is forbidden in authentication, encryption, repositories, transforms, and response filters; deliberately small control-plane documents require an explicit cap.
- **Reactive buffer ownership (`planned / needs-validation`)**: every owned pooled `DataBuffer` MUST be transferred, consumed/released, or retained with a matched later release exactly once across success, error, cancellation, discard, over-limit, ignored-body, and injected-fault paths. Terminal cleanup MUST be lifecycle-owned and MUST NOT perform synchronous filesystem work on an arbitrary Reactor terminal thread.
- **Blocking-work isolation (`planned / needs-validation`)**: filesystem/channel access, credential/policy reload, durable audit append/fsync, ACL access, encryption inspection output, cleanup, replica sink writes, and CPU-heavy transform/EC work MUST use finite role-specific bulkheads with explicit thread, task/byte queue, serialization, saturation, cancellation, and lifecycle rules. Wrapping synchronous work in `Mono`/`Flux` or selecting `boundedElastic` without those bounds is insufficient.
- **Deterministic fault injection (accepted ADR 0030; final P4 after active EP-10 and normal-path closure; implementation `planned / needs-validation`)**: deterministic storage faults MUST enter through a transport-neutral application `StorageFaultInjectionPort` with a no-op default, versioned closed semantic points/actions, stable operation/occurrence/offset/frame coordinates, finite fire/evidence budgets, dual non-production-profile plus opt-in-property activation, fail-closed plan validation, and payload/secret-free evidence. Faults MUST NOT weaken `N=3/W=2`, create degraded writes or false durable acknowledgements, put payload bytes in Ratis, or publish a reference before checksum-valid durable acknowledgements.
- **S3 object API boundary**: public object-storage endpoints MUST be AWS S3-compatible. Internal/admin endpoints are allowed only under the separate `admin-api-adapter` `/admin/**` surface and must not be counted as S3 API coverage.
- **Control-plane boundary**: the Admin Control Plane MUST NOT expose object/bucket CRUD, listing, multipart, metadata, tagging, ACL, versioning, or other S3 semantics. Those remain exclusively on the S3 Data Plane. Admin-side S3 diagnostics MUST call S3 as a client and MUST NOT bypass through storage-engine ports or private routes.
- **Product-shell neutrality (ADR 0026, implementation-informed)**: the reusable Magrathea Product Shell MUST contain no Object Storage domain or API logic. Product-specific routes, labels, permissions, endpoint paths, clients, models, and screens belong to a Product Extension; this repository's extension is the Object Storage Admin Application.
- **Pluggable S3 adapter**: S3 API lives in `s3-reactive-api-adapter` and is loaded via Spring Boot auto-configuration, not component scanning
- **S3 API toggle**: `s3.api.enabled=false` disables S3 routes even when the module is present
- **RouterFunction**: NO `@Controller`, NO `@RequestMapping` — functional WebFlux style only
- **XML serialization**: Jackson 3 XML annotated record objects (`tools.jackson.dataformat.xml.annotation.*`) with `XmlMapper`, NOT StringBuilder
- **Jackson 3 XML**: dependency `tools.jackson.dataformat:jackson-dataformat-xml`, NOT `com.fasterxml.jackson.dataformat:jackson-dataformat-xml`
- **XML codec**: custom `JacksonXmlEncoder` extending `AbstractJacksonEncoder<XmlMapper>` must be registered via `WebFluxConfigurer` or `HandlerStrategies`
- **Internal cluster protocol (ADRs 0027/0028, implementation-informed for the fixed slice)**: inter-node APIs remain internal-only protobuf gRPC over HTTP/2 with mandatory mTLS. The implemented grpc-java bridge uses 64 KiB maximum payload frames and MUST propagate demand, cancellation, and deadlines while bounding queues, message sizes, in-flight work, idle timeouts, and retries.
- **Fixed identity and membership boundary (ADRs 0027/0028)**: the implemented first slice has exactly A/B/C, one voter and one replica server per JVM, with a stable UUID recovered from a distinct persisted identity root. The identical static manifest MUST NOT rewrite recovered non-empty Ratis state. It is not dynamic membership: admission, promotion, demotion, replacement, and removal remain unimplemented consensus-controlled work, and health suspicion MUST NOT auto-evict a member.
- **Cluster consistency boundary (ADRs 0027/0028/0029, implementation-informed)**: the Ratis log orders bucket and object-reference generations and bounded current-generation repair lifecycle in the fixed slice. Whole-object bytes and repair side effects MUST remain outside that log and move directly between selected replica services. Wider namespace/configuration state, topology/policy epochs, fencing/tombstones, and other durable cluster jobs remain future ADR 0027 scope unless separately evidenced.
- **Fixed-slice durability policy (ADR 0028)**: unconditional whole-object writes use `N=3, W=2`, publish a reference only after checksum-valid durable acknowledgements and consensus commit, and fail instead of degrading below either quorum. Reads resolve the consensus-committed generation. Bounded repair under ADR 0029 does not imply clustered multipart, conditional/versioned or chunked writes, erasure coding, broad healing, rebalance, cleanup, or general partition tolerance.
- **Topology and policy purity (ADR 0027)**: `DistributedPlacementPlanner`, `QuorumPolicy`, `AntiEntropyPlanner`, `RebalancePlanner`, and `DistributedReadinessReporter` remain pure policy components without transport, Ratis, persistence, or retry dependencies. Concrete parent-linked `zone → rack → host → disk-set → device` membership/topology lifecycle remains planned beyond fixed A/B/C.
- **Repair authority (accepted ADR 0029, implementation-informed)**: durable repair is restricted to a missing or corrupt target already named by the current consensus-committed `WHOLE_OBJECT` generation on fixed A/B/C. PA-6 observations, findings, plans, and `HealingTask` values remain planning output; application code proposes work, consensus owns job lifecycle, and fenced workers perform network/filesystem effects outside the replicated state machine. A process-local scheduler only wakes and queries committed work; `ClusterNodeRuntime` owns its start/stop lifecycle.
- **Repair identity and fencing (accepted ADR 0029)**: one logical job is deduplicated by `(REPAIR, bucket, objectKey, referenceGeneration, artifactId, targetNodeId)`. Candidate source is an attempt hint, not identity. Every accepted claim/reclaim increments `claimGeneration`; only the exact `(jobId, claimGeneration)` assigned to the stable node/process session may renew or report a result. Ratis term and claim expiry are not safety fences.
- **Single-pass repair read constraint (accepted ADR 0029)**: GET MUST NOT completely preflight a present local payload and then read it again. Known local absence may trigger a verified direct remote fetch and durable local publication before response streaming begins. Corruption discovered during the single response pass MUST fail that request and durably ensure repair; it MUST NOT transparently retry that corrupt request from another replica. Only a later GET may benefit from completed repair.
- **Repair persistence and data path (accepted ADR 0029)**: the implemented Ratis snapshot/command schema version 2 preserves repair specifications, states, attempt/dedup history, claim generations, owner sessions, retry timing, and last-applied metadata and migrates version 1 explicitly. Payload bytes and side effects remain outside Ratis and use the direct mTLS replica path with complete length/checksum verification and crash-safe publication.
- **Repair-only slice boundary (accepted ADR 0029)**: `REQ-CLUSTER-019..026`, including the bounded seven-point real-filesystem/gRPC interruption scope in `024`, are implemented and validated. Broad `REQ-CLUSTER-017` remains partial because prepared-artifact intents, orphan cleanup, rebalance, and periodic anti-entropy remain absent; `REQ-CLUSTER-014` also remains partial, and `006/007/015/016/018` remain not implemented. Dynamic membership, erasure coding, multipart behavior, rolling upgrades, general chaos, and broader partition handling are deferred. No production-readiness claim follows.

=== Domain Constraints

- `object-store-domain`: uses ONLY AWS S3 native terminology — no custom abstractions like BinaryContent
- `object-store-domain`: zero dependencies except test dependencies
- No `shared-domain` module exists
- `storage-engine-domain`, `storage-engine-reactive-application`, `storage-engine-reactive-infrastructure`, and `object-store-reactive-repository-storage-engine-infrastructure` form the Storage Engine bounded context. Policy/YAML/device planning for `MINIO_STANDARD` and end-to-end storage-engine runtime read/write (object bytes, manifests, object references) are verified; distributed multi-node execution and further Enterprise Production Readiness phases (EP-1 security, EP-3 streaming completion, EP-4 space management) remain open per `PLAN.md`.

=== Durability Constraints (Storage Engine backend)

- Every durable metadata document (bucket registry, multipart upload state, per-object configuration, S3 object manifest references) is committed with the same crash-safe discipline: write to a temp file in the target directory, then atomically rename over the final path (`DurableJson`, `S3ObjectManifestReferenceStore`). Readers and restart recovery never observe a torn or partially written document.
- Concurrent mutation of the same durable key is serialized through striped per-key `ReentrantLock`s (typically 64 stripes); unrelated keys use independent stripes and are not serialized against each other.
- The Storage Engine backend is selected explicitly via the `storage-engine` Spring profile; the in-memory backend remains the default and does not persist across restarts.

=== Infrastructure Constraints

- `object-store-reactive-infrastructure`: reactive repository implementations only; no HTTP API routes
- `InMemoryReactiveS3ObjectRepository`: reactive in-memory S3 object metadata + content store via Mono/Flux
- `ContentStore` is removed; object content is saved through `S3ObjectRepository.save(S3ObjectWrite)`
- `S3ObjectWrite` and `S3ObjectContent` are domain interfaces without framework types; their application implementations carry `Flux<DataBuffer>`
- **Payload-array boundary target (`planned / needs-validation`)**: complete object payloads MUST NOT cross application/repository contracts as `byte[]`. Bounded arrays remain valid for digests, small capped control-plane codecs, generated protobuf values, and explicitly bounded frames. The current `StorageContext`/`ReactiveStorageOrchestrator` read contract still exposes bounded `Flux<byte[]>`, while signed-payload and SSE-S3 paths contain whole-payload arrays requiring supported-path validation; the legacy in-memory adapter also contains arrays but is isolated behind `legacy-in-memory-test`, is not the single-node/default product backend, and may be retired separately. See link:../audits/2026-07-14-reactive-streaming-and-pipeline-chaos-source-audit.adoc[the 2026-07-14 source audit].

=== XML Serialization Constraints

- **FORBIDDEN: regex XML parsing** — `java.util.regex.Pattern`, `String.contains()`, and `extractXmlValue()` for XML are forbidden. All XML parsing must use Jackson 3 codec
- **JacksonXmlDecoder mandatory**: `JacksonXmlCodecConfig` must register both the encoder (response) and the decoder (request) for XML. Without the decoder, `request.bodyToMono(Dto.class)` will not work
- **Command/Query DTO pattern**: every PUT/POST receiving XML uses a Jackson XML-annotated Command DTO; every GET producing XML uses a Query DTO. No generic "request" or "response" DTOs
- **DTOs in separate files**: each Command/Query DTO lives in a dedicated file in the `dto/command/` or `dto/query/` package. No monolithic files (e.g., no `S3Request.java`)
- **No custom parsers**: no classes like `S3XmlParser` wrapping `XmlMapper`. The framework deserializes directly — `S3XmlParser` and `S3XmlResponses` removed, ADR 0008 implemented and verified
- **Application DTO as Command**: XML Command DTOs may coincide with application DTOs (e.g., `CorsConfigurationCommand`) if they have the required Jackson XML annotations
- **Mapping in handler**: conversion from XML Command to domain value object occurs in the handler, not in a separate parser

=== Testing Constraints

- Domain tests: pure JUnit, NO Spring context
- Integration tests: Cucumber BDD with WebTestClient in `s3-reactive-api-adapter`
- AWS CLI compatibility tests: targeted AWS CLI Cucumber in `s3-reactive-api-adapter`; `test-aws-cli.sh` / Maven profile `aws-cli-tests` remains a separate broad compatibility script
- Coverage verification: the root-only `coverage` profile (`mvn -B -Pcoverage clean verify --no-transfer-progress`) MUST publish one reactor-wide JaCoCo HTML/XML/CSV report under `target/site/jacoco-aggregate` plus the threshold-free `target/site/quality-evidence/summary.json`; module-local execution data is aggregate input, not a competing report. OpenClover remains optional/legacy.
- Dependency vulnerability verification: the opt-in `scripts/run-dependency-check.sh` gate uses OWASP Dependency-Check 12.2.2, fails on unsuppressed CVSS >= 7.0 findings, and MUST fail closed when vulnerability data or scan completeness is unavailable. `NVD_API_KEY` MUST be injected through the environment/CI secret, never committed or passed as a Maven command-line property. Complete reports belong at `target/dependency-check-report.{json,html}`; normalized/error evidence belongs at `target/site/dependency-check-analysis.json`, `target/dependency-check-scan-error.json`, and `target/dependency-check-scan.log`.
- EP-8 supply-chain evidence MUST include CycloneDX JSON and XML application SBOMs, an exact-image CycloneDX SBOM, an SPDX-normalized dependency-license inventory, retained fail-closed OWASP evidence, and runtime validation of non-root operation, dropped capabilities, no privilege escalation, isolated host namespaces, no engine socket, and a read-only root filesystem with explicit writable mounts. Evidence MUST bind to one clean full revision and immutable local image ID, state that no publication occurred, preserve unavailable user-namespace remapping truthfully, and make no compliance, vulnerability-cleanliness, or remediation claim when evidence does not support one.
