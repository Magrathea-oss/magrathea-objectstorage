[[section-architecture-constraints]]
== Architecture Constraints

=== Technical Constraints

- **Java 21**: latest LTS, records, sealed interfaces, pattern matching
- **Spring Boot 4 WebFlux**: reactive non-blocking stack
- **Maven multi-module**: flat sibling modules per bounded context
- **S3 object API boundary**: public object-storage endpoints MUST be AWS S3-compatible. Internal/admin endpoints are allowed only under the separate `admin-api-adapter` `/admin/**` surface and must not be counted as S3 API coverage.
- **Control-plane boundary**: the Admin Control Plane MUST NOT expose object/bucket CRUD, listing, multipart, metadata, tagging, ACL, versioning, or other S3 semantics. Those remain exclusively on the S3 Data Plane. Admin-side S3 diagnostics MUST call S3 as a client and MUST NOT bypass through storage-engine ports or private routes.
- **Product-shell neutrality (ADR 0026, implementation-informed)**: the reusable Magrathea Product Shell MUST contain no Object Storage domain or API logic. Product-specific routes, labels, permissions, endpoint paths, clients, models, and screens belong to a Product Extension; this repository's extension is the Object Storage Admin Application.
- **Pluggable S3 adapter**: S3 API lives in `s3-reactive-api-adapter` and is loaded via Spring Boot auto-configuration, not component scanning
- **S3 API toggle**: `s3.api.enabled=false` disables S3 routes even when the module is present
- **RouterFunction**: NO `@Controller`, NO `@RequestMapping` — functional WebFlux style only
- **XML serialization**: Jackson 3 XML annotated record objects (`tools.jackson.dataformat.xml.annotation.*`) with `XmlMapper`, NOT StringBuilder
- **Jackson 3 XML**: dependency `tools.jackson.dataformat:jackson-dataformat-xml`, NOT `com.fasterxml.jackson.dataformat:jackson-dataformat-xml`
- **XML codec**: custom `JacksonXmlEncoder` extending `AbstractJacksonEncoder<XmlMapper>` must be registered via `WebFluxConfigurer` or `HandlerStrategies`
- **Planned cluster protocol (ADR 0027)**: inter-node APIs MUST remain internal-only protobuf gRPC over HTTP/2 with mandatory mTLS. Reactor bridges MUST propagate demand and cancellation and bound queues, message sizes, in-flight work, deadlines, and retries.
- **Planned identity and membership (ADR 0027)**: node identity is a stable persisted UUID. Static seeds are bootstrap hints only; admission, promotion, demotion, replacement, and removal are authoritative only after consensus commit. Health suspicion MUST NOT auto-evict a member.
- **Planned consistency boundary (ADR 0027)**: the consensus log owns membership, namespace/configuration state, object-reference/manifest generations, topology/policy epochs, fencing/tombstones, and durable cluster jobs. Object/replica/EC bytes MUST remain outside the consensus log and move directly between selected nodes.
- **Planned durability policy (ADR 0027)**: initial replicated writes use `N=3, W=2`, publish a reference only after checksum-valid durable acknowledgements and consensus commit, and do not degrade below policy by default. Reads resolve the consensus-committed generation before applying a configurable checksum-valid data quorum (default `1`) and alternate retry/repair.
- **Planned topology (ADR 0027)**: YAML catalogs extend additively to concrete parent-linked `zone → rack → host → disk-set → device` identities. `DistributedPlacementPlanner`, `QuorumPolicy`, `AntiEntropyPlanner`, `RebalancePlanner`, and `DistributedReadinessReporter` remain pure policy components without transport, Ratis, persistence, or retry dependencies.

=== Domain Constraints

- `object-store-domain`: uses ONLY AWS S3 native terminology — no custom abstractions like BinaryContent
- `object-store-domain`: zero dependencies except test dependencies
- No `shared-domain` module exists
- `storage-engine-domain`, `storage-engine-reactive-application`, `storage-engine-reactive-infrastructure`, and `object-store-reactive-repository-storage-engine-infrastructure` form the Storage Engine bounded context. Policy/YAML/device planning for `MINIO_STANDARD` and end-to-end storage-engine runtime read/write (object bytes, manifests, object references) are verified; distributed multi-node execution and further Enterprise Production Readiness phases (EP-1 security, EP-3 streaming completion, EP-4 space management) remain open per `PLAN.md`.

=== Durability Constraints (Storage Engine backend)

- Every durable metadata document (bucket registry, multipart upload state, per-object configuration, S3 object manifest references) is committed with the same crash-safe discipline: write to a temp file in the target directory, then atomically rename over the final path (`DurableJson`, `S3ObjectManifestReferenceStore`). Readers and restart recovery never observe a torn or partially written document.
- Concurrent mutation of the same durable key is serialized through striped per-key `ReentrantLock`s (typically 64 stripes); unrelated keys use independent stripes and are not serialized against each other.
- The Storage Engine backend is selected explicitly via the `storage-engine` Spring profile; the in-memory backend remains the default and does not persist across restarts.

=== Infrastructure Constraints

- `object-store-reactive-infrastructure`: reactive repository implementations only; no HTTP API routes
- `InMemoryReactiveS3ObjectRepository`: reactive in-memory S3 object metadata + content store via Mono/Flux
- `ContentStore` is removed; object content is saved through `S3ObjectRepository.save(S3ObjectWrite)`
- `S3ObjectWrite` and `S3ObjectContent` are domain interfaces without framework types; their application implementations carry `Flux<DataBuffer>`
- byte[] only exists inside repository content storage internals, never in domain signatures

=== XML Serialization Constraints

- **FORBIDDEN: regex XML parsing** — `java.util.regex.Pattern`, `String.contains()`, and `extractXmlValue()` for XML are forbidden. All XML parsing must use Jackson 3 codec
- **JacksonXmlDecoder mandatory**: `JacksonXmlCodecConfig` must register both the encoder (response) and the decoder (request) for XML. Without the decoder, `request.bodyToMono(Dto.class)` will not work
- **Command/Query DTO pattern**: every PUT/POST receiving XML uses a Jackson XML-annotated Command DTO; every GET producing XML uses a Query DTO. No generic "request" or "response" DTOs
- **DTOs in separate files**: each Command/Query DTO lives in a dedicated file in the `dto/command/` or `dto/query/` package. No monolithic files (e.g., no `S3Request.java`)
- **No custom parsers**: no classes like `S3XmlParser` wrapping `XmlMapper`. The framework deserializes directly — `S3XmlParser` and `S3XmlResponses` removed, ADR 0008 implemented and verified
- **Application DTO as Command**: XML Command DTOs may coincide with application DTOs (e.g., `CorsConfigurationCommand`) if they have the required Jackson XML annotations
- **Mapping in handler**: conversion from XML Command to domain value object occurs in the handler, not in a separate parser

=== Testing Constraints

- Domain tests: pure JUnit, NO Spring context
- Integration tests: Cucumber BDD with WebTestClient in `s3-reactive-api-adapter`
- AWS CLI compatibility tests: targeted AWS CLI Cucumber in `s3-reactive-api-adapter`; `test-aws-cli.sh` / Maven profile `aws-cli-tests` remains a separate broad compatibility script
- Coverage verification: the root-only `coverage` profile (`mvn -B -Pcoverage clean verify --no-transfer-progress`) MUST publish one reactor-wide JaCoCo HTML/XML/CSV report under `target/site/jacoco-aggregate` plus the threshold-free `target/site/quality-evidence/summary.json`; module-local execution data is aggregate input, not a competing report. OpenClover remains optional/legacy.
- Dependency vulnerability verification: the opt-in `scripts/run-dependency-check.sh` gate uses OWASP Dependency-Check 12.2.2, fails on unsuppressed CVSS >= 7.0 findings, and MUST fail closed when vulnerability data or scan completeness is unavailable. `NVD_API_KEY` MUST be injected through the environment/CI secret, never committed or passed as a Maven command-line property. Complete reports belong at `target/dependency-check-report.{json,html}`; normalized/error evidence belongs at `target/site/dependency-check-analysis.json`, `target/dependency-check-scan-error.json`, and `target/dependency-check-scan.log`.
- Planned EP-8 release evidence MUST include CycloneDX JSON and XML SBOMs, an SPDX-normalized dependency-license inventory, retained fail-closed OWASP evidence, and runtime validation of non-root operation, dropped unnecessary capabilities, no privilege escalation, and a read-only root filesystem with explicit writable mounts. Dependency remediation is outside this approved EARLY slice; findings and scan failures MUST remain visible.
