== Cross-cutting Concepts

=== Hexagonal Architecture (Ports and Adapters)

The system follows the hexagonal architecture pattern with three layers:

- **Domain (core):** pure business logic, zero framework dependencies
- **Application:** orchestration, reactive bridge via Mono/Flux
- **Infrastructure:** HTTP adapter (RouterFunction), persistence adapter (in-memory)

Dependencies flow only inward: infrastructure → application → domain.

=== Reactive Non-blocking I/O

All HTTP request handling is reactive:

- RouterFunction instead of @Controller
- Flux<DataBuffer> for binary upload/download (never byte[] in handlers)
- Mono/Flux throughout the reactive application and infrastructure layers
- Schedulers.boundedElastic() only for operations that consume the stream in memory

=== Domain Purity

The `object-store-domain` module is strictly pure:

- NO Spring imports
- NO JPA, NO jakarta.validation
- NO reactive types (Mono, Flux, DataBuffer)
- Repository interfaces with CompletableFuture (standard Java async)
- S3ObjectWrite/S3ObjectContent without framework types

Advanced configuration value objects (BucketAnalyticsConfiguration, BucketInventoryConfiguration, BucketMetricsConfiguration, BucketIntelligentTieringConfiguration) follow the same pattern: immutable records, zero framework dependencies, validation with Objects.requireNonNull.





=== S3 API Compatibility

The public object API exposes AWS S3-compatible endpoints. Internal/admin endpoints are exposed separately by `admin-api-adapter` under `/admin/**` and must not be counted as S3 API coverage:

- Same HTTP signature (methods, headers, query parameters)
- Same XML format (ListAllMyBucketsResult, ListBucketResult, Error, etc.)
- Same error encoding (NoSuchBucket, NoSuchKey, BucketAlreadyExists)
- Pluggable via Spring Boot auto-configuration (s3-reactive-api-adapter attaches as an optional module)
- Toggle with `s3.api.enabled=false`

=== Admin API and Static Documentation Assets

`admin-api-adapter` exposes internal `/admin/**` JSON endpoints for health, read-only storage policy catalogs, read-only storage device catalogs, read-only disk-set/topology catalogs, and non-persistent storage policy validation. `bootstrap-application` also serves generated UI/documentation assets from classpath `static/**` resources, including ARC42 JSON and copied C4/ARC42 image files. These resources support operations and documentation; they are not part of the S3-compatible object API.

The current runtime mutability decision is configuration-as-code: YAML-backed storage policies, devices, and disk sets are managed outside the running process and exposed as read-only catalog data. `POST /admin/storage-policies/validate` validates a submitted payload and returns structured results without persisting it. Policy mutation requests are rejected at runtime with `405 Method Not Allowed`.

Planned Admin UI screens are policy list/detail/validation report, device list/detail, disk-set/topology overview/detail, and backend/status dashboard. Frontend implementation must be handed off to an appropriate frontend workflow/agent before changing `magrathea-ui`.

=== Pluggable S3 API Module

`s3-reactive-api-adapter` is an auto-configuring module that plugs into the Spring Boot context:

- `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
- `S3ApiConfig` guarded by `@ConditionalOnClass({ ReactiveBucketService.class, ReactiveObjectService.class })`
- `s3.api.enabled=true` by default
- If `s3-reactive-api-adapter` is not in the classpath, S3 routes are not loaded

=== Command/Query DTO Pattern for XML Serialization

The system exposes an XML S3-compatible API where each operation has a structured XML body. Instead of using custom parsers (regex, explicit XmlMapper) or generic "request"/"response" DTOs, the system follows the DDD CQS (Command-Query Separation) pattern:

- **Commands**: represent the intention to change state (PUT, POST, DELETE). They are Jackson XML-annotated DTOs that the framework deserializes directly from the HTTP body. Examples: `CorsConfigurationCommand`, `LifecycleConfigurationCommand`, `DeleteObjectsCommand`
- **Queries**: represent the response to a read request (GET). They are Jackson XML-annotated DTOs that the framework serializes directly into the HTTP body. Examples: `ListBucketsQuery`, `GetBucketCorsQuery`, `ErrorResponse`

Benefits:
- Zero custom parsing: Spring Boot 4 WebFlux deserializes/serializes with registered codecs
- Separate DTOs per operation: each Command/Query lives in its own file, organized by domain
- DDD alignment: Commands express intent, Queries express reads — semantically clear
- Maintainability: modifying an XML tag means modifying a single DTO, not a regex parser and a separate mapping

=== XML Codec Infrastructure

Spring Boot 4 WebFlux handles XML serialization via codecs registered in `JacksonXmlCodecConfig`:

- **JacksonXmlEncoder**: serializes response DTOs into XML
- **JacksonXmlDecoder**: deserializes request body into Command DTOs (added with ADR 0008)
- Both use `tools.jackson.dataformat:jackson-dataformat-xml` (Jackson 3, not Jackson 2)
- DTOs use `tools.jackson.dataformat.xml.annotation.*` annotations (`@JacksonXmlRootElement`, `@JacksonXmlProperty`, `@JacksonXmlElementWrapper`, `@JacksonXmlText`)

The decoder allows `request.bodyToMono(CorsConfigurationCommand.class)` to work for XML — the framework reads the body, passes it to the XML decoder, and produces the annotated record. No explicit call to `XmlMapper`.

=== Command-Query Separation (CQS) Repository Pattern

The application layer separates write and read concerns at the repository level:

- **Command repositories**: `BucketCommandRepository`, `S3ObjectCommandRepository`, `MultipartUploadCommandRepository` — accept Mono<Void> return types for write operations
- **Query repositories**: `BucketQueryRepository`, `S3ObjectQueryRepository`, `MultipartUploadQueryRepository` — return Mono<T> or Flux<T> for read operations

This enables independent optimization of read and write paths, and aligns with CQS principles at the persistence boundary.

=== Handler Minimal Pattern

All handlers (except `S3BucketConfigHandler`) follow the **extract → delegate → response** pattern:

```java
public Mono<ServerResponse> putObject(ServerRequest request) {
    return objectService.saveObjectWithContent(
            S3RequestExtractor.extractObjectKey(request),
            S3RequestExtractor.extractStorageClass(request),
            S3RequestExtractor.extractChecksum(request),
            S3RequestExtractor.extractEncryption(request),
            S3RequestExtractor.extractContentType(request),
            S3RequestExtractor.extractContentLength(request),
            S3RequestExtractor.extractUserMetadata(request),
            request.bodyToFlux(DataBuffer.class))
        .flatMap(result -> S3ResponseBuilder.ok(result.aggregate()))
        .onErrorResume(BucketNotFoundException.class,
            e -> S3WebSupport.xmlError(HttpStatus.NOT_FOUND, "NoSuchBucket", "Bucket not found"))
        .onErrorResume(Throwable.class,
            e -> S3WebSupport.xmlError(HttpStatus.INTERNAL_SERVER_ERROR, "InternalError", e.getMessage()));
}
```

Key constraints:
- **NO** validation — postponed to repository preconditions
- **NO** bucket existence checks — postponed to repository preconditions
- **NO** ETag computation — postponed to repository on store
- **NO** handler-local state — postponed to repository
- **NO** CORS/website logic — postponed to repository/service

=== Postponed Items

The following cross-cutting concerns are intentionally absent from handlers and will be implemented in the repository layer:

| Concern | Target |
|---|---|
| Validation (Spring Reactive Validator) | Repository preconditions |
| Bucket existence checks | Repository preconditions |
| Bucket name validation | Repository preconditions |
| ETag computation | Repository (generated on store) |
| Handler-local ACL/grants/tagging | Repository |
| CORS validation | Repository/Service |
| Website routing | Repository/Service |

=== Reactive Error Handling

Error handling follows the reactive pattern with typed exceptions and S3 XML error serialization:

==== Exception Hierarchy

```
S3Exception (abstract)
├── NoSuchBucketException     → 404 NoSuchBucket
├── NoSuchKeyException        → 404 NoSuchKey
├── BucketAlreadyExistsException → 409 BucketAlreadyExists
├── InvalidArgumentException  → 400 InvalidArgument
├── AccessDeniedException     → 403 AccessDenied
└── InternalServerException   → 500 InternalServerError
```

==== Handler Error Chain

```java
// Handler-level error handling
public Mono<ServerResponse> handleGet(ServerRequest request) {
    return service.findBucket(name)
        .flatMap(bucket -> service.getObject(bucket, key))
        .map(object -> buildResponse(object))
        .onErrorResume(NoSuchBucketException.class, e -> errorResponse(404, "NoSuchBucket"))
        .onErrorResume(NoSuchKeyException.class, e -> errorResponse(404, "NoSuchKey"))
        .onErrorResume(InvalidArgumentException.class, e -> errorResponse(400, "InvalidArgument"))
        .onErrorResume(throwable -> errorResponse(500, "InternalError"));
}
```

All error responses use `S3WebSupport.errorResponse()` which creates a Jackson 3 XML annotated `S3Error` record.

=== Domain Events and State Transitions

==== Aggregate Root Pattern

All three aggregates (Bucket, S3Object, MultipartUpload) follow the same pattern:

1. State transitions return NEW immutable instances — no mutation
2. Each transition records a domain event in an accumulated list
3. `domainEvents()` retrieves pending events
4. `clearEvents()` returns a clean instance ready for persistence

==== State Transition Rules

- State transitions return NEW immutable instances with accumulated events
- `domainEvents()` retrieves pending events; `clearEvents()` returns a clean instance for persistence

Each event carries the specific value object type (e.g., `BucketLifecycleConfiguration`, `BucketNotificationConfiguration`) so consumers can react to precise changes.

=== AWS CLI Header Compatibility

The system must handle headers that the AWS CLI v2 sends by default on every `aws s3api put-object` request. Phase 9 first-increment AWS CLI Cucumber now verifies that `put-object` succeeds with the default AWS CLI headers. Compatibility is still partial: checksum and SigV4 headers are accepted/ignored for this scope, but not validated.

==== Default Headers Sent by AWS CLI v2

| Header | Value | Purpose | Handler Status |
|--------|-------|---------|----------------|
| `x-amz-sdk-checksum-algorithm` | `crc64nvme` (default) | Identifies the checksum algorithm used | [yellow]#Accepted by first AWS CLI increment; not validated# |
| `x-amz-checksum-crc64nvme` | BASE64 CRC64-NVME hash | Payload integrity hash computed automatically by AWS CLI | [yellow]#Accepted by first AWS CLI increment; not validated# |
| `X-Amz-Date` | ISO 8601 timestamp | SigV4 signing timestamp | [yellow]#Accepted by first AWS CLI increment; not validated# |
| `X-Amz-Content-SHA256` | SHA256 hex string | SigV4 payload hash | [yellow]#Accepted by first AWS CLI increment; not validated# |
| `Authorization` | SigV4 signature string | SigV4 signature | [yellow]#Accepted by first AWS CLI increment; not validated# |
| `User-Agent` | SDK identifier string | Client identification | [yellow]#Accepted by first AWS CLI increment; not used# |
| `Expect` | `100-continue` | Hop-by-hop header for large payloads | [yellow]#Accepted by first AWS CLI increment; not optimized# |

==== Checksum Algorithm Mapping

When `--checksum-algorithm <alg>` is specified, AWS CLI v2 **automatically computes** the hash and sends it in the corresponding header:

| CLI `--checksum-algorithm` | Header Sent | Automatic Computation |
|---------------------------|-------------|---------------------|
| `crc64nvme` (default) | `x-amz-checksum-crc64nvme` | [green]#Yes# — AWS CLI computes CRC64-NVME |
| `crc32` | `x-amz-checksum-crc32` | [green]#Yes# |
| `crc32c` | `x-amz-checksum-crc32c` | [green]#Yes# |
| `sha1` | `x-amz-checksum-sha1` | [green]#Yes# |
| `sha256` | `x-amz-checksum-sha256` | [green]#Yes# |
| `sha512` | `x-amz-checksum-sha512` | [green]#Yes# |
| `xxhash64` | `x-amz-checksum-xxhash64` | [green]#Yes# |
| `xxhash3` | `x-amz-checksum-xxhash3` | [green]#Yes# |
| `xxhash128` | `x-amz-checksum-xxhash128` | [green]#Yes# |

The server must read `x-amz-sdk-checksum-algorithm`, re-compute the hash of the received body with that algorithm, and compare against the corresponding `x-amz-checksum-<alg>` header. A mismatch must return `400 InvalidArgument (ChecksumMismatch)`.

==== Content-Type Handling

AWS CLI v2 does NOT send `Content-Type` by default unless `--content-type` is explicitly passed. The handler currently falls back to `application/octet-stream`. When AWS CLI is used, the stored content-type is incorrect — the object will be served with `application/octet-stream` instead of the actual content type.

==== Current Gap Summary

| Gap | Impact | Severity |
|-----|--------|---------|
| Checksum headers accepted but not validated | First AWS CLI object CRUD increment can pass, but data integrity validation is still absent — corrupted payloads may be accepted silently | [red]#High# |
| SigV4 headers accepted but not validated | First AWS CLI object CRUD increment can pass, but authentication is still absent — anyone can write/read objects in trusted deployments | [red]#High# |
| Content-Type not sent by AWS CLI | Wrong content-type may be stored when using AWS CLI unless explicitly provided | [yellow]#Medium# |
| User-Agent accepted but ignored | No client identification for analytics | [green]#Low# |
| Expect accepted but not optimized | No explicit 100-continue optimization | [green]#Low# |

=== Testing Strategy

Three levels of testing:

1. **Domain:** Pure JUnit, zero Spring — test invariants and business rules.
2. **Application:** Reactive application service tests.
3. **S3 Integration:** Cucumber BDD with WebTestClient — verify HTTP endpoints.
4. **AWS CLI Compatibility:** targeted AWS CLI Cucumber plus the separate/stale `test-aws-cli.sh` compatibility script.

Latest Phase 9 first-increment evidence: `mvn -B -pl s3-reactive-api-adapter -am test -Dsurefire.failIfNoSpecifiedTests=false` passed with **247 tests**, **0 failures**, **0 errors**, and **1 skipped** `@unsupported-awscli` scenario. The AWS CLI subset covers canonical object CRUD basics: put default headers, get content, head, list v1/v2, delete, idempotent delete, and `STANDARD` storage class via object attributes. Key-with-slashes remains skipped because current route matching does not support slash-containing keys.
