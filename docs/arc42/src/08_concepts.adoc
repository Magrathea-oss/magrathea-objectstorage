== Cross-cutting Concepts

=== Hexagonal Architecture (Ports and Adapters)

The system follows the hexagonal architecture pattern with three layers:

- **Domain (core):** pure business logic, zero framework dependencies
- **Application:** orchestration, reactive bridge via Mono/Flux
- **Infrastructure:** HTTP adapter (RouterFunction), persistence adapter (in-memory)

Dependencies flow only inward: infrastructure → application → domain.

=== Reactive Non-blocking I/O

All HTTP request handling is reactive:

- RouterFunction instead of @Controller
- Flux<DataBuffer> for binary upload/download (never byte[] in handlers)
- Mono/Flux throughout the reactive application and infrastructure layers
- Schedulers.boundedElastic() only for operations that consume the stream in memory

=== Domain Purity

The `object-storage-domain` module is strictly pure:

- NO Spring imports
- NO JPA, NO jakarta.validation
- NO reactive types (Mono, Flux, DataBuffer)
- Repository interfaces with CompletableFuture (standard Java async)
- S3ObjectWrite/S3ObjectContent without framework types

Advanced configuration value objects (BucketAnalyticsConfiguration, BucketInventoryConfiguration, BucketMetricsConfiguration, BucketIntelligentTieringConfiguration) follow the same pattern: immutable records, zero framework dependencies, validation with Objects.requireNonNull.

=== Domain Events and State Transitions

The Bucket aggregate root emits sealed `ObjectStorageEvent` records on every state transition:

- `BucketCreated`, `BucketDeleted`, `BucketVersioningEnabled`, `BucketVersioningSuspended`
- `BucketEncryptionEnabled` (separate from encryption configuration!)

==== Bucket Configuration Events

Instead of a single generic `BucketConfigurationChanged`, each configuration feature emits a specific event subtype:

```
ObjectStorageEvent
├── BucketCreated
├── BucketDeleted
├── BucketVersioningEnabled
├── BucketVersioningSuspended
├── BucketEncryptionEnabled
├── BucketCorsConfigurationChanged
├── BucketLifecycleConfigurationChanged
├── BucketEncryptionConfigurationChanged
├── BucketLoggingConfigurationChanged
├── BucketWebsiteConfigurationChanged
├── BucketNotificationConfigurationChanged
├── BucketReplicationConfigurationChanged
├── BucketPolicyChanged
├── BucketPublicAccessBlockChanged
├── BucketOwnershipControlsChanged
├── BucketRequestPaymentChanged
├── BucketAccelerateConfigurationChanged
├── BucketAnalyticsConfigurationAdded
├── BucketAnalyticsConfigurationRemoved
├── BucketInventoryConfigurationAdded
├── BucketInventoryConfigurationRemoved
├── BucketMetricsConfigurationAdded
├── BucketMetricsConfigurationRemoved
├── BucketIntelligentTieringConfigurationAdded
├── BucketIntelligentTieringConfigurationRemoved
```

Each event carries the specific value object type (e.g., `BucketLifecycleConfiguration`, `BucketNotificationConfiguration`) so consumers can react to precise changes.

==== State Transition Rules

- State transitions return NEW immutable instances with accumulated events
- `domainEvents()` retrieves pending events; `clearEvents()` returns a clean instance for persistence

S3Object and MultipartUpload aggregates follow the same pattern with `ObjectCreated`, `ObjectDeleted`, `ObjectEtagUpdated`, `ObjectStorageClassChanged`, `ContentDescriptorCreated`, `PartUploaded`, `MultipartUploadCreated`, `MultipartUploadCompleted`, `MultipartUploadAborted`.

=== Registry-based Handler Integration

The `S3BucketConfigHandler` is refactored into a registry-based design to eliminate copypasta (previously 925 lines of identical pattern).

==== Components

| Component | Responsibility |
|-----------|---------------|
| `ConfigType` | Enum or sealed interface identifying the configuration type (e.g., CORS, Lifecycle, Encryption) |
| `ConfigHandlerStrategy` | Interface with methods for: extracting config from Bucket aggregate, applying config change (creating proper event), serializing to Query DTO, deserializing from Command DTO |
| `ConfigHandlerRegistry` | A `Map<ConfigType, ConfigHandlerStrategy>` populated at startup |

==== How it works

1. `S3ProxyRouter` dispatches a config request to `S3BucketConfigHandler` with a `ConfigType`
2. The handler looks up the matching `ConfigHandlerStrategy` from the registry
3. The strategy performs the GET/PUT/DELETE operation using its typed methods
4. No switch/if-else chains — each config type is a few lines of strategy registration

==== Benefits

- Adding a new config type = write one strategy class + register it
- No copypasta — every config type uses the same delegation pattern
- Each strategy is independently testable
- The Bucket aggregate stays decoupled from handler concerns

==== Config Type Categorization

| Category | Types | Pattern |
|----------|-------|---------|
| **Singleton** | CORS, Lifecycle, Encryption, Logging, Website, Notification, Replication, Policy, PublicAccessBlock, OwnershipControls, RequestPayment, Accelerate | Single value object, `with*()` / `without*()` methods |
| **Multi-instance** | Analytics, Inventory, Metrics, Intelligent-Tiering | Map<String, ValueObject>, `add*()` / `remove*()` / `list*()` methods |

=== S3 API Compatibility

The system exposes ONLY AWS S3-compatible endpoints:

- Same HTTP signature (methods, headers, query parameters)
- Same XML format (ListAllMyBucketsResult, ListBucketResult, Error, etc.)
- Same error encoding (NoSuchBucket, NoSuchKey, BucketAlreadyExists)
- Pluggable via Spring Boot auto-configuration (s3-reactive-api-adapter attaches as an optional module)
- Toggle with `s3.api.enabled=false`

=== Pluggable S3 API Module

`s3-reactive-api-adapter` is an auto-configuring module that plugs into the Spring Boot context:

- `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
- `S3ApiConfig` guarded by `@ConditionalOnClass({ ReactiveBucketService.class, ReactiveObjectService.class })`
- `s3.api.enabled=true` by default
- If `s3-reactive-api-adapter` is not in the classpath, S3 routes are not loaded

=== Command/Query DTO Pattern for XML Serialization

The system exposes an XML S3-compatible API where each operation has a structured XML body. Instead of using custom parsers (regex, explicit XmlMapper) or generic "request"/"response" DTOs, the system follows the DDD CQS (Command-Query Separation) pattern:

- **Commands**: represent the intention to change state (PUT, POST, DELETE). They are Jackson XML-annotated DTOs that the framework deserializes directly from the HTTP body. Examples: `CorsConfigurationCommand`, `LifecycleConfigurationCommand`, `DeleteObjectsCommand`
- **Queries**: represent the response to a read request (GET). They are Jackson XML-annotated DTOs that the framework serializes directly into the HTTP body. Examples: `ListBucketsQuery`, `GetBucketCorsQuery`, `ErrorResponse`

Benefits:
- Zero custom parsing: Spring Boot 4 WebFlux deserializes/serializes with registered codecs
- Separate DTOs per operation: each Command/Query lives in its own file, organized by domain
- DDD alignment: Commands express intent, Queries express reads — semantically clear
- Maintainability: modifying an XML tag means modifying a single DTO, not a regex parser and a separate mapping

=== XML Codec Infrastructure

Spring Boot 4 WebFlux handles XML serialization via codecs registered in `JacksonXmlCodecConfig`:

- **JacksonXmlEncoder**: serializes response DTOs into XML
- **JacksonXmlDecoder**: deserializes request body into Command DTOs (added with ADR 0008)
- Both use `tools.jackson.dataformat:jackson-dataformat-xml` (Jackson 3, not Jackson 2)
- DTOs use `tools.jackson.dataformat.xml.annotation.*` annotations (`@JacksonXmlRootElement`, `@JacksonXmlProperty`, `@JacksonXmlElementWrapper`, `@JacksonXmlText`)

The decoder allows `request.bodyToMono(CorsConfigurationCommand.class)` to work for XML — the framework reads the body, passes it to the XML decoder, and produces the annotated record. No explicit call to `XmlMapper`.

=== Command-Query Separation (CQS) Repository Pattern

The application layer separates write and read concerns at the repository level:

- **Command repositories**: `BucketCommandRepository`, `S3ObjectCommandRepository`, `MultipartUploadCommandRepository` — accept Mono<Void> return types for write operations
- **Query repositories**: `BucketQueryRepository`, `S3ObjectQueryRepository`, `MultipartUploadQueryRepository` — return Mono<T> or Flux<T> for read operations

This enables independent optimization of read and write paths, and aligns with CQS principles at the persistence boundary.

=== Testing Strategy

Three levels of testing:

1. **Domain:** Pure JUnit, zero Spring — test invariants and business rules
2. **S3 Integration:** Cucumber BDD with WebTestClient — verify HTTP endpoints
3. **AWS CLI Compatibility:** `test-aws-cli.sh` bash script + `mvn verify -Paws-cli-tests`
