== Cross-cutting Concepts

=== Hexagonal Architecture (Ports and Adapters)

Il sistema segue il pattern hexagonal architecture con tre layer:

- **Dominio (core):** puro business logic, zero framework dependencies
- **Applicazione:** orchestrazione, bridge CompletableFuture ↔ Mono/Flux
- **Infrastruttura:** adapter HTTP (RouterFunction), adapter persistenza (in-memory)

Le dipendenze vanno solo verso l'interno: infrastruttura → applicazione → dominio.

=== Reactive Non-blocking I/O

Tutta la gestione delle richieste HTTP è reattiva:

- RouterFunction invece di @Controller
- Flux<DataBuffer> per upload/download binario (mai byte[] nei handler)
- Mono.fromFuture() per bridge CompletableFuture → reattivo
- Schedulers.boundedElastic() solo per operazioni che consumano il flusso in memoria

=== Domain Purity

Il modulo `object-storage-domain` è rigorosamente puro:

- NO Spring imports
- NO JPA, NO jakarta.validation
- NO reactive types (Mono, Flux, DataBuffer)
- Repository interfaces con CompletableFuture (standard Java async)
- S3ObjectWrite/S3ObjectContent senza tipi framework

I value object di configurazione avanzata (BucketAnalyticsConfiguration, BucketInventoryConfiguration, BucketMetricsConfiguration, BucketIntelligentTieringConfiguration) seguono lo stesso pattern: record immutabili, zero dipendenze framework, validation con Objects.requireNonNull.

=== S3 API Compatibility

Il sistema espone SOLO endpoint AWS S3-compatible:

- Stessa firma HTTP (metodi, header, query parameters)
- Stesso formato XML (ListAllMyBucketsResult, ListBucketResult, Error, ecc.)
- Stessa codifica degli errori (NoSuchBucket, NoSuchKey, BucketAlreadyExists)
- Pluggable via Spring Boot auto-configuration (s3-api si attacca come modulo opzionale)
- Toggle con `s3.api.enabled=false`

=== Pluggable S3 API Module

`s3-api` è un modulo auto-configurante che si innesta nel contesto Spring Boot:

- `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
- `S3ApiConfig` guardato da `@ConditionalOnClass({ BucketService.class, ObjectService.class })`
- `s3.api.enabled=true` per default
- Se `s3-api` non è nel classpath, le route S3 non vengono caricate

=== Command/Query DTO Pattern for XML Serialization

Il sistema espone un'API XML S3-compatible dove ogni operazione ha un body XML strutturato. Invece di usare parser custom (regex, XmlMapper esplicito) o DTO generici "request"/"response", il sistema segue il pattern CQS (Command-Query Separation) del DDD:

- **Commands**: rappresentano l'intenzione di cambiare stato (PUT, POST, DELETE). Sono DTO annotati Jackson XML che il framework deserializza direttamente dal body HTTP. Esempi: `CorsConfigurationCommand`, `LifecycleConfigurationCommand`, `DeleteObjectsCommand`
- **Queries**: rappresentano la risposta a una richiesta di lettura (GET). Sono DTO annotati Jackson XML che il framework serializza direttamente nel body HTTP. Esempi: `ListBucketsQuery`, `GetBucketCorsQuery`, `ErrorResponse`

Vantaggi:
- Zero parsing custom: Spring Boot 4 WebFlux deserializza/serializza con i codec registrati
- DTO separati per operazione: ogni Command/Query vive nel proprio file, organizzato per dominio
- Allineamento DDD: i Command esprimono intento, i Query esprimono lettura — semanticamente chiari
- Manutenibilità: modificare un XML tag significa modificare un solo DTO, non un parser regex e un mapping separato

=== XML Codec Infrastructure

Spring Boot 4 WebFlux gestisce la serializzazione XML tramite codec registrati in `JacksonXmlCodecConfig`:

- **JacksonXmlEncoder**: serializza response DTO in XML (gia' presente)
- **JacksonXmlDecoder**: deserializza request body in Command DTO (aggiunto con ADR 0008)
- Entrambi usano `tools.jackson.dataformat:jackson-dataformat-xml` (Jackson 3, non Jackson 2)
- I DTO usano annotazioni `tools.jackson.dataformat.xml.annotation.*` (`@JacksonXmlRootElement`, `@JacksonXmlProperty`, `@JacksonXmlElementWrapper`, `@JacksonXmlText`)

Il decoder permette a `request.bodyToMono(CorsConfigurationCommand.class)` di funzionare per XML — il framework legge il body, lo passa al decoder XML, e produce il record annotato. Nessuna chiamata esplicita a `XmlMapper`.

=== Testing Strategy

Tre livelli di test:

1. **Dominio:** JUnit puro, zero Spring — testano invarianti e regole di business
2. **Integrazione S3:** Cucumber BDD con WebTestClient — verificano endpoint HTTP
3. **Compatibilità AWS CLI:** `test-aws-cli.sh` script bash + `mvn verify -Paws-cli-tests`
