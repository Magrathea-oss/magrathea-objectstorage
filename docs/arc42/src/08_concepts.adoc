[[section-concepts]]
== Cross-cutting Concepts

=== Hexagonal Architecture (Ports and Adapters)

The system follows the hexagonal architecture pattern with three layers:

- **Domain (core):** pure business logic, zero framework dependencies
- **Application:** orchestration, reactive bridge via Mono/Flux
- **Infrastructure:** HTTP adapter (RouterFunction), persistence adapters (in-memory default, or durable Storage Engine backend selected via Spring profile)

Dependencies flow only inward: infrastructure → application → domain.

=== Durable Metadata Commit Pattern (EP-2)

Every durable document written by the Storage Engine backend follows the same crash-safe commit discipline, shared by `S3ObjectManifestReferenceStore`, `BucketStore`, `MultipartUploadStateStore`, and `ObjectConfigMetadataStore` (all in `object-store-reactive-repository-storage-engine-infrastructure`):

1. Serialize the document (Jackson 3 `ObjectMapper` with field-only visibility via the shared `DurableJson` helper, or a `Properties`-based format for the object-reference store).
2. Write the serialized bytes to a temp file in the **same directory** as the final path.
3. Atomically rename (`StandardCopyOption.ATOMIC_MOVE`, falling back to a non-atomic replace if the filesystem does not support atomic moves) the temp file over the final path.
4. Delete the temp file if it still exists after the move (defensive cleanup).

This guarantees a reader or a crash never observes a torn or partially written document: the final path always contains either the previous complete document or the new complete document, never a mixture. Concurrent writers to the **same** key are serialized through a striped `ReentrantLock` array (typically 64 stripes, selected by `Math.floorMod(key.hashCode(), stripes)`); writers to **different** keys proceed independently. At process startup, each store's constructor performs a `loadAll()` pass over its directory to repopulate an in-memory cache from the durable files, so restart-safety in storage-engine mode does not depend on incremental replay of events.

=== Fixed-Cluster Publication, Identity, and Transport (EP-10 first slice)

[NOTE]
====
Status: *implementation-informed / partial*. This concept is evidenced only for fixed A/B/C and the unconditional whole-object slice in `REQ-CLUSTER-001..005` and `008..013`.
====

The cluster profile composes one Ratis voter and one independent replica data server per JVM. Each node recovers a stable UUID from its identity root and keeps identity, Ratis, object, temporary, and runtime roots separate. A common static A/B/C manifest bootstraps the group but cannot rewrite recovered non-empty state or perform membership changes.

Publication is ordered in two stages: direct data transfer first, authoritative metadata second. A whole-object `PUT` stages immutable replicas outside Ratis, validates length and checksum, and requires `W=2` durable acknowledgements from fixed `N=3`. Only then may the control group commit an object-reference generation. Loss of either threshold fails the S3 operation; unreferenced staged bytes remain unreachable, and the slice has no degraded-write mode. Bucket generations use the same consensus authority without object payload traffic.

Ratis control and replica data use different internal servers, ports, executors, bounds, metrics, and lifecycle ownership. Both use mTLS bound to the expected stable node identity. The grpc-java bridge limits payload frames to 64 KiB and explicitly bounds demand, queues, in-flight frames, deadlines, idle timeout, retries, and terminal cleanup. Test-local certificates establish exercised peer authentication only; production enrollment, custody, rotation, revocation, expiry handling, and recovery remain absent.

This concept does not extend single-node multipart/versioning/EC support into cluster mode. Dynamic membership, clustered multipart, conditional/versioned and chunked writes, erasure coding, rebalance, orphan cleanup, rolling upgrades, and broader partition behavior remain future work. Bounded current-generation repair and bounded periodic current-reference discovery are documented separately below; wider anti-entropy/healing topologies remain absent.

=== Consensus-owned Durable Current-generation Repair (accepted ADR 0029)

[NOTE]
====
Status: *implementation-informed / bounded*. The concept applies only to a missing or corrupt target already named by the current consensus-committed `WHOLE_OBJECT` generation on fixed A/B/C. `REQ-CLUSTER-019..026`, including `024`, are implemented and validated; broad `REQ-CLUSTER-017` remains partial.
====

==== Planning, ownership, and execution

PA-6 `AntiEntropyPlanner` remains deterministic policy. A replica observation, finding, plan, source hint, or `HealingTask` says what ought to happen; it does not own work and cannot establish completion. Cluster application code re-resolves the current committed reference, maps a still-applicable finding into an ensure proposal, and dispatches only committed claims. `RebalancePlanner` is not used because repair restores an existing obligation and does not select or move placement.

Ratis owns durable lifecycle metadata. Workers own direct network/filesystem effects through application ports. The replicated state machine performs deterministic transitions only: no wall-clock reads, RPC, filesystem access, sleeping, or retry loops occur in `applyTransaction`. `ClusterNodeRuntime` owns the process-local scheduler lifecycle; startup and committed-work notifications carry no job data and merely wake bounded queries of committed due work. Scheduler memory is operational state, never work authority.

==== Deterministic identity and immutable specification

One logical job is keyed by:

`(REPAIR, bucket, objectKey, referenceGeneration, artifactId, targetNodeId)`

Ensure/upsert returns the existing job for that key. Candidate source is not part of identity because another named replica may be selected on a later attempt. The immutable specification records the exact committed artifact length/SHA-256, topology/policy epochs, target stable UUID, and reference facts. A newer generation has a different key; an old job is never rewritten to point at it.

==== Consensus lifecycle

[cols="2,5",options="header"]
|===
|State |Meaning

|`READY`
|Applicable committed work is unclaimed and eligible now.

|`CLAIMED`
|One stable node/process session owns the current time-bounded claim and fencing token.

|`RETRY_WAIT`
|A retryable attempt failed; a command-supplied deterministic next-eligible time has not arrived.

|`SUCCEEDED`
|The exact artifact is durably valid at the promised target and success for the current token is committed.

|`BLOCKED`
|Safe automatic progress is unavailable: no verified source remains, an integrity conflict exists, or bounded retries are exhausted. The system emits a repair/integrity alert rather than copying unverified bytes or retrying forever.

|`OBSOLETE`
|The bound generation is no longer current or no longer names the exact artifact/target obligation. This state is terminal.
|===

The normal path is absent → `READY` → `CLAIMED` → `SUCCEEDED`. Retryable failure uses `CLAIMED` → `RETRY_WAIT` → `CLAIMED`; unsafe or exhausted work uses `CLAIMED` → `BLOCKED`; and a reference mismatch moves non-obsolete work to `OBSOLETE`. A fresh observation may re-evaluate a `BLOCKED` or previously `SUCCEEDED` job only while the same reference remains current, without resetting attempt history or claim generation.

==== Claim fencing, retry, and idempotence

Every accepted claim/reclaim increments attempt number and `claimGeneration`. The safety token is exactly `(jobId, claimGeneration)`. A claim also records the stable node UUID and a process-session ID generated once per process start. A restarted process receives a new session and cannot act as its predecessor. Ratis term and claim deadline are liveness information, not fencing tokens.

Renew, fail, block, succeed, and obsolete commands compare the exact current token/session. Duplicate accepted commands return their committed result; stale or non-`CLAIMED` tokens have no state effect. Transport/deadline/transient source or target failures use bounded deterministic backoff. Wrong-checksum sources are excluded; another named source may be tried. No verified source, unsafe target conflict, or exhausted attempts produces `BLOCKED`. A changed reference produces `OBSOLETE`, not a retry.

A worker queries committed state after an uncertain command result. Crash after durable publication but before completion is safe: retry probes the exact target and treats already-valid durable bytes as idempotent success. Token-specific staging and immutable expected content make repeated side effects safe, while only the current token may change job state.

==== Current-generation and direct-transfer invariants

The consensus-selected reference—not local manifests or pairwise comparison—is authority. Ensure validates the current generation and exact target obligation. The worker rechecks currentness before source choice, immediately before target publication, and before success. A source must be a different named replica of the same reference and must verify the complete committed length/SHA-256 during transfer. Missing, corrupt, unavailable, or mismatching replicas are never sources.

Whole-object bytes remain outside Ratis and travel over the existing bounded mTLS replica path. Publication occurs only after complete verification, file fsync, atomic publication/replacement, and parent-directory fsync. Existing exact bytes are success. A corrupt target is replaced only by the verified exact artifact; no unverified payload is returned or copied.

==== GET integrity boundary

Known absence and discovered corruption have intentionally different request behavior:

- **Missing before response:** durably ensure/deduplicate repair, optionally claim inline, fetch and fully verify a different named current-reference replica, publish it durably, and then open the repaired local artifact once for the response. There is no complete local-payload preflight and no second filesystem read.
- **Corruption during the single response pass:** fail that GET, durably ensure repair, and do not transparently retry another replica because bytes may already have reached the client. Scheduling does not make the request successful. A later GET may succeed after repair.
- **Ensure cannot commit:** the corrupt request still fails and an explicit unscheduled-repair alert is required. The system must not claim durable scheduling.

==== Snapshot migration and recovery

The implemented Ratis snapshot schema version 2 explicitly migrates reference-only version 1. Version 2 preserves existing membership/bucket/reference state plus every repair specification, lifecycle state, attempt/dedup history required for audit, owner/session/deadline, attempt number, claim generation, retry policy/eligibility, terminal reason, command timestamps, and last-applied term/index. Version 1 loads with an empty repair map and is written as version 2 at the next snapshot; unknown future versions fail closed. Object payloads and staging files are excluded.

Snapshot/log replay and leader change preserve one logical job and its fences. Scheduler wake-up may reclaim only through a committed newer claim generation. `REQ-CLUSTER-026` validates migration/restoration. On 2026-07-14, `REQ-CLUSTER-024` additionally passed all seven interruption examples with actual B-to-C grpc-java/mTLS reads, token-specific `FileLocalArtifactStore` staging/publication, filesystem byte/hash inspection, exact-target no-recopy reconciliation, stale-token fencing, interrupted version-2 snapshot installation recovered from the last valid snapshot plus log, committed completion with its reply withheld, and live A-to-C leadership transfer. B was independently crashed/restarted with distinct PIDs while retaining its original non-empty roots; A/C voters and source C remained in the parent Cucumber JVM. This is bounded repair evidence, not broad healing or general chaos.

==== Explicit deferrals

Prepared-artifact intents, retirement tombstones, orphan cleanup, rebalance, anti-entropy beyond bounded fixed A/B/C current-reference obligations, adding an unacknowledged third placement, dynamic membership/node replacement, erasure coding, chunk repair, multipart, conditional/versioned operations, and broader/asymmetric partitions require separate requirements and decisions.

=== Bounded Periodic Current-reference Anti-entropy (accepted ADR 0031)

[NOTE]
====
Status: *implementation-informed / bounded*. `REQ-CLUSTER-027` is `@implemented-and-validated` for exactly 2 scenarios / 36 steps. Broad `REQ-CLUSTER-017` and EP-10 remain partial.
====

==== Canonical pages and process-local cursor

Ratis exposes a read-only `ReferencePage` of only current `ObjectReferenceGeneration` records in canonical bucket/object-key order. The positive caller limit is capped at 256; malformed/trailing encodings, oversized pages, inconsistent last-record cursors, and non-terminal empty pages fail closed. The returned cursor is exclusive and identifies the last record. A page sequence reads applied current state; it is not one frozen consensus snapshot.

`ClusterAntiEntropyScheduler` stores that cursor only in process memory. It advances after every page record and required durable ensure completes, remains unchanged on query/probe/ensure failure, resets after a terminal page, and starts before the first page when the scheduler is reconstructed. Repeated first-page work is expected and deduplicates through ADR 0029. No cursor enters the control log or snapshot.

==== Named-obligation inspection and serial bounds

Each fixed node inspects only current references naming its stable UUID. It serially probes one local target at a time for existence, committed length, and SHA-256 on `boundedElastic`. Exact targets cause no repair operation. Missing/corrupt targets delegate to the existing repair coordinator, consensus lifecycle, claim/session fences, retry scheduler, UUID-bound mTLS replica data path, and fsync/atomic publication rules. The discovery cycle owns at most one page and one target probe; overlap is counted and rejected rather than silently widening concurrency.

==== Failure, lifecycle, observability, and authority

Query, probe, or ensure failure aborts the page without cursor progress and is retried by a later periodic cycle. Repair failure remains committed retryable/blocked repair state and is not counted as discovery success. Current-reference races are rejected during ensure or by the existing pre-publication/current-success fences; later cycles discover a new generation under a distinct canonical job. Closing the scheduler cancels delayed work and leaves no active page or target probe.

Process-local status exposes bounded cycle/page/record, exact/missing/corrupt, query/probe/ensure/repair failure, retry, reset, stale, deduplication, overlap, active, and closed counters/flags without payloads, cursor values, or credentials. These values are observability, not durable authority.

PA-6 `AntiEntropyPlanner` findings/plans remain pure planner output and are never execution evidence. Object payloads remain outside Ratis. This concept adds no rebalance, automated orphan cleanup, reference rewriting, wider topology/healing, production-scale fairness/capacity, ADR 0030 general chaos, or production-readiness claim. See link:../../adr/0031-bounded-periodic-current-reference-anti-entropy.md[ADR 0031].

[[concept-reactive-payload-pipeline]]
=== Repository-wide Reactive Payload Pipeline

[WARNING]
====
Status: *implementation-informed / non-conformant in identified paths*. The storage-engine write path has bounded streaming controls, but repository-wide payload streaming and non-blocking thread use are not established. The dated link:../audits/2026-07-14-reactive-streaming-and-pipeline-chaos-source-audit.adoc[source audit] is the bounded implementation-evidence satellite; this section is the central target contract and current status.
====

The reactive contract couples demand, byte bounds, ownership, blocking isolation, integrity, and terminal cleanup. `request(n)` and `limitRate(n)` bound elements, not bytes. Every payload representation therefore needs a finite maximum frame size plus finite demand, prefetch, concurrency, queue, retry, and retained-residual counts. A whole request cached as one `DataBuffer` is one element and defeats count-only byte boundedness.

The target payload discipline is:

- HTTP and application payload paths carry bounded `Flux<DataBuffer>` or an equally explicit transport-neutral bounded-frame abstraction; they do not expose a complete payload as `byte[]`.
- Full-payload `collectList`, array reduction, replay cache, or filter-level materialization is forbidden on payload-plane paths. Deliberately small control-plane documents may be materialized only behind an explicit byte cap.
- Arrays remain valid for cryptographic digests, control-plane codecs, generated protobuf values, and bounded frames such as the current 64 KiB filesystem/gRPC blocks. A `byte[]` occurrence is not a defect without representation and bound analysis.
- Each stage publishes a byte budget covering element size, upstream credit, operator prefetch, concurrent inner stages, scheduler queue, retained residual/composite state, transport in-flight frames, retries, and fault-induced queues.
- The no-aggregation rule applies across authentication, encryption, default/development repositories, transforms, persistence, replica transfer, and response decoration; a streaming storage adapter cannot compensate for an earlier whole-body cache.

At the 2026-07-14 review point, exact signed-payload authentication and SSE-S3 PUT contain complete-payload aggregation in source. Signed-body replay exposure is configuration arithmetic, not measured load evidence or a failed Storage Engine semantic gate; P0 must reproduce the risk in supported compositions before changing prior EP status. The in-memory repository also grows arrays by repeatedly copying the accumulated prefix, but owner authority confirms that Storage Engine is the single-node/default product backend. The in-memory adapter has now been isolated behind `legacy-in-memory-test`; it is not a production-default exception to EP-3 evidence and may be retired separately.

The current single-node filesystem read remains a positive bounded control: `FileSystemStorageNode.streamFrom` opens once and emits at most 64 KiB per array without full aggregation. The application read contract nevertheless exposes `Flux<byte[]>` through `StorageContext` and `ReactiveStorageOrchestrator`; that boundary is planned for migration. Preserve one-open/single-pass GET while changing the contract.

==== DataBuffer ownership and terminal behavior

Every pooled `DataBuffer` has exactly one current owner. A stage must explicitly choose one disposition: transfer ownership downstream, consume and release, or retain with a corresponding later release. Success, error, cancellation, discard, over-limit rejection, ignored-body handling, and injected-fault paths must all terminate ownership exactly once.

Current positive controls include `doOnDiscard(DataBuffer.class, DataBufferUtils::release)` and `finally` release paths in the fixed-window dedup and fixed-stripe EC steps. Current gaps include the over-limit branch in `S3CapacityWebFilter`, whole-array reductions in `InMemoryReactiveS3ObjectRepository`, and the unspecified ignored-body contract of `writeGetObjectResponse`. Runtime leak freedom remains `needs-validation`; static source inspection cannot prove pooled-buffer behavior.

Cancellation is not rollback. Publication remains fenced against cancellation, while idempotent deletion/quarantine of unpublished state is lifecycle-owned asynchronous work. Filesystem cleanup must not run synchronously on an arbitrary Reactor terminal thread. Exactly one terminal outcome must stop demand/production, close channels, release owned/discarded buffers, prevent a false acknowledgement, and make unfinished cleanup visible.

==== Blocking-work bulkheads

Reactive syntax does not make synchronous work non-blocking. Credential/policy reload, tamper-evident audit append/fsync, ACL file access, encrypted inspection copies, filesystem cleanup, replica sink writes, and CPU-heavy transform/EC work must be deferred to finite role-specific executors or schedulers. Each bulkhead must define thread count, queue/task and queued-byte bounds, serialization key, saturation behavior, cancellation/lifecycle ownership, and evidence. `boundedElastic` or an executor name alone is not a complete bound.

The current gRPC server manual demand and dedicated cluster producer/server platform executors are retained. The repair client currently writes each received frame synchronously from its gRPC callback. The planned correction preserves one-message demand, hands a frame to a bounded serialized disk executor, and requests the next frame only after local acceptance. Security file reload/audit and ACL operations similarly require lifecycle-owned snapshots or bounded blocking adapters rather than synchronous calls in reactive security/handler callbacks.

==== Streaming integrity boundaries

Integrity claims name the exact representation and boundary:

- source-stream length/digest describes bytes observed during one subscription;
- transform/ciphertext/shard digests describe their named output representation;
- persisted-byte verification describes a complete reread of unpublished storage bytes;
- committed metadata binds the published artifact to algorithm, representation, length, and digest;
- response-time verification can fail only when mismatch becomes observable and may occur after response commitment; and
- at-rest scrub detects later corruption and is not an every-GET preflight.

`FileSystemStorePort` is a positive control: it streams with `DataBufferUtils.write`, calculates an incremental source digest, fsyncs the temporary artifact, and performs distinct persisted-byte verification before publication. Single-node GET deliberately remains one-open/single-pass and performs no complete payload preflight or second read. `FileSystemStorageNode.streamFrom` currently does not calculate/compare the payload checksum online; this is an explicit limitation, not a claim that EP-3 promised preflight. Any future incremental GET verification must preserve one pass and document the late-failure client contract.

Dedup and EC are bounded but need operational budgets. A permitted 1 GiB dedup window plus copy is not an acceptable default production byte discipline even though it is finite. EC retains a stripe copy plus data/parity arrays and permits `k + m <= 32`; P2 must couple stripe size, shard count, downstream demand, and CPU concurrency to one explicit byte/CPU budget.

[[concept-deterministic-storage-fault-injection]]
=== Deterministic Storage-pipeline Fault Injection (ADR 0030)

[NOTE]
====
Status: *accepted target architecture; planned / needs-validation*. No production implementation, Cucumber requirement, replay result, or fault-safety result is claimed. See link:../../adr/0030-deterministic-storage-pipeline-fault-injection.md[ADR 0030].
====

A transport-neutral application `StorageFaultInjectionPort` provides the sole deterministic decision boundary. Production composition uses a no-op default. The contract contains no `DataBuffer`, protobuf, gRPC, Ratis, `Path`, channel, HTTP, payload, key, credential, or secret type; owning adapters apply a typed decision to exclusively owned bytes or side effects.

Plans use a versioned closed schema for semantic point, representation, selector, action, and outcome. Matching derives from stable plan/rule identity, operation identity, target, point, attempt/occurrence, absolute byte offset, and/or frame index. Every operation, evaluation, evidence, and fire count is finite. Unknown versions, stages, actions, combinations, selectors, or unbounded/overflowing parameters fail activation closed. Process-global randomness, percentages, and schedule-dependent shared seeded draws remain exploratory scaffolding and are not deterministic regression evidence.

Version 1 initially permits only:

- a bit flip after the selected source byte contributed to source checksum/length but before persistence or transform consumption;
- corruption of one unpublished temporary-file byte after writing and before persisted-byte verification; and
- mutation of one received replica-frame byte before sink digest/length update, staging, and verification.

The typed catalog also defines before/after boundaries around checksum calculation, transforms, temp write, persisted verification, file and parent-directory fsync, atomic publication, replica send/receive/sink acceptance, durable-ack emission/acceptance, and Ratis reference proposal/commit. Catalog presence does not enable an action. Delay, cancellation, exception, short I/O, fsync failure, atomic-publication failure, acknowledgement failure, and Ratis failure remain deferred until a later contract version gives each finite, cancellation-aware, before/after semantics.

Activation requires both an explicitly non-production fault-injection profile and a separate opt-in property. A production profile conflict, missing/mismatched node plan, malformed plan, or inability to retain required bounded evidence fails the experiment closed. Evidence includes schema/plan ID and hash, rule/decision ID, operation and opaque target identity, point, representation, occurrence/offset/frame, bounded fire count, fired/skipped reason, and typed outcome. It contains no payload excerpts, object keys, paths, headers, credentials, tokens, certificates, secrets, keys, or free-form exception messages.

`FileSystemWriteFaultInjector`, `PropertyControlledFileSystemWriteFaultInjector`, `ReplicaTransferFaultPlan`, and the exploratory `FaultInjectingStorageCluster`/`ChaosStrategy` are planned to converge behind this port. No legacy mechanism may independently select or report a deterministic fault after migration.

Faults never weaken the normal safety protocol: no degraded writes; fixed `N=3/W=2`; payload bytes outside Ratis; no durable acknowledgement for corrupt/unverified bytes; and no reference proposal before checksum-valid durable acknowledgements and current fencing/epoch checks. An injected terminal condition may leave only unreachable temporary or complete orphaned bytes allowed by the existing publication model; it must never expose a reachable partial artifact or manufacture a durable acknowledgement.

This work is deferred behind active EP-10 normal-path completion and follows the P0-P4 sequence in link:11_technical_risks.adoc#reactive-streaming-remediation-roadmap[section 11] only when activated. It does not replace the EP roadmap. Deferred P0-P3 review supported normal Storage Engine paths; ADR 0030 chaos engineering begins only in final P4.

=== S3 Security Backbone (EP-1)

Opt-in secured S3 mode (`s3.security.enabled=true`) uses Spring Security Reactive / WebFlux Security as the production backbone per ADR 0023. The adapter contributes a `SecurityWebFilterChain` with CSRF, HTTP Basic, form login, logout, and anonymous authentication disabled for secured mode. SigV4 request validation enters through a `ServerAuthenticationConverter` and reactive authentication manager; deny-by-default and explicit-deny authorization enter through a reactive authorization manager; authentication and access-denied failures are serialized as S3 XML errors by Spring Security handlers. === Command/Query DTO Pattern for XML Serialization

The system exposes an XML S3-compatible API where each operation has a structured XML body. Instead of using custom parsers (regex, explicit XmlMapper) or generic "request"/"response" DTOs, the system follows the DDD CQS (Command-Query Separation) pattern:

- **Commands**: represent the intention to change state (PUT, POST, DELETE). They are Jackson XML-annotated DTOs that the framework deserializes directly from the HTTP body. Examples: `CorsConfigurationCommand`, `LifecycleConfigurationCommand`, `DeleteObjectsCommand`
- **Queries**: represent the response to a read request (GET). They are Jackson XML-annotated DTOs that the framework serializes directly into the HTTP body. Examples: `ListBucketsQuery`, `GetBucketCorsQuery`, `ErrorResponse`

Benefits:
- Zero custom parsing: Spring Boot 4 WebFlux deserializes/serializes with registered codecs
- Separate DTOs per operation: each Command/Query lives in its own file, organized by domain
- DDD alignment: Commands express intent, Queries express reads — semantically clear
- Maintainability: modifying an XML tag means modifying a single DTO, not a regex parser and a separate mapping

=== XML Codec Infrastructure

Spring Boot 4 WebFlux handles XML serialization via codecs registered in `JacksonXmlCodecConfig`:

- **JacksonXmlEncoder**: serializes response DTOs into XML
- **JacksonXmlDecoder**: deserializes request body into Command DTOs (added with ADR 0008)
- Both use `tools.jackson.dataformat:jackson-dataformat-xml` (Jackson 3, not Jackson 2)
- DTOs use `tools.jackson.dataformat.xml.annotation.*` annotations (`@JacksonXmlRootElement`, `@JacksonXmlProperty`, `@JacksonXmlElementWrapper`, `@JacksonXmlText`)

The decoder allows `request.bodyToMono(CorsConfigurationCommand.class)` to work for XML — the framework reads the body, passes it to the XML decoder, and produces the annotated record. No explicit call to `XmlMapper`.

=== Command-Query Separation (CQS) Repository Pattern

The application layer separates write and read concerns at the repository level:

- **Command repositories**: `BucketCommandRepository`, `S3ObjectCommandRepository`, `MultipartUploadCommandRepository` — accept Mono<Void> return types for write operations
- **Query repositories**: `BucketQueryRepository`, `S3ObjectQueryRepository`, `MultipartUploadQueryRepository` — return Mono<T> or Flux<T> for read operations

This enables independent optimization of read and write paths, and aligns with CQS principles at the persistence boundary.

=== Handler Minimal Pattern

All handlers, including `S3BucketConfigHandler` (via registry/strategy dispatch onto the durable `Bucket` aggregate), follow the **extract → delegate → response** pattern:

```java
public Mono<ServerResponse> putObject(ServerRequest request) {
    return objectService.saveObjectWithContent(
            S3RequestExtractor.extractObjectKey(request),
            S3RequestExtractor.extractStorageClass(request),
            S3RequestExtractor.extractChecksum(request),
            S3RequestExtractor.extractEncryption(request),
            S3RequestExtractor.extractContentType(request),
            S3RequestExtractor.extractContentLength(request),
            S3RequestExtractor.extractUserMetadata(request),
            request.bodyToFlux(DataBuffer.class))
        .flatMap(result -> S3ResponseBuilder.ok(result.aggregate()))
        .onErrorResume(BucketNotFoundException.class,
            e -> S3WebSupport.xmlError(HttpStatus.NOT_FOUND, "NoSuchBucket", "Bucket not found"))
        .onErrorResume(Throwable.class,
            e -> S3WebSupport.xmlError(HttpStatus.INTERNAL_SERVER_ERROR, "InternalError", e.getMessage()));
}
```

Key constraints:
- **NO** validation — postponed to repository preconditions
- **NO** bucket existence checks — postponed to repository preconditions
- **NO** ETag computation — postponed to repository on store
- **NO** handler-local state — postponed to repository
- **NO** CORS/website logic — postponed to repository/service

=== Reactive Error Handling

Error handling follows the reactive pattern with typed exceptions and S3 XML error serialization:

==== Exception Hierarchy

```

├── NoSuchKeyException        → 404 NoSuchKey
├── BucketAlreadyExistsException → 409 BucketAlreadyExists
```

==== Handler Error Chain

```java
// Handler-level error handling
public Mono<ServerResponse> handleGet(ServerRequest request) {
    return service.findBucket(name)
        .flatMap(bucket -> service.getObject(bucket, key))
        .map(object -> buildResponse(object))
        .onErrorResume(NoSuchBucketException.class, e -> errorResponse(404, "NoSuchBucket"))
        .onErrorResume(NoSuchKeyException.class, e -> errorResponse(404, "NoSuchKey"))
```

=== Domain Events and State Transitions

==== Aggregate Root Pattern

All three aggregates (Bucket, S3Object, MultipartUpload) follow the same pattern:

1. State transitions return NEW immutable instances — no mutation
2. Each transition records a domain event in an accumulated list
3. `domainEvents()` retrieves pending events
4. `clearEvents()` returns a clean instance ready for persistence

==== State Transition Rules

- State transitions return NEW immutable instances with accumulated events
- `domainEvents()` retrieves pending events; `clearEvents()` returns a clean instance for persistence

Each event carries the specific value object type (e.g., `BucketLifecycleConfiguration`, `BucketNotificationConfiguration`) so consumers can react to precise changes.

=== AWS CLI Header Compatibility

The system must handle headers that the AWS CLI v2 sends by default on every `aws s3api put-object` request. Phase 9 first-increment AWS CLI Cucumber now verifies that `put-object` succeeds with the default AWS CLI headers. Compatibility is still partial: checksum headers are accepted/ignored for this scope, while SigV4 validation exists only when the opt-in `s3.security.enabled=true` secured mode is active.

==== Default Headers Sent by AWS CLI v2

[cols="2,2,3,3",options="header"]
|===
|Header |Value |Purpose |Handler Status

|`x-amz-sdk-checksum-algorithm` |`crc64nvme` (default) |Identifies the checksum algorithm used |[yellow]#Accepted; not validated#
|`x-amz-checksum-crc64nvme` |BASE64 CRC64-NVME hash |Payload integrity hash computed automatically by AWS CLI |[yellow]#Accepted; not validated#
|`X-Amz-Date` |ISO 8601 timestamp |SigV4 signing timestamp |[yellow]#Accepted without validation in default unsecured mode; validated for clock skew in opt-in secured mode#
|`X-Amz-Content-SHA256` |SHA256 hex string or `UNSIGNED-PAYLOAD` |SigV4 payload hash |[yellow]#Accepted without validation in default unsecured mode; exact SHA256 values are body-verified in opt-in secured mode after SigV4 signature validation; `UNSIGNED-PAYLOAD` remains accepted for trusted clients#
|`Authorization` |SigV4 signature string |SigV4 signature |[yellow]#Accepted without validation in default unsecured mode; verified against configured in-process credentials in opt-in secured mode#
|`User-Agent` |SDK identifier string |Client identification |[yellow]#Accepted; not used#
|`Expect` |`100-continue` |Hop-by-hop header for large payloads |[yellow]#Accepted; not optimized#
|===

==== Checksum Algorithm Mapping

When `--checksum-algorithm <alg>` is specified, AWS CLI v2 **automatically computes** the hash and sends it in the corresponding header:

[cols="2,2,3",options="header"]
|===
|CLI `--checksum-algorithm` |Header Sent |Automatic Computation

|`crc64nvme` (default) |`x-amz-checksum-crc64nvme` |[green]#Yes# — AWS CLI computes CRC64-NVME
|`crc32` |`x-amz-checksum-crc32` |[green]#Yes#
|`crc32c` |`x-amz-checksum-crc32c` |[green]#Yes#
|`sha1` |`x-amz-checksum-sha1` |[green]#Yes#
|`sha256` |`x-amz-checksum-sha256` |[green]#Yes#
|`sha512` |`x-amz-checksum-sha512` |[green]#Yes#
|`xxhash64` |`x-amz-checksum-xxhash64` |[green]#Yes#
|`xxhash3` |`x-amz-checksum-xxhash3` |[green]#Yes#
|`xxhash128` |`x-amz-checksum-xxhash128` |[green]#Yes#

The server must read `x-amz-sdk-checksum-algorithm`, re-compute the hash of the received body with that algorithm, and compare against the corresponding `x-amz-checksum-<alg>` header. A mismatch must return `400 InvalidArgument (ChecksumMismatch)`.

==== Content-Type Handling

AWS CLI v2 does NOT send `Content-Type` by default unless `--content-type` is explicitly passed. The handler currently falls back to `application/octet-stream`. When AWS CLI is used, the stored content-type is incorrect — the object will be served with `application/octet-stream` instead of the actual content type.

=== Future Cluster Concepts Beyond the Fixed Slice (ADR 0027)

[NOTE]
====
Status: *planned / not implemented beyond the bounded ADR 0028, ADR 0029, and ADR 0031 slices*. The implemented identity, transport, publication, current-generation repair, and fixed A/B/C periodic current-reference discovery subsets are documented above; this section records only the remaining target.
====

**Membership and authority.** Consensus-controlled admission, catch-up, promotion, demotion, replacement, removal, topology/policy epochs, fencing, tombstones, wider namespace/configuration generations, and durable jobs beyond bounded repair remain future work. Health suspicion must not become membership mutation.

**Transfer and reads.** Chunked, multipart, conditional/versioned, and EC transfer/reconstruction are not implemented in cluster mode. Bounded current-generation whole-object repair is implemented under ADR 0029, and bounded periodic inspection of fixed A/B/C current named obligations is implemented under ADR 0031. Wider anti-entropy/healing topologies, rebalance, and automated orphan cleanup remain absent rather than inferred from those slices or from `N=3/W=2` replication.

**Operations and security.** Production certificate enrollment/rotation/revocation/recovery, rolling and mixed-version upgrades, broader partition handling, online multi-node backup/DR, and deployment hardening require separate evidence. Test-local mTLS and one-coordinator-stop validation do not satisfy these obligations.

**Policy separation.** Future cluster adapters must continue executing authoritative decisions without moving Ratis/protobuf/Reactor/persistence concerns into `DistributedPlacementPlanner`, `QuorumPolicy`, `AntiEntropyPlanner`, `RebalancePlanner`, or `DistributedReadinessReporter`.
