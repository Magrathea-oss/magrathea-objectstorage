== Cross-cutting Concepts

=== Hexagonal Architecture (Ports and Adapters)

Il sistema segue il pattern hexagonal architecture con tre layer:

- **Dominio (core):** puro business logic, zero framework dependencies
- **Applicazione:** orchestrazione, bridge CompletableFuture ↔ Mono/Flux
- **Infrastruttura:** adapter HTTP (RouterFunction), adapter persistenza (in-memory)

Le dipendenze vanno solo verso l'interno: infrastruttura → applicazione → dominio.

=== Reactive Non-blocking I/O

Tutta la gestione delle richieste HTTP è reattiva:

- RouterFunction invece di @Controller
- Flux<DataBuffer> per upload/download binario (mai byte[] nei handler)
- Mono.fromFuture() per bridge CompletableFuture → reattivo
- Schedulers.boundedElastic() solo per operazioni che consumano il flusso in memoria

=== Domain Purity

Il modulo `object-storage-domain` è rigorosamente puro:

- NO Spring imports
- NO JPA, NO jakarta.validation
- NO reactive types (Mono, Flux, DataBuffer)
- Repository interfaces con CompletableFuture (standard Java async)
- S3ObjectWrite/S3ObjectContent senza tipi framework

=== S3 API Compatibility

Il sistema espone SOLO endpoint AWS S3-compatible:

- Stessa firma HTTP (metodi, header, query parameters)
- Stesso formato XML (ListAllMyBucketsResult, ListBucketResult, Error, ecc.)
- Stessa codifica degli errori (NoSuchBucket, NoSuchKey, BucketAlreadyExists)
- Pluggable via Spring Boot auto-configuration (s3-api si attacca come modulo opzionale)
- Toggle con `s3.api.enabled=false`

=== Pluggable S3 API Module

`s3-api` è un modulo auto-configurante che si innesta nel contesto Spring Boot:

- `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
- `S3ApiConfig` guardato da `@ConditionalOnClass({ BucketService.class, ObjectService.class })`
- `s3.api.enabled=true` per default
- Se `s3-api` non è nel classpath, le route S3 non vengono caricate

=== Testing Strategy

Tre livelli di test:

1. **Dominio:** JUnit puro, zero Spring — testano invarianti e regole di business
2. **Integrazione S3:** Cucumber BDD con WebTestClient — verificano endpoint HTTP
3. **Compatibilità AWS CLI:** `test-aws-cli.sh` script bash + `mvn verify -Paws-cli-tests`
