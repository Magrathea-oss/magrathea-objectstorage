[[section-solution-strategy]]
== Solution Strategy

=== Architecture Strategy

- **Hexagonal Architecture** (Ports and Adapters): domain is pure business logic, application and infrastructure are adapters
- **DDD with strategic design**: separate Object Store and Storage Engine bounded contexts, connected only through the object-store-to-storage-engine ACL adapter
- **Reactive by design**: WebFlux RouterFunction, DataBuffer streaming, native Mono/Flux application and repository interfaces

=== Key Technical Decisions

[cols="2,3",options="header"]
|===
|Decision |Rationale

|Maven multi-module per bounded context
|Enforces dependency boundaries, domain isolation

|Records for aggregates/value objects
|Immutability, zero boilerplate, Java 21 idiomatic

|Sealed interfaces for domain events
|Exhaustive pattern matching, type safety

|Reactive repository interfaces
|Native Mono/Flux command/query APIs for non-blocking service composition

|RouterFunction over @Controller
|Functional style, precise content negotiation

|Jackson XML for S3 responses
|Auto-serialization, no StringBuilder XML

|DataBuffer for binary streaming
|Reactive non-blocking I/O, WebFlux native

|Crash-safe temp-file + atomic-rename commits for durable metadata
|Readers and restart recovery never observe a torn document (`BucketStore`, `MultipartUploadStateStore`, `ObjectConfigMetadataStore`, `S3ObjectManifestReferenceStore`)

|Striped per-key locking for durable-metadata writers
|Serializes concurrent same-key mutation without serializing unrelated keys

|Repository-wide bounded payload and role-specific bulkheads
|Makes byte bounds, pooled-buffer ownership, terminal cleanup, filesystem/crypto/CPU scheduling, and integrity scopes one end-to-end contract rather than local reactive syntax (`planned / needs-validation`)

|Transport-neutral deterministic storage fault injection (ADR 0030)
|Provides replayable typed semantic points and bounded evidence without coupling application contracts to WebFlux, filesystem, gRPC, or Ratis. The exact committed-EC-shard test subset is planned before self-healing; generalized chaos remains final (`accepted target; implementation planned`)
|===

=== Bucket Configuration Redesign (ADR 0011)

ADR 0011 records a course correction: each configuration feature (CORS, Lifecycle, Encryption, etc.) is modeled as a dedicated value object on the Bucket aggregate with specific `with*()` / `without*()` methods and a specific domain event type per config change, replacing the earlier flat `Bucket.Configuration` record.

=== Reactive Migration (ADR 0010)

ADR 0010 records the API/composition migration from ADR 0009: application services return `Mono<T>` or `Flux<T>` natively and handlers chain reactive operators without future `.join()` bridges. That historical decision does not establish repository-wide byte boundedness, pooled-buffer ownership, or event-loop safety. The 2026-07-14 static review identified full-payload aggregation in signed authentication and SSE-S3 plus synchronous security/ACL/cleanup/repair I/O on reactive or callback paths; these are source-review risks that require reproduction in supported Storage Engine compositions, not failed semantic gates. Quadratic accumulation in the legacy in-memory adapter is isolated `legacy-in-memory-test` scope and does not define the single-node backend. See link:../audits/2026-07-14-reactive-streaming-and-pipeline-chaos-source-audit.adoc[the bounded source audit].

[[solution-reactive-streaming-chaos]]
=== Repository-wide Streaming and Pipeline-chaos Strategy (ADR 0030)

[NOTE]
====
Status: *accepted sequencing / needs-validation*. EP-10 remains active. After distributed shard placement/transfer, only the bounded no-op-default ADR 0030 kernel and exact committed-shard unavailable/corruption actions move before EC self-healing. Generalized deterministic and realistic chaos remains final. No fault implementation or evidence is claimed.
====

The target establishes bounded payload frames and explicit buffer ownership from S3 ingress through security, transform, persistence, replica transfer, and response egress. Blocking filesystem/crypto/policy/audit and CPU-intensive transform work use finite role-specific bulkheads. Integrity continues to distinguish source-stream checksums, transformed representation checksums, persisted-byte verification, committed metadata, single-pass response behavior, and at-rest scrub.

After source-review risks are reproduced and confirmed against supported Storage Engine compositions, conservative corrections and static architecture gates establish allowed payload representations and byte-budget formulas. Transform/dedup/EC work then proceeds under those contracts, followed by normal-path semantic, load, pooled-buffer, cancellation, and saturation evidence.

The shared `StorageFaultInjectionPort` kernel remains transport-neutral, no-op by default, dual-guarded for non-production use, deterministic, finite, fail closed, and payload/secret free in evidence. Its planned early EC subset may resolve only exact schema-3 committed shard identities, quarantine or corrupt at most one finite stripe under explicit recoverable/`EXPECT_BLOCKED` blast-radius rules, and restore pristine bytes during lifecycle-owned teardown. It cannot target raw paths, metadata, unrelated objects, or arbitrary faults. This tooling enables self-healing acceptance but does not implement detection, consensus jobs, transfer, or publication.

Broader source/temp/replica actions, delay/cancellation/failure actions, network/process disruption, interleavings, and realistic generalized chaos remain final after normal-path closure. See link:../../adr/0030-deterministic-storage-pipeline-fault-injection.md[ADR 0030] for normative details.

=== Bounded Local Erasure-coded Reconstruction (accepted ADR 0032)

[NOTE]
====
Status: *implementation-informed / implemented-and-validated*. All five `REQ-PIPELINE-017` scenarios pass in the focused 5-scenario / 46-step gate, including all 15 four-of-six EC 4+2 survivor combinations.
====

New reconstructable EC writes use manifest schema 3. Each typed data/parity shard binds explicit stripe index, shard index, `k`, `m`, parity role, logical stripe length, original/stored length, lowercase SHA-256, and ordered transport-neutral location identity. Schemas 0/1/2 retain their read compatibility paths, but schema-2 EC metadata without explicit layout is not reconstructed by inference.

`BoundedEcReconstructionPort` accepts committed facts and caller-resolved checksum-valid bytes for exactly one stripe. The stateless `BoundedEcReconstructionAdapter` validates metadata/survivors, performs GF(256) work on an injected one-worker/16-queue scheduler, verifies every regenerated shard against committed length/SHA-256, and returns only a verified logical stripe plus regenerated-shard output. Decoder-owned retention is at most one 4 MiB stripe, six 1 MiB buffers, and fixed matrix/index workspace; earlier-stripe and whole-object retention remain zero.

The decoder has no repository or publication capability. It neither replaces a shard nor creates a scanner/daemon, cluster transfer, Ratis repair job, rebalance, cleanup, or fault. Fixed distributed placement/transfer is a separate subsequent mechanism under ADR 0033. See link:../../adr/0032-bounded-local-erasure-coded-reconstruction.md[ADR 0032].

=== Fixed Distributed EC 4+2 Placement and Transfer (accepted ADR 0033)

[NOTE]
====
Status: *implementation-informed / implemented-and-validated for bounded internal `REQ-CLUSTER-015`*. The focused gate passes 5 scenarios / 40 steps. Clustered S3 EC reads and self-healing remain absent.
====

EC configuration is deliberately fixed to 4+2 until a later parameterized codec/placement decision. For the one-stripe bounded publication, committed A/B/C membership assigns A=`0,3`, B=`1,4`, and C=`2,5`; one node loss therefore leaves four authoritative shard identities. `ClusterEcWriteCoordinator` incrementally verifies local shard and object checksums, transfers four remote shards over bounded grpc-java/mTLS streams, and requires six exact durable acknowledgements before publication.

Ratis commits one `EC_4_2` reference containing object facts and all schema-3 shard locations only after current-generation and epoch validation. Payload remains outside Ratis, and snapshot-v2 restart restores the same reference. Missing, checksum-conflicting, or stale evidence publishes nothing and never falls back to whole-object replication or a reduced shard set.

This is an internal placement/transfer/publication mechanism, not clustered S3 read integration, survivor collection, shard repair, monitoring, rebalance, cleanup, dynamic membership, or generalized chaos. See link:../../adr/0033-fixed-distributed-ec-4-2-placement-and-transfer.md[ADR 0033].

=== Handler Minimal Pattern

All handlers, including `S3BucketConfigHandler` (via its registry/strategy dispatch onto the Bucket aggregate), follow the **extract → delegate → response** pattern:

1. **Extract**: `S3RequestExtractor` parses HTTP headers, path variables, and query parameters into typed primitives and DTOs
2. **Delegate**: The handler calls a single reactive service method — no validation, no bucket checks, no ETag computation
3. **Response**: `S3ResponseBuilder` or direct XML serialization converts the service result into an HTTP response

All cross-cutting concerns (validation, bucket existence checks, CORS, website routing, ETag) are **postponed to the repository layer**.

=== Registry-Based Handler Dispatch

The `S3BucketConfigHandler` uses a registry/strategy pattern to dispatch configuration operations without copypasta. Each config type has a `ConfigStrategy` entry in an internal registry map; adding a new config type requires only one strategy registration. As of the EP-2 metadata durability work, **all bucket configuration families (including ABAC, bucket metadata configuration, bucket metadata-table configuration, object-lock, inventory-table, and journal-table) are modeled as value objects on the `Bucket` aggregate** and persisted through `ReactiveBucketService.updateBucket(...)`. The handler holds no configuration state of its own; in storage-engine mode, every family is durable via `BucketStore`.

=== Reusable Admin UI Strategy (ADR 0026)

[NOTE]
====
Status: *implementation-informed*. The Product Shell/Product Extension split and the `REQ-ADMIN-001..031` bounded scope are implemented and validated. EP-7 overall remains partial because credential/tenant administration and real operational report providers are absent.
====

The reusable **Magrathea Product Shell** provides only the product-neutral frame, navigation/layout, **Design Tokens**, **Shell Primitives**, localization infrastructure, route/extension composition, documentation integration, and common UI states. Product behavior is supplied through explicit **Product Extension** contracts; static boundary and deterministic extension-removal tests enforce the separation.

The **Object Storage Admin Application** is the Product Extension owned by this repository. It owns every Object Storage route, screen, model, permission, endpoint path, API client, and response mapping. Its implemented screens use the **Admin Control Plane** for health/readiness, backend and catalog status, read-only configuration, validation, capacity/quota, and report availability. Object and bucket semantics remain on the **S3 Data Plane**; the optional HeadObject diagnostic calls S3 as an ordinary client and has no storage-engine bypass.

Operational report availability is fail-closed: without a configured recovery, garbage-collection, scrub, audit, metrics, or traces provider, the route returns HTTP 503 `report-provider-not-configured` and the UI renders `not-configured`. This is a truthful contract, not provider implementation.

=== Bounded Authoritative Cluster Strategy (ADRs 0027 and 0028)

[NOTE]
====
Status: *implementation-informed / partial*. The fixed A/B/C first slice is implemented and validated; EP-10 as a whole is not complete and no distributed production-readiness claim follows.
====

The implemented slice separates an authoritative **Cluster Control Plane** from the direct **Cluster Data Plane** while co-locating one Ratis voter and one replica server in each of three JVMs. Stable UUIDs and separate identity, Ratis, object, temporary, and runtime roots survive a complete restart. An identical static A/B/C manifest bootstraps the group but does not implement admission, replacement, removal, or any other dynamic membership transition.

For unconditional `CreateBucket`, whole-object `PUT`, and whole-object `GET`, Apache Ratis orders bucket and object-reference generations. Whole-object bytes remain outside the Ratis log and transfer directly over independent grpc-java services with mTLS and bounded manual flow control. Replication is fixed at `N=3, W=2`: success requires two checksum-valid durable replica acknowledgements followed by consensus publication, and the implementation does not degrade below either data or control quorum.

The Java 21 shared real-process WebTestClient/AWS CLI gate validates `REQ-CLUSTER-001..005`, `019`, and `020` in both modes (14 scenarios, 188 steps). The original first-slice subset accounts for 10 scenarios / 108 steps: cross-node bucket visibility, exact-byte read after coordinator A stops, failure below data/control quorum, and complete A/B/C restart. Focused real Ratis, gRPC, mTLS, flow-control, cancellation/deadline, and cross-module gates validate `REQ-CLUSTER-008..013`. Ordinary root Maven success is supporting integration evidence, not a substitute for these semantic gates.

On 2026-07-14, the separate `ReqCluster014ArchitectureContractCucumberTest` passed 1 scenario / 17 steps in `REQ-CLUSTER-014`'s sole internal validation mode. It parsed 17 reactor modules, 87 direct non-test Maven dependencies plus 2 profile declarations, 473 tracked production Java sources through JDK ASTs, and one versioned proto3 contract; executed the layering script; and passed 12 missing/empty/malformed plus 4 supported POSIX unreadable fail-closed probes with exact-path and protected-input hash preservation. Coordination 7/54, control 2/19, data 4/40, repair 22/294, EP-8 architecture 13/103, source hygiene, and appendix freshness regressions passed. CI is wired but has not run. This validates source and build boundaries only; it is not S3 behavior and proves no runtime side effect.

The existing Object Store-to-Storage Engine adapter remains the only S3 integration boundary, and PA-6 placement/quorum/planning types remain pure domain policy. `REQ-CLUSTER-014` is implemented and validated only for that repository-rooted architecture contract. Clustered multipart, conditional/versioned or chunked writes, distributed erasure coding, dynamic membership and certificate lifecycle, rolling upgrades, and broader partition/fault behavior remain not implemented (`REQ-CLUSTER-006/007`, `015/016/018`). Local ADR 0032 reconstruction does not change those cluster statuses. Broad `REQ-CLUSTER-017` remains partial: bounded current-generation repair and bounded periodic current-reference anti-entropy on fixed A/B/C exist, while rebalance, automated orphan cleanup, and wider healing/topology coverage remain absent. See link:../../adr/0027-authoritative-cluster-control-plane-and-direct-quorum-data-path.md[ADR 0027] for the wider authority and link:../../adr/0028-first-three-node-cluster-implementation-baseline.md[ADR 0028] for the accepted baseline.

=== Consensus-owned Durable Current-generation Repair (accepted ADR 0029)

[NOTE]
====
Status: *implementation-informed / bounded*. `REQ-CLUSTER-019..026`, including `024`, are `@implemented-and-validated`; `REQ-CLUSTER-014` is separately `@implemented-and-validated` only for its internal repository architecture mode. Broad `REQ-CLUSTER-017` remains `@partial`; fixed internal EC 4+2 `015` is separately validated under ADR 0033, while `006/007/016/018` remain `@not-implemented`. No production-readiness, S3 behavior, runtime-side-effect, or broad healing claim follows from `014`.
====

The bounded EP-10 strategy restores only an obligation already present in the current consensus-committed `ObjectReferenceGeneration`: a named target whose exact `WHOLE_OBJECT` artifact is missing or corrupt. It neither changes placement nor publishes a new generation. Application code maps still-applicable evidence into one deterministic repair job; Ratis owns `READY`, `CLAIMED`, `RETRY_WAIT`, `SUCCEEDED`, `BLOCKED`, and `OBSOLETE` lifecycle facts; and an application worker performs direct verified transfer and crash-safe target publication outside consensus.

The implementation uses the semantic key `(REPAIR, bucket, objectKey, referenceGeneration, artifactId, targetNodeId)` for ensure/deduplication. Source selection remains an attempt hint. Monotonic claim generation plus stable node/process-session ownership fences stale workers across retries, restarts, and leader changes. Workers revalidate the current generation before source selection, publication, and success; generation changes make the old job `OBSOLETE` rather than rewriting it.

The scheduler is process-local wake-up machinery, not job authority. `ClusterNodeRuntime` starts and stops it with the node process; startup and committed-work signals carry no job ownership and cause bounded queries of consensus state. The scheduler serially dispatches only committed work for the local target, while Ratis remains the durable lifecycle authority across process restart and leader change.

GET preserves the single-pass integrity contract. Known absence before response streaming is repaired from a different named, fully verified replica and then served from one local filesystem read. Corruption discovered while the present local payload is streamed fails that request, durably schedules repair, and permits only a later GET to succeed. There is no complete local-payload preflight, second filesystem read, or transparent corrupt-read retry.

Snapshot schema version 2 persists job metadata and explicitly migrates reference-only version 1; payload bytes do not enter snapshots. The 2026-07-13 evidence remains unchanged: the shared Java 21 real-process gate passed 14 scenarios / 188 steps for `REQ-CLUSTER-001..005`, `019`, and `020`; the focused repair-only subset passed 4 / 80; repair-control `021..026` passed 22 / 210; and the data-plane regression passed.

On 2026-07-14, focused `ReqCluster024CucumberTest` passed 7 scenarios / 168 steps. B ran as an independently crashable/restarted JVM with distinct PIDs and its original non-empty identity, Ratis, and filesystem roots, while the A/C voters and source-C real grpc-java server remained in the parent Cucumber JVM. Every interruption point exercised actual B-to-C grpc-java reads with UUID-bound mTLS, token-specific `FileLocalArtifactStore` staging/publication, filesystem byte/hash inspection, exact-target no-recopy reconciliation, stale-token fencing, interrupted version-2 snapshot installation with last-valid-snapshot-plus-log recovery, completion committed with its reply withheld, and live A-to-C leadership transfer. The same validation set passed repair-control 22 / 294, data-plane 4 / 40, control/TLS regressions, module compilation, layering, source hygiene, and diff checks.

This is bounded `024` evidence, not `027` discovery evidence, general chaos, broad partition tolerance, rolling upgrade, dynamic membership, rebalance, orphan cleanup, or production readiness. Prepared-artifact intents and orphan cleanup, rebalance, anti-entropy beyond current named A/B/C obligations, dynamic membership, erasure coding, multipart, and broader partition behavior remain separate future decisions. See link:../../adr/0029-consensus-owned-durable-repair-for-current-whole-object-generation.md[accepted ADR 0029]. The earlier architecture evidence was reviewed on 2026-07-13 against dirty working-tree base `3eb2a508317a`; the domain-expert wiki page `analyses/ep-10-fixed-cluster-durable-repair-and-orphan-cleanup-slice.md` remains research/recommendation input, not a requirement or implementation claim.

=== Bounded Periodic Current-reference Anti-entropy (accepted ADR 0031)

[NOTE]
====
Status: *implementation-informed / bounded*. Both `REQ-CLUSTER-027` scenarios are `@implemented-and-validated`; the focused gate passes 2 scenarios / 36 steps. Broad `REQ-CLUSTER-017` and EP-10 remain partial.
====

Each fixed node runs one process-local discovery scheduler under `ClusterNodeRuntime`. Ratis serves only current `ObjectReferenceGeneration` records in canonical bounded pages with an exclusive last-record cursor. Page records are inspected serially, only when the committed replica set names the local UUID; local filesystem probes verify committed length/SHA-256 on `boundedElastic`. Exact targets cause no work, while missing/corrupt targets delegate to ADR 0029's existing consensus-owned canonical job, claim fencing, retry, grpc-java UUID-bound mTLS transfer, and crash-safe publication path.

The cursor advances only after every record and required durable ensure in the page completes. Query, probe, or ensure failure leaves it unadvanced for a later cycle; a terminal page and process reconstruction both restart from the first page. The cursor is never consensus state, pages are not one frozen multi-page snapshot, payload is absent from Ratis, and repeated observations deduplicate through the repair identity. Bounded counters expose cycle/page/record/target/failure/retry/reset/stale/dedup/overlap state without payloads or credentials.

This strategy makes no PA-6 `AntiEntropyPlanner` execution claim and does not implement rebalance, automated orphan cleanup, wider healing/topologies, dynamic membership, production scale/readiness, or ADR 0030 faults. See link:../../adr/0031-bounded-periodic-current-reference-anti-entropy.md[accepted ADR 0031].

The active continuation is distributed EC shard placement/transfer, the planned bounded ADR 0030 committed-shard test subset, EC self-healing, shard rebalance, and fenced cleanup; generalized chaos remains final. Whole-object `N=3/W=2` replication remains valid below the arranged EC minimum. Inter-cluster/NAS-backed replication remains later and NAS-dependent.

=== Quality Strategy

- **Testability**: domain tested with pure JUnit (no Spring), integration tested with Cucumber BDD, targeted S3 compatibility verified with AWS CLI; the first AWS CLI object CRUD increment passes, but full S3 scenario parity remains open
- **Durable backend authority**: Storage Engine is the default and only supported single-node product backend; legacy in-memory adapters remain isolated behind the explicit `legacy-in-memory-test` profile and may be retired separately
- **Traceability**: ADRs document every architectural decision, ARC42 provides full architecture documentation
