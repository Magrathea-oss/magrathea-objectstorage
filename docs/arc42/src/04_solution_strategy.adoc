== Solution Strategy

=== Architecture Strategy

- **Hexagonal Architecture** (Ports and Adapters): domain is pure business logic, application and infrastructure are adapters
- **DDD with strategic design**: separate Object Store and Storage Engine bounded contexts, connected only through the object-store-to-storage-engine ACL adapter
- **Reactive by design**: WebFlux RouterFunction, DataBuffer streaming, native Mono/Flux application and repository interfaces

=== Key Technical Decisions

| Decision | Rationale |
|----------|-----------|
| Maven multi-module per bounded context | Enforces dependency boundaries, domain isolation |
| Records for aggregates/value objects | Immutability, zero boilerplate, Java 21 idiomatic |
| Sealed interfaces for domain events | Exhaustive pattern matching, type safety |
| Reactive repository interfaces | Native Mono/Flux command/query APIs for non-blocking service composition |
| RouterFunction over @Controller | Functional style, precise content negotiation |
| Jackson XML for S3 responses | Auto-serialization, no StringBuilder XML |
| DataBuffer for binary streaming | Reactive non-blocking I/O, WebFlux native |

=== Bucket Configuration Redesign (ADR 0011)

ADR 0011 records a course correction: each configuration feature (CORS, Lifecycle, Encryption, etc.) is modeled as a dedicated value object on the Bucket aggregate with specific `with*()` / `without*()` methods and a specific domain event type per config change, replacing the earlier flat `Bucket.Configuration` record.

=== Reactive Migration (ADR 0010)

ADR 0010 records a course correction that completed the reactive migration from ADR 0009. All application services now return `Mono<T>` or `Flux<T>` natively (no `.join()`), handlers chain reactive operators directly (no `.subscribeOn()`), and repository implementations are fully reactive (no `Mono.fromCallable` stubs). Runtime effects (CORS validation, Website routing, OwnershipControls enforcement, PublicAccessBlock evaluation, Accelerate headers, RequestPayment enforcement) were added.

=== Handler Minimal Pattern

All handlers (except `S3BucketConfigHandler`) follow the **extract → delegate → response** pattern:

1. **Extract**: `S3RequestExtractor` parses HTTP headers, path variables, and query parameters into typed primitives and DTOs
2. **Delegate**: The handler calls a single reactive service method — no validation, no bucket checks, no ETag computation
3. **Response**: `S3ResponseBuilder` or direct XML serialization converts the service result into an HTTP response

All cross-cutting concerns (validation, bucket existence checks, CORS, website routing, ETag) are **postponed to the repository layer**.

=== Registry-Based Handler Dispatch

The `S3BucketConfigHandler` uses a registry/strategy pattern to dispatch configuration operations without copypasta. Each config type has a `ConfigHandlerStrategy` implementation registered in `ConfigHandlerRegistry`; adding a new config type requires only one strategy class and its registration.

=== Quality Strategy

- **Testability**: domain tested with pure JUnit (no Spring), integration tested with Cucumber BDD, targeted S3 compatibility verified with AWS CLI; the first AWS CLI object CRUD increment passes, but full S3 scenario parity remains open
- **Simplicity and explicit backend selection**: single-node in-memory stores remain the default backend; Storage Engine is selected explicitly and must be verified separately
- **Traceability**: ADRs document every architectural decision, ARC42 provides full architecture documentation
