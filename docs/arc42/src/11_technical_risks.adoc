== Risks and Technical Debts

This chapter is the central mutable architecture risk and remediation-status source. `PLAN.md` may schedule work, and dated audits remain bounded evidence satellites, but neither supersedes the risk boundaries recorded here. An earlier EP capability status does not close a newly discovered cross-cutting quality risk; closure requires the specific semantic, load, leak, cancellation, and fault evidence named below.

==== Closure rules

- Static review does not revoke prior semantic evidence. P0 changes status only when a supported Storage Engine composition reproduces a regression or confirms a contract violation.
- The owner-confirmed single-node/default backend is Storage Engine. Legacy in-memory adapters are test/retirement scope, not an alternate product backend.
- P1 static counts cannot close a semantic risk; they prevent regression and define the finite budget that P3 must measure.
- P2 must preserve the positive controls: no production `DataBufferUtils.join`/`block*`, one-open/single-pass GET, `DataBufferUtils.write` plus incremental digest/persisted verification, 64 KiB cluster frames, manual gRPC demand, dedicated executors, and chunker discard release.
- P3 closes the normal path before any deterministic chaos implementation begins.
- P4 deterministic hook evidence cannot substitute for realistic filesystem/network/process fault evidence, and exploratory random chaos cannot substitute for replay evidence.
- A phase closes only on the exact implementation and validation evidence in its exit criteria. Unknown, skipped, stale, or runner-missing evidence remains `needs-validation`; no test result is inferred from this plan.
