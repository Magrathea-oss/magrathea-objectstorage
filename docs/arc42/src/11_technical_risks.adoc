[[section-technical-risks]]
== Risks and Technical Debts

Risk status is cross-referenced with the Enterprise Production Readiness Plan (`PLAN.md`, EP-0..EP-11) so that mitigation ownership stays traceable. No risk below is closed until the referenced EP phase is `@implemented-and-validated`.

[cols="1,4,3,4,2",options="header"]
|===
|# |Risk |Impact |Mitigation |EP reference

|1
|**Bare local default backend is in-memory** — the unpackaged `single-node` profile's reactive in-memory repositories do not persist to disk. The Storage Engine backend (`storage-engine` profile) now durably persists object bytes/manifests/references AND (as of EP-2) the bucket registry, multipart upload state, and per-object/bucket configuration. Packaged JVM/native single-node containers activate `storage-engine` by default with packaged YAML catalogs.
|Data loss on restart when operators bypass the packaged container defaults and run the bare local `single-node` profile
|Keep the in-memory profile explicitly documented as development/test compatibility; require packaged single-node deployment paths to activate storage-engine and keep CI Docker readiness expecting configured catalogs
|EP-2 (`@implemented-and-validated` for the declared storage-engine metadata durability scope; the in-memory profile remains explicitly exempt for dev/test compatibility)

|2
|**S3 semantic depth** — the in-scope route surface is mapped, but route count is not semantic completion. AWS CLI parity has improved for the canonical object CRUD subset and metadata durability, but full S3 scenario parity is incomplete.
|AWS SDK features may expect deeper semantics than currently implemented
|Track semantic gaps in `PLAN.md`; prioritize AWS CLI/SDK compatibility cases and keep unsupported cases explicitly tagged (`@not-implemented`, `@partial`, `@config-only`)
|Ongoing (S3 API Semantic Completion Plan, `PLAN.md`)

|3
|**Streaming remains bounded to the validated preview envelope** — EP-3 removed full-object aggregation and introduced typed whole-object/dedup/EC artifacts; EP-6 now enforces and validates the 256 MiB object envelope under `-Xmx256m`.
|Larger objects, higher concurrency, or changed limits may exceed the tested memory/latency envelope even though the streaming architecture remains bounded
|Treat configured EP-6 bounds as preview safety limits; rerun capacity/load/soak evidence after raising them and reserve portable performance claims for KA-6
|EP-3 and EP-6 (`@implemented-and-validated` for declared single-node scopes)

|4
|**Security must be explicitly enabled** — EP-1 secured mode now proves SigV4, exact payload hashes, deny-by-default, explicit deny, PublicAccessBlock/public ACL denial, expected-owner mismatch, durable redacted tamper-evident audit, durable credentials/policies, and SSE-S3 encrypted-at-rest inspection through Spring Security Reactive plus WebTestClient/AWS CLI/component specs. The remaining risk is operational: the default profile is still anonymous/permit-all for trusted environments.
|Deployments that forget `s3.security.enabled=true` still allow anonymous writes/reads
|Document secured-mode deployment defaults and keep CI gates for EP-1 WebTestClient, AWS CLI/e2e, and backing-service specs
|EP-1 (`@implemented-and-validated` for local built-in scope)

|5
|**Space management remains single-node scoped** — typed GC, dedup reachability, quotas, ENOSPC mapping, and transform-aware scrubbing are implemented, but quota coordination and reclamation are not distributed.
|A future multi-node deployment could race quota or ownership decisions if it reused the single-process coordination model
|Keep the validated single-node boundary explicit and implement distributed coordination only with EP-10 quorum/metadata semantics
|EP-4 (`@implemented-and-validated` single-node scope; distributed extension in EP-10)

|6
|**Operability is bounded to the `0.1.0` single-node JVM preview** — full-gate CI, immutable OCI release rehearsal, persistent-volume replacement, shutdown draining, offline backup/DR, durable schema compatibility, probes, and Prometheus/Alertmanager delivery are validated. Live Loki/external paging and online/multi-node DR are not part of this artifact.
|Operators must not infer HA, rolling upgrade, online backup, external pager delivery, or production-scale performance from the preview release gates
|Keep release scope and OCI identity explicit; assign sustained load to EP-6, deployment-specific paging to deployment integration, and multi-node DR/rolling operation to EP-10
|EP-5 (`@implemented-and-validated` for declared preview scope)

|7
|**Performance evidence is runner-specific, not sizing guidance** — deterministic CI and soak runs prove bounded behavior on their recorded JVM/OS/CPU environment but do not establish portable throughput, hardware sizing, or competitor comparisons.
|Operators could overgeneralize observed p99/operation counts to different disks, CPUs, kernels, or workloads
|Publish machine-readable run manifests and the mandatory non-benchmark disclaimer; use KA-6 for pinned-hardware repeated comparative methodology
|EP-6 (`@implemented-and-validated` safety envelope; KA-6 remains pending)

|8
|**Distributed readiness is modeled, not executed** — placement, quorum, anti-entropy, and rebalance planning are unit-validated in `storage-engine-domain`, but there is no real networked membership, replication execution, or multi-node e2e validation.
|Multi-node/cluster claims would be premature; no HA guarantee exists today
|Do not claim cluster readiness until EP-10 (S3 Cluster) is validated end to end by multi-node scenarios, gated on the EP-8 cluster architecture ADR
|EP-8 / EP-10 (`@absent`, high priority after ADR acceptance)

|9
|**Product-shell boundary regression** — ADR 0026 separation and REQ-ADMIN-032 task-group composition are implemented and validated, but future extensions could reintroduce Object Storage routes, endpoint knowledge, or models into the reusable shell.
|Other Magrathea products would inherit Object Storage coupling or need to fork the shell.
|Keep static boundary, example-product composition, and deterministic extension-removal tests mandatory; Product Shell source must remain product-neutral.
|EP-7 (`REQ-ADMIN-001..037` mitigation validated; ongoing regression risk)

|10
|**Admin Panel becomes an alternate object API** — future product pressure could add bucket/object semantics to the Admin Control Plane or call the storage engine directly.
|S3 compatibility, authorization, auditing, and semantic validation could be bypassed; two conflicting object APIs would emerge.
|Retain the validated Admin route-inventory guard. Keep object/bucket semantics exclusively on the S3 Data Plane; HeadObject diagnostics use the separately configured S3 client.
|EP-7 / INV-3 (boundary implemented and validated; ongoing regression risk)

|11
|**EP-7 unavailable providers may be mistaken for delivered operations** — REQ-ADMIN-037 now validates named unavailable evidence, operational impact, valid next steps, and disabled missing operations, while the Admin routes correctly expose `not-configured`; nevertheless no real recovery, garbage-collection, scrub, audit, metrics, or traces provider is present, and credential/tenant administration is also absent.
|Operators or release notes could still infer operational records or administration capabilities that the system cannot provide, especially by overgeneralizing fixture-backed configured-storage-engine UX or the bounded live bare-single-node browser run.
|Keep HTTP 503 `report-provider-not-configured` explicit, prohibit fabricated records and enabled missing operations, distinguish fixture evidence from live evidence, and retain EP-7 `@partial` until real providers and credential/tenant administration are implemented and semantically validated or formally descoped.
|EP-7 (`@partial`; human scope review required before closure)

|12
|**Dependency vulnerability status is currently unknown** — OWASP Dependency-Check 12.2.2 failed closed during its initial NVD update after 40,000/364,882 records because NVD returned HTTP 429. The retained analysis has `scanStatus=error`, `clean=null`, and 8 scan errors; no JSON/HTML vulnerability report was produced.
|Vulnerable dependencies may remain unidentified. Zero finding counters from the absent report could be misread as a clean scan.
|Do not make vulnerability-finding or clean-scan claims. Inject `NVD_API_KEY` securely through CI/environment and rerun `scripts/run-dependency-check.sh`; retain the CVSS 7.0 gate and require a complete report plus executable semantic validation before closing `REQ-QUAL-003/004`.
|Quality/security evidence (`@partial`, `needs-review`)

|13
|**Embedded Apache Ratis fit is unproven** — ADR 0027 selects Ratis as the planned Raft implementation, but persistence/snapshots, Reactor and mTLS integration, membership change, upgrades, leader transfer, partition behavior, and resource bounds have no repository proof.
|A poor integration could block event loops, lose control state, complicate upgrades, or violate quorum/fencing assumptions.
|Run the time-boxed spike and fault matrix before production implementation acceptance; preserve the consensus boundary so an external service/alternative adapter remains possible.
|EP-8 / EP-10 (`planned`, `needs-review`)

|14
|**Initial voter/data co-location couples failures and resources** — each of three initial storage-node deployments also hosts a metadata voter.
|Host/resource failure can remove both data capacity and a voter; two such losses remove control quorum.
|Place voters across the best available independent topology domains, enforce bounded transport/Ratis resources, validate one-loss/two-loss behavior, and retain a future dedicated-voter deployment option.
|EP-10 (`planned`)

|15
|**Topology/catalog migration is undefined** — current YAML lacks concrete parent-linked `zone/rack/host/disk-set/device` identities and persisted UUID ownership.
|Invalid or stale topology can violate failure-domain placement or fence legitimate writers.
|Define schema migration, duplicate/broken-parent/device-ownership validation, consensus epochs, rollback, and disaster-recovery behavior before implementation.
|EP-10 (`planned`, `needs-review`)

|16
|**Supply-chain artifacts and expanded hardening are absent** — CycloneDX JSON/XML, SPDX-normalized inventory, CI reconciliation, and read-only-root/capability/no-new-privileges runtime proof are decisions only. Current OWASP evidence is fail-closed but incomplete, and remediation is intentionally out of scope for this slice.
|Release consumers lack complete component/license evidence and may overread current non-root checks as full hardening.
|Generate and schema-validate artifacts, report unresolved licenses/findings truthfully, retain OWASP failure semantics, and execute S3/Admin/persistence validation under hardened runtime settings. Revisit remediation separately.
|EP-8 (`planned`, `needs-review`)
|===
