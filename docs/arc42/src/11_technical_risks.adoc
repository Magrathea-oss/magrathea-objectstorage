[[section-technical-risks]]
== Risks and Technical Debts

Risk status is cross-referenced with the Enterprise Production Readiness Plan (`PLAN.md`, EP-0..EP-11) so that mitigation ownership stays traceable. No risk below is closed until the referenced EP phase is `@implemented-and-validated`.

[cols="1,4,3,4,2",options="header"]
|===
|# |Risk |Impact |Mitigation |EP reference

|1
|**Bare local default backend is in-memory** — the unpackaged `single-node` profile's reactive in-memory repositories do not persist to disk. The Storage Engine backend (`storage-engine` profile) now durably persists object bytes/manifests/references AND (as of EP-2) the bucket registry, multipart upload state, and per-object/bucket configuration. Packaged JVM/native single-node containers activate `storage-engine` by default with packaged YAML catalogs.
|Data loss on restart when operators bypass the packaged container defaults and run the bare local `single-node` profile
|Keep the in-memory profile explicitly documented as development/test compatibility; require packaged single-node deployment paths to activate storage-engine and keep CI Docker readiness expecting configured catalogs
|EP-2 (`@implemented-and-validated` for the declared storage-engine metadata durability scope; the in-memory profile remains explicitly exempt for dev/test compatibility)

|2
|**S3 semantic depth** — the in-scope route surface is mapped, but route count is not semantic completion. AWS CLI parity has improved for the canonical object CRUD subset and metadata durability, but full S3 scenario parity is incomplete.
|AWS SDK features may expect deeper semantics than currently implemented
|Track semantic gaps in `PLAN.md`; prioritize AWS CLI/SDK compatibility cases and keep unsupported cases explicitly tagged (`@not-implemented`, `@partial`, `@config-only`)
|Ongoing (S3 API Semantic Completion Plan, `PLAN.md`)

|3
|**Streaming completeness gap** — known full-body/full-part aggregation defects have been removed from PutObject, GetObject, UploadPart, UploadPartCopy, and multipart part-store paths. Pipeline-unit and WebTestClient Cucumber now stream a deterministic 256 MiB REQ-PIPELINE-002 upload through the production staged pipeline and real filesystem, measure live source-payload buffers and bounded request signals, validate the committed manifest and exact streamed readback, and guard against whole-object assembly. REQ-PIPELINE-013 separately observes a shared four-DataBuffer demand ceiling at existing adapter boundaries.
|`OutOfMemoryError` risk is reduced for the validated large write and adapter boundaries; client-driven HTTP cancellation remains unvalidated, and deterministic pre-response integrity checking currently adds a second filesystem read with unvalidated first-byte latency impact
|Keep Cucumber static architecture, staged pipeline-unit/WebTestClient, and finite-demand runtime coverage; add client-driven cancellation validation, and benchmark or redesign the integrity-preflight latency trade-off
|EP-3 (`@partial` — selected staged lifecycle/publication/cleanup behavior, the large pipeline-unit and WebTestClient write contract, and finite adapter demand are validated; client cancellation and read preflight trade-offs remain open)

|4
|**Security must be explicitly enabled** — EP-1 secured mode now proves SigV4, exact payload hashes, deny-by-default, explicit deny, PublicAccessBlock/public ACL denial, expected-owner mismatch, durable redacted tamper-evident audit, durable credentials/policies, and SSE-S3 encrypted-at-rest inspection through Spring Security Reactive plus WebTestClient/AWS CLI/component specs. The remaining risk is operational: the default profile is still anonymous/permit-all for trusted environments.
|Deployments that forget `s3.security.enabled=true` still allow anonymous writes/reads
|Document secured-mode deployment defaults and keep CI gates for EP-1 WebTestClient, AWS CLI/e2e, and backing-service specs
|EP-1 (`@implemented-and-validated` for local built-in scope)

|5
|**Space management and data hygiene absent** — no chunk garbage collection/reference counting beyond on-demand `FileSystemRecoveryScanner`, no quotas, no defined ENOSPC behavior, no background scrubbing.
|Deleted-object chunks are never reclaimed; disk-full behavior is undefined; latent corruption may go undetected
|Implement chunk GC/refcounting, per-bucket/tenant quotas, defined ENOSPC → S3 error mapping, and periodic integrity scrubbing
|EP-4 (`@absent`, high priority)

|6
|**Operability gaps** — the first CI/operability slice now runs appendix/source-hygiene checks, focused Cucumber/security/semantic/EP-5 gates, root JVM Docker build+smoke, manual native Docker build+smoke, Admin API `/admin/live` + `/admin/ready` probes, SIGTERM shutdown/recovery validation for committed objects plus active PutObject, multipart UploadPart, and CompleteMultipartUpload requests, including two concurrent PutObjects, cancelled-and-aborted part cleanup, abort-wins overlapping multipart completion, and a bounded mixed load of three writes and two throttled reads, first offline backup/restore rehearsal, first single-node DR rehearsal with declared RTO/RPO, object-manifest, multipart-session, bucket-registry, object-configuration, object-reference, and object-ACL schema-version compatibility validation, first Prometheus/Loki SLO alert rule bundle with runbook validation, and opt-in live Prometheus-to-Alertmanager webhook delivery. Online/incremental DR, multi-node DR, live Loki/external paging delivery, schema contracts for any future durable metadata families, larger mixed read/write concurrent draining, repeated contention covering completion-wins multipart races and sustained/production-scale shutdown load behavior are not verified.
|Recovery procedures beyond offline single-node backup/restore and single-node DR remain undocumented/unrehearsed; alert rules are shipped but not end-to-end fired through Alertmanager/Loki receivers; schema contracts beyond object manifests remain incomplete; broader operational regressions outside the focused CI slice may still reach `main`
|Expand CI toward the full Maven gate and release hardening; document online/multi-node DR when supported; add broader graceful-shutdown in-flight draining validation
|EP-5 (`@partial`, high priority)

|7
|**Distributed readiness is modeled, not executed** — placement, quorum, anti-entropy, and rebalance planning are unit-validated in `storage-engine-domain`, but there is no real networked membership, replication execution, or multi-node e2e validation.
|Multi-node/cluster claims would be premature; no HA guarantee exists today
|Do not claim cluster readiness until EP-10 (S3 Cluster) is validated end to end by multi-node scenarios, gated on the EP-8 cluster architecture ADR
|EP-8 / EP-10 (`@absent`, high priority after ADR acceptance)
|===
