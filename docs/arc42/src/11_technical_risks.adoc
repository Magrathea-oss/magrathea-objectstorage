[[section-technical-risks]]
== Risks and Technical Debts

Risk status is cross-referenced with the Enterprise Production Readiness Plan (`PLAN.md`, EP-0..EP-11) so that mitigation ownership stays traceable. No risk below is closed until the referenced EP phase is `@implemented-and-validated`.

[cols="1,4,3,4,2",options="header"]
|===
|# |Risk |Impact |Mitigation |EP reference

|1
|**Bare local default backend is in-memory** — the unpackaged `single-node` profile's reactive in-memory repositories do not persist to disk. The Storage Engine backend (`storage-engine` profile) now durably persists object bytes/manifests/references AND (as of EP-2) the bucket registry, multipart upload state, and per-object/bucket configuration. Packaged JVM/native single-node containers activate `storage-engine` by default with packaged YAML catalogs.
|Data loss on restart when operators bypass the packaged container defaults and run the bare local `single-node` profile
|Keep the in-memory profile explicitly documented as development/test compatibility; require packaged single-node deployment paths to activate storage-engine and keep CI Docker readiness expecting configured catalogs
|EP-2 (`@implemented-and-validated` for the declared storage-engine metadata durability scope; the in-memory profile remains explicitly exempt for dev/test compatibility)

|2
|**S3 semantic depth** — the in-scope route surface is mapped, but route count is not semantic completion. AWS CLI parity has improved for the canonical object CRUD subset and metadata durability, but full S3 scenario parity is incomplete.
|AWS SDK features may expect deeper semantics than currently implemented
|Track semantic gaps in `PLAN.md`; prioritize AWS CLI/SDK compatibility cases and keep unsupported cases explicitly tagged (`@not-implemented`, `@partial`, `@config-only`)
|Ongoing (S3 API Semantic Completion Plan, `PLAN.md`)

|3
|**Streaming remains bounded to the validated preview envelope** — EP-3 removed full-object aggregation and introduced typed whole-object/dedup/EC artifacts; EP-6 now enforces and validates the 256 MiB object envelope under `-Xmx256m`.
|Larger objects, higher concurrency, or changed limits may exceed the tested memory/latency envelope even though the streaming architecture remains bounded
|Treat configured EP-6 bounds as preview safety limits; rerun capacity/load/soak evidence after raising them and reserve portable performance claims for KA-6
|EP-3 and EP-6 (`@implemented-and-validated` for declared single-node scopes)

|4
|**Security must be explicitly enabled** — EP-1 secured mode now proves SigV4, exact payload hashes, deny-by-default, explicit deny, PublicAccessBlock/public ACL denial, expected-owner mismatch, durable redacted tamper-evident audit, durable credentials/policies, and SSE-S3 encrypted-at-rest inspection through Spring Security Reactive plus WebTestClient/AWS CLI/component specs. The remaining risk is operational: the default profile is still anonymous/permit-all for trusted environments.
|Deployments that forget `s3.security.enabled=true` still allow anonymous writes/reads
|Document secured-mode deployment defaults and keep CI gates for EP-1 WebTestClient, AWS CLI/e2e, and backing-service specs
|EP-1 (`@implemented-and-validated` for local built-in scope)

|5
|**Space management remains single-node scoped** — typed GC, dedup reachability, quotas, ENOSPC mapping, and transform-aware scrubbing are implemented, but quota coordination and reclamation are not distributed.
|A future multi-node deployment could race quota or ownership decisions if it reused the single-process coordination model
|Keep the validated single-node boundary explicit and implement distributed coordination only with EP-10 quorum/metadata semantics
|EP-4 (`@implemented-and-validated` single-node scope; distributed extension in EP-10)

|6
|**Operability is bounded to the `0.1.0` single-node JVM preview** — full-gate CI, immutable OCI release rehearsal, persistent-volume replacement, shutdown draining, offline backup/DR, durable schema compatibility, probes, and Prometheus/Alertmanager delivery are validated. Live Loki/external paging and online/multi-node DR are not part of this artifact.
|Operators must not infer HA, rolling upgrade, online backup, external pager delivery, or production-scale performance from the preview release gates
|Keep release scope and OCI identity explicit; assign sustained load to EP-6, deployment-specific paging to deployment integration, and multi-node DR/rolling operation to EP-10
|EP-5 (`@implemented-and-validated` for declared preview scope)

|7
|**Performance evidence is runner-specific, not sizing guidance** — deterministic CI and soak runs prove bounded behavior on their recorded JVM/OS/CPU environment but do not establish portable throughput, hardware sizing, or competitor comparisons.
|Operators could overgeneralize observed p99/operation counts to different disks, CPUs, kernels, or workloads
|Publish machine-readable run manifests and the mandatory non-benchmark disclaimer; use KA-6 for pinned-hardware repeated comparative methodology
|EP-6 (`@implemented-and-validated` safety envelope; KA-6 remains pending)

|8
|**Distributed evidence is limited to fixed A/B/C and bounded repair** — real consensus, whole-object `N=3/W=2` replication, failover/restart, and current-generation repair are validated only for `001..005/008..013/019..023/025/026`; `014/017/024` are partial, while multipart/conditional/versioned/EC transfer, dynamic membership, broad anti-entropy/rebalance/cleanup, and broader partitions remain absent or unvalidated.
|Consumers may overgeneralize one-coordinator-stop or bounded repair evidence into HA, broad S3 clustering, broad healing, or distributed production readiness.
|Keep EP-10 `@partial`; name the exact validated and partial requirement boundaries and prohibit broader claims until each later capability has semantic and fault evidence.
|EP-10 (`@partial`, bounded fixed-cluster repair validated; broader distributed readiness not claimed)

|9
|**Product-shell boundary regression** — ADR 0026 separation and REQ-ADMIN-032 task-group composition are implemented and validated, but future extensions could reintroduce Object Storage routes, endpoint knowledge, or models into the reusable shell.
|Other Magrathea products would inherit Object Storage coupling or need to fork the shell.
|Keep static boundary, example-product composition, and deterministic extension-removal tests mandatory; Product Shell source must remain product-neutral.
|EP-7 (`REQ-ADMIN-001..037` mitigation validated; ongoing regression risk)

|10
|**Admin Panel becomes an alternate object API** — future product pressure could add bucket/object semantics to the Admin Control Plane or call the storage engine directly.
|S3 compatibility, authorization, auditing, and semantic validation could be bypassed; two conflicting object APIs would emerge.
|Retain the validated Admin route-inventory guard. Keep object/bucket semantics exclusively on the S3 Data Plane; HeadObject diagnostics use the separately configured S3 client.
|EP-7 / INV-3 (boundary implemented and validated; ongoing regression risk)

|11
|**EP-7 unavailable providers may be mistaken for delivered operations** — REQ-ADMIN-037 now validates named unavailable evidence, operational impact, valid next steps, and disabled missing operations, while the Admin routes correctly expose `not-configured`; nevertheless no real recovery, garbage-collection, scrub, audit, metrics, or traces provider is present, and credential/tenant administration is also absent.
|Operators or release notes could still infer operational records or administration capabilities that the system cannot provide, especially by overgeneralizing fixture-backed configured-storage-engine UX or the bounded live bare-single-node browser run.
|Keep HTTP 503 `report-provider-not-configured` explicit, prohibit fabricated records and enabled missing operations, distinguish fixture evidence from live evidence, and retain EP-7 `@partial` until real providers and credential/tenant administration are implemented and semantically validated or formally descoped.
|EP-7 (`@partial`; human scope review required before closure)

|12
|**Dependency vulnerability status remains unknown/error** — the historical EP-8 clean-revision OWASP Dependency-Check 12.2.2 run failed closed after NVD HTTP 429 at 190,000/364,891 records. Evidence records `scanStatus=error`, `clean=null`, and 8 errors; EP-8 makes no vulnerability-findings, clean-scan, zero-vulnerability, or remediation claim. The later dirty-working-tree root test did not refresh the assessment.
|Vulnerable dependencies may remain unidentified. Monitoring/evidence wiring completion could be misread as dependency safety.
|Keep the CVSS 7.0 policy and historical clean-revision error evidence visible; rerun through scheduled/manual CI with securely injected `NVD_API_KEY`. Require a complete assessment before any vulnerability cleanliness statement, independently of EP-8 architecture/evidence status.
|EP-8 monitoring wiring (`@implemented-and-validated`); vulnerability assessment (`unknown/error`, `needs-review`)

|13
|**Apache Ratis proof is bounded** — the fixed three-voter lifecycle, persisted restart, consensus bucket/reference publication, control mTLS, and loss-of-control-quorum refusal are exercised. Membership change, rolling upgrade/leader-transfer behavior, snapshots under sustained load, asymmetric partitions, and operational resource limits remain unproven.
|The bounded integration can still fail under lifecycle and network conditions outside the first slice.
|Preserve the consensus adapter boundary; add focused membership, upgrade, snapshot/load, and partition fault matrices before extending the support claim.
|EP-10 first slice (`implementation-informed`); wider Ratis lifecycle (`not implemented` / `needs-review`)

|14
|**Implemented voter/data co-location couples failures and resources** — each fixed A/B/C JVM hosts both one metadata voter and one replica server.
|A process/host/resource failure removes both roles; two voter losses block publication, and the current evidence does not establish failure-domain placement or capacity isolation.
|Keep separate ports/executors/limits, deploy across independent domains when evaluated, retain a future dedicated-voter option, and do not infer two-node-loss tolerance from the validated refusal to publish.
|EP-10 first slice (`implementation-informed`, ongoing deployment risk)

|15
|**Topology/catalog migration is undefined** — current YAML lacks concrete parent-linked `zone/rack/host/disk-set/device` identities and persisted UUID ownership.
|Invalid or stale topology can violate failure-domain placement or fence legitimate writers.
|Define schema migration, duplicate/broken-parent/device-ownership validation, consensus epochs, rollback, and disaster-recovery behavior before implementation.
|EP-10 (`planned`, `needs-review`)

|16
|**Supply-chain and focused cluster evidence can be overread** — the clean-revision SBOM/hardening packet predates four newly composed cluster modules, while the fixed A/B/C semantic gates are separate evidence. No artifact was published, OWASP remains unknown/error, and EP-10 capabilities outside the first slice are absent.
|Consumers could mistake historical inventory generation for current complete-reactor coverage or compliance, a local image for a release, a dirty root pass for supply-chain acceptance, single-node hardening for clustered deployment hardening, or bounded failover for production distributed readiness.
|Keep historical hashes explicitly scoped, `REQ-SUPPLY-001` at `@implemented-not-e2e-validated` until a new clean packet covers all production modules, `NOASSERTION`/manual-review entries, `published=false`, unavailable userns status, and OWASP error visible. Report focused EP-10 gates separately from ordinary root Maven and prohibit broader membership, broad healing/cleanup, partition, or readiness claims.
|EP-8 architecture/other evidence guards (declared statuses retained); `REQ-SUPPLY-001` (`@implemented-not-e2e-validated`); EP-10 (`@partial`); compliance/publication/production readiness (`not claimed`)

|17
|**The implemented fixed A/B/C bootstrap can be mistaken for dynamic membership** — one identical static manifest and stable UUID roots recover successfully, but there is no admission, promotion, demotion, replacement, or removal flow.
|Operators could treat configuration edits, address changes, timeout, or seed order as membership transitions, risking split-brain, duplicate identity, or unsafe replacement.
|Keep the mode named fixed bootstrap, reject mismatch against non-empty persisted state, and prohibit membership claims. Implement ADR 0027's consensus-controlled lifecycle and fencing in a separate validated slice.
|EP-10 first slice (`implementation-informed`); dynamic membership (`not implemented`, `needs-review`)

|18
|**The manual Reactor/grpc-java bridge remains correctness-sensitive** — focused `REQ-CLUSTER-011/012` scenarios exercise 64 KiB frames, slow demand, finite bounds, cancellation, deadline, and cleanup, but wider workloads and repeated network faults are not covered.
|Undiscovered demand bugs could still deadlock, buffer excessively, block event loops, leak resources, or mishandle terminal races outside the tested slice.
|Keep the bridge infrastructure-only and every bound explicit; extend stress, repeated-fault, and long-duration evidence before raising payload/concurrency limits or broadening transfer semantics.
|EP-10 first slice (`implementation-informed`; residual high risk)

|19
|**Pinned versions are bounded build evidence, not general compatibility or security evidence** — Ratis 3.2.2, grpc-java 1.82.2, and protobuf/protoc 3.25.8 resolve and pass focused lifecycle/TLS/persistence gates; Testcontainers 2.0.2 is pinned but not used by the real-process S3 acceptance gate.
|Rolling/mixed-version operation, future dependency convergence, vulnerability status, shaded transport changes, and upgrade compatibility remain unknown.
|Retain separate modules/ports and focused convergence gates; require upgrade, mixed-version, and complete vulnerability evidence before changing the baseline or claiming compatibility/security.
|EP-10 / ADR 0028 (`accepted` bounded baseline; wider compatibility `needs-review`)

|20
|**Test-local mTLS can be overread as production PKI readiness** — focused control/data scenarios reject wrong or mismatched peers, but ephemeral local CA, node certificates, and trust stores prove only the exercised test environment.
|Passing TLS and failover gates can conceal absent operational enrollment, key custody, trust distribution, rotation, revocation, expiry monitoring, and disaster recovery.
|Keep production mTLS mandatory and test-only certificate infrastructure visibly separate. Require an operational certificate lifecycle and stable-node-identity evidence before any production security claim.
|EP-10 first slice (`implementation-informed`); production PKI (`blocked`, owner/review point: security and operations)

|21
|**Bounded repair can be overread as broad healing** — accepted ADR 0029 and the implementation restore one missing/corrupt target named by the current reference, but broad `REQ-CLUSTER-017` remains partial and does not include periodic anti-entropy, rebalance, or orphan cleanup.
|Operators could treat a planner result or process-local scheduler as durable ownership, infer fleet-wide discovery/cleanup behavior, or mistake bounded repair for production readiness.
|Keep consensus as job/lifecycle authority and scheduler signals message-free; report `019/020/021/022/023/025/026` separately from partial `017/024`; retain prepared-artifact intents, orphan cleanup, rebalance, broad anti-entropy, dynamic membership, EC, multipart, and partitions as explicit exclusions.
|EP-10 repair (`implementation-informed`, bounded); broad healing/cleanup (`partial/absent`; owners: cluster implementation and operations)

|22
|**Single-pass corrupt GET has a late-failure client contract** — corruption may be detected after response headers or bytes are emitted, so the server cannot safely replace the stream with a remote replica or necessarily send a fresh S3 XML error body.
|Clients can receive a transport/body-integrity failure rather than a conventional pre-body S3 error; a transparent retry could concatenate or disguise corrupt bytes.
|Preserve no-preflight/no-second-read and prohibit transparent corrupt-read retry. Keep checksum/body assertions in both real-process modes, document that scheduling does not make the current request successful, and require any future client-contract change to preserve emitted-byte safety.
|EP-10 `REQ-CLUSTER-020` (`implementation-informed`, high-impact ongoing contract risk; owners: S3/API and requirements)

|23
|**Repair lifecycle growth is not production-bounded and crash-side-effect evidence is incomplete** — version-2 migration and lifecycle restoration are validated, but jobs, attempts, dedup history, retry metadata, and terminal states enlarge snapshots, and the full seven-point real-filesystem/gRPC crash matrix was not executed.
|Unbounded history could increase snapshot/recovery time; a side-effect crash outside covered combinations could delay or duplicate exact publication even though fencing protects lifecycle state.
|Keep unknown-version fail-closed behavior and monotonic claim generation; complete the seven-point side-effect matrix before upgrading `REQ-CLUSTER-024`; measure snapshot growth before defining terminal-history retention/compaction and never silently discard dedup/fencing evidence.
|EP-10 `REQ-CLUSTER-024` (`partial`, `needs-review`; owners: Ratis adapter, data plane, and operations)

|24
|**Current-generation races and unavailable sources can leave repair blocked or unreachable exact bytes** — the reference can change during transfer, or every named source can be missing/corrupt/unavailable.
|An old artifact might be durably published but non-authoritative, while endless retry or copying an unverified source would threaten integrity and capacity.
|Retain implemented currentness checks before source selection, publication, and success; keep changed work terminal `OBSOLETE`, finite retries, visible `BLOCKED`, and integrity alerts. Never return/copy unverified bytes. Leave resulting unreachable artifacts in place until separately fenced orphan cleanup is decided.
|EP-10 bounded repair (`implementation-informed`; owner: cluster application/operations); orphan cleanup (`absent/deferred`, separate ADR required)
|===
