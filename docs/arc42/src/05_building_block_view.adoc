== Building Block View

=== Level 1 — Container Diagram

The C2 model distinguishes the public S3 adapter, the internal/admin adapter, the bootstrap runtime assembly, object-store services and repository ports, the in-memory backend, and the Storage Engine backend. Current Storage Engine module names are `storage-engine-domain`, `storage-engine-reactive-application`, and `storage-engine-reactive-infrastructure`.

.C2 Container Diagram — Magrathea ObjectStore
image::../../c4/images/Container.png[Container Diagram, width=600]

==== bootstrap-application (runtime assembly)

**Blackbox Description:**
Spring Boot executable assembly. It wires the selected backend, includes the S3/admin adapters on the classpath, and serves the generated Vue/admin documentation assets from classpath `static/**` resources, including copied C4 images and generated ARC42 JSON.

- **Provided Interface:** Executable application on the configured HTTP port
- **Required Interface:** S3/admin adapters, object-store services, selected backend beans
- **Technology:** Spring Boot 4, Java 21, classpath static resources

==== s3-reactive-api-adapter (HTTP adapter)

**Blackbox Description:**
Exposes the S3-compatible REST API via RouterFunction WebFlux. Translates HTTP/XML/binary requests into calls to reactive application services. Serializes XML responses with Jackson 3.

Handlers follow a strict **minimal pattern**:
- **Extract** HTTP headers, path variables, and query parameters into primitives/DTOs via `S3RequestExtractor`
- **Delegate** to reactive application services (no validation, no bucket checks, no ETag computation)
- **Response** via `S3ResponseBuilder` or direct XML serialization

All validation, bucket existence checks, ETag computation, CORS validation, and website routing are **postponed to the repository layer**.

- **Provided Interface:** S3 REST API route surface (111 in-scope routes; route inventory only, not semantic completion)
- **Required Interface:** ReactiveBucketService, ReactiveObjectService, ReactiveMultipartUploadService
- **Technology:** Spring Boot 4 WebFlux, Java 21, RouterFunction, Jackson 3 XML

==== reactive-object-store (Reactive object storage capability)

**Blackbox Description:**
Implements reactive object CRUD, multipart upload, content retrieval, and Phase F advanced object use cases using Mono/Flux. Contains reactive application services (ReactiveObjectService, ReactiveMultipartUploadService), domain aggregates (S3Object, MultipartUpload), and repository ports.

- **Provided Interface:** ReactiveObjectService, ReactiveMultipartUploadService
- **Required Interface:** BucketRepository (read), S3ObjectCommandRepository, S3ObjectQueryRepository, MultipartUploadCommandRepository, MultipartUploadQueryRepository
- **Technology:** Java 21, Spring @Service, Reactor Mono/Flux, domain aggregates

==== reactive-bucket-management (Reactive bucket management capability)

**Blackbox Description:**
Implements reactive bucket lifecycle, metadata (ACL, tagging), configuration (CORS, lifecycle, policy, encryption, logging, website, notification, replication, request payment, ownership controls, public access block, accelerate), advanced configuration (analytics, inventory, metrics, intelligent-tiering), and Phase F bucket metadata/ABAC/object-lock configuration use cases using Mono/Flux. Contains ReactiveBucketService, the Bucket aggregate root (with dedicated configuration value objects and domain events), and repository ports. Configuration lives inside the Bucket aggregate — there is no separate configuration service.

- **Provided Interface:** ReactiveBucketService
- **Required Interface:** BucketCommandRepository, BucketQueryRepository
- **Technology:** Java 21, Spring @Service, Reactor Mono/Flux, domain aggregates

==== reactive-repository-application (CQS repository interfaces)

**Blackbox Description:**
Implements the Command-Query Separation (CQS) pattern at the application layer. Provides separate command and query repository interfaces for each aggregate (Bucket, S3Object, MultipartUpload). Bridges domain repository ports to reactive persistence adapters.

- **Provided Interface:** BucketCommandRepository, BucketQueryRepository, S3ObjectCommandRepository, S3ObjectQueryRepository, MultipartUploadCommandRepository, MultipartUploadQueryRepository
- **Required Interface:** Reactive persistence implementations
- **Technology:** Java 21, Mono/Flux, CQS pattern

==== reactive-infrastructure (Legacy in-memory test/retirement adapters)

**Blackbox Description:**
Contains reactive `ConcurrentHashMap` repositories using Mono/Flux. They reset when the process stops and are not a supported single-node product backend. They are isolated behind the explicit `legacy-in-memory-test` profile and may be retired separately.

- **Provided Interface:** InMemoryReactiveBucketRepository, InMemoryReactiveS3ObjectRepository, InMemoryReactiveMultipartUploadRepository
- **Required Interface:** CQS repository interfaces
- **Technology:** Java 21, ConcurrentHashMap, Reactor
- **Tag:** Legacy test adapter

==== object-store-reactive-repository-storage-engine-infrastructure (Durable storage-engine backend)

**Blackbox Description:**
Implements the CQS repository interfaces backed by the Storage Engine, selected explicitly via the `storage-engine` Spring profile. Bridges the Object Store bounded context to the Storage Engine through the object-store-to-storage-engine ACL adapter (`ObjectStoreToStorageEngineTranslator`). Every durable document (bucket registry, multipart upload state, per-object configuration, S3 object manifest references) is committed with a crash-safe temp-file + atomic-rename write and reloaded at process startup, so this backend survives a full process restart — unlike the in-memory default.

- **Provided Interface:** `StorageEngineReactiveBucketRepository`, `StorageEngineReactiveS3ObjectRepository`, `StorageEngineReactiveMultipartUploadRepository`
- **Required Interface:** `ReactiveStorageOrchestrator` (chunk/manifest read-write), CQS repository interfaces
- **Technology:** Java 21, Reactor, `tools.jackson.core:jackson-databind` (field-visibility JSON mapper), filesystem I/O on `Schedulers.boundedElastic()`
- **Durable stores (EP-2 metadata durability):**
 * `BucketStore` — bucket aggregates including the full `BucketConfig` document (all configuration families) under `metadata/buckets/`
 * `MultipartUploadStateStore` — multipart upload sessions (upload id, key, initiated timestamp, recorded parts) under `metadata/multipart-uploads/`
 * `ObjectConfigMetadataStore` — per-object legal hold, object lock, retention, encryption, and restore state as one self-consistent document per bucket/key under `metadata/object-config/`
 * `S3ObjectManifestReferenceStore` — latest bucket/key → storage-engine manifest reference, striped-lock read-compose-write under `metadata/s3-object-references/`
 * `DurableJson` — shared Jackson mapper and atomic write/read/wipe helpers used by the stores above

==== admin-api-adapter (internal/admin HTTP adapter)

**Blackbox Description:**
Exposes `/admin/**` JSON endpoints for health/readiness, backend and catalog status, read-only configuration catalogs, non-persistent validation, bucket capacity/quota, route inventory, and operational-report availability. These endpoints are internal/administrative and are not part of S3 object API coverage.

The catalog mutability contract is read-only at runtime: YAML-backed storage policies, devices, and disk sets are exposed through list/detail endpoints; policy validation returns structured results without persisting; policy mutation returns `405 Method Not Allowed`. Recovery, garbage-collection, scrub, audit, metrics, and traces report routes return HTTP 503 `report-provider-not-configured` when no real provider is configured.

- **Provided Interface:** Admin JSON API under `/admin/**` for the validated `REQ-ADMIN-023..031` contract set
- **Required Interface:** Catalog, selected-backend, capacity/quota, route-inventory, and optional operational-report provider ports
- **Technology:** Spring Boot 4 WebFlux, Java 21, RouterFunction

==== storage-engine-reactive-application and storage-engine-reactive-infrastructure

**Blackbox Description:**
`storage-engine-reactive-application` owns reactive orchestration ports such as the chunker, storage policy catalog, `ReactiveStorageOrchestrator`, and the transport-neutral `BoundedEcReconstructionPort`. `storage-engine-reactive-infrastructure` provides YAML-backed catalogs, filesystem persistence, physical EC encoding, and the stateless local reconstruction adapter. Local runtime read/write, physical EC shards, schema-3 reconstruction facts, and output-only decoding are validated. Fixed A/B/C EC 4+2 placement/transfer is separately implemented in the cluster application/data/control modules; clustered S3 EC reads remain pending.

`MINIO_STANDARD` maps to S3 storage class `STANDARD`, disables deduplication, enables erasure coding as `4 data / 2 parity`, uses replication factor `1`, disables compression, and leaves encryption disabled by default.

- **Provided Interface:** Storage Engine orchestration, bounded reconstruction, and catalog ports/adapters
- **Required Interface:** Storage Engine domain model and filesystem/configuration resources
- **Technology:** Java 21, Reactor, YAML configuration, filesystem adapters

==== Bounded local EC reconstruction components (accepted ADR 0032)

[NOTE]
====
Status: *implementation-informed / bounded*. `REQ-PIPELINE-017` passes 5 scenarios / 46 steps, including all 15 four-of-six EC 4+2 survivor combinations.
====

[cols="3,5",options="header"]
|===
|Component |Implemented responsibility

|`EcShardLayout` and schema-3 manifest mapping
|Bind every committed EC artifact to explicit stripe/shard/k/m/parity/logical-length/stored-length/SHA-256/transport-neutral-location facts. Schema-2 EC without explicit layout is not reconstructed by inference.

|`BoundedEcReconstructionPort`
|Application-owned request/result boundary for one committed stripe. It accepts caller-resolved survivor snapshots and returns only verified logical-stripe and regenerated-shard output plus separated workspace measurements. It exposes no repository or publication operation.

|`BoundedEcReconstructionAdapter`
|Stateless infrastructure implementation using the shared `GaloisField256Codec`; validates committed metadata/bytes, reconstructs from exactly `k` survivors, verifies regenerated output, preserves short-stripe logical length, and fails closed.

|`boundedEcReconstructionScheduler`
|Dedicated Spring-owned Reactor scheduler with one worker and queue capacity 16; keeps GF(256) work off the caller thread and is disposed with the context.
|===

The decoder owns at most one 4 MiB logical stripe, six 1 MiB shard buffers, and fixed matrix/index workspace, with zero earlier-stripe or whole-object retention. Immutable request/result snapshots are reported separately. No shard replacement, scanner/daemon, Ratis repair job, rebalance, cleanup, or fault-injection component is introduced by this local boundary.

==== Fixed distributed EC 4+2 components (accepted ADR 0033)

[NOTE]
====
Status: *implementation-informed / bounded*. `REQ-CLUSTER-015` passes 5 scenarios / 40 steps in real Ratis, grpc-java/mTLS, restart, and filesystem modes.
====

[cols="3,5",options="header"]
|===
|Component |Implemented responsibility

|`PreparedEcObject`, `PreparedEcShard`, `EcShardReference`
|Transport-neutral one-stripe fixed EC 4+2 schema-3 input and authoritative shard/location facts. Other geometries are rejected by configuration and reconstruction boundaries.

|`FixedEc42Placement`
|Assign two shards per arranged failure domain: A=`0,3`, B=`1,4`, C=`2,5`.

|`ClusterEcWriteCoordinator`
|Incrementally verifies prepared shard/object integrity, transfers four remote shards with bounded concurrency, collects six exact acknowledgements, revalidates membership, and delegates publication.

|`EcReferencePublicationService` and `compareAndPublishEc`
|Reject incomplete/conflicting evidence before command creation and commit one fenced `EC_4_2` reference through Ratis.

|Ratis `PublishEc` command/reference codec
|Persist object facts and six schema-3 shard locations without payload bytes; retain snapshot-v2 backward compatibility and restart recovery.
|===

These components do not implement clustered S3 EC reads, repair/self-healing, rebalance, cleanup, dynamic membership, or parameterized EC.

=== Implemented Fixed Three-Node First Slice (ADRs 0027 and 0028)

[NOTE]
====
Status: *implementation-informed / partial*. The fixed A/B/C decomposition and its bounded behavior are implemented and validated. The diagram does not represent dynamic membership, broader S3 clustering, healing beyond bounded current-reference obligations, rebalance, general partition tolerance, or production readiness.
====

.Cluster Node Runtime components
image::../../c4/images/ClusterNodeComponents.png[Cluster Node Runtime Components, width=600]

The **Cluster Node Runtime** is one logical C2 container instantiated as fixed nodes A, B, and C. Each JVM co-locates one Ratis voter and one replica data server while keeping control and data transports, ports, executors, limits, metrics, and lifecycle ownership separate. Its implemented first-slice C3 decomposition is:

[cols="3,5",options="header"]
|===
|Module |Responsibility

|`storage-engine-cluster-application`
|Transport-neutral cluster use cases and ports for consensus-authoritative bucket/object-reference generations, bounded current-reference pages, local artifacts, fixed `N=3/W=2` whole-object write coordination, committed-reference reads, bounded repair, and process-local periodic discovery. It has no Ratis, protobuf, generated-stub, or gRPC dependencies.

|`cluster-protocol`
|Versioned application-owned protobuf contracts and generated messages/stubs for internal cluster communication. It contains no S3 API or policy decisions.

|`cluster-control-ratis-infrastructure`
|Apache Ratis 3.2.2 with `ratis-grpc`, the deterministic control state machine, canonical bounded read-only current-reference paging, fixed A/B/C bootstrap, persisted version-2 log/snapshot repair state and version-1 migration, control-plane mTLS, and infrastructure mapping. The discovery cursor is never persisted; dynamic membership remains future scope.

|`cluster-data-grpc-infrastructure`
|Independent grpc-java 1.82.2 `grpc-netty-shaded` replica servers/clients using protobuf 3.25.8, checksum/length validation, durable acknowledgements, deadlines/cancellation, replica mTLS, and a bounded manual 64 KiB flow-control bridge.
|===

`bootstrap-application` composes one runtime per JVM, and `object-store-reactive-repository-storage-engine-infrastructure` selects cluster bucket/object repositories for the cluster profile. That adapter remains the only S3-to-Storage Engine integration boundary; no Ratis or replica listener is a public object API. Each node owns separate stable-UUID identity, Ratis, object, temporary, and runtime roots.

The implemented behavior is restricted to unconditional `CreateBucket`, single-part whole-object `PUT`, and whole-object `GET`. Consensus orders bucket and object-reference generations; payload bytes do not enter the Ratis log. Direct replication uses fixed `N=3, W=2`, and no object-reference generation is published below `W=2` durable checksum-valid acknowledgements or without control quorum.

The PA-6 `DistributedPlacementPlanner`, `QuorumPolicy`, `AntiEntropyPlanner`, `RebalancePlanner`, and `DistributedReadinessReporter` remain pure components inside `storage-engine-domain`; they do not acquire Ratis, gRPC, protobuf, Spring, filesystem, certificate, retry, or lifecycle dependencies. `REQ-CLUSTER-014` is now implemented and validated for its sole internal repository-rooted architecture mode: the gate parses every reactor module and direct non-test/profile dependency, tracked production Java boundary through JDK ASTs, and versioned cluster proto3 contract; executes the layering guard; and fails closed on missing, unreadable, empty, or malformed inputs.

On 2026-07-14, `ReqCluster014ArchitectureContractCucumberTest` passed 1 scenario / 17 steps over 17 modules, 87 direct non-test dependencies plus 2 profile declarations, 473 tracked production Java sources, one proto3 contract, 12 missing/empty/malformed probes, and 4 supported POSIX unreadable probes with exact-path and input-hash preservation. Coordination 7/54, control 2/19, data 4/40, repair 22/294, EP-8 architecture 13/103, source hygiene, and appendix freshness regressions passed. The CI job is wired but has not run. This evidence adds no building block, public API, deployment unit, or runtime side effect; PA-6 plans remain side-effect-free planning output rather than proof of copied, repaired, rebalanced, or reclaimed data.

==== Implemented Bounded Repair Extension (accepted ADR 0029)

[NOTE]
====
Status: *implementation-informed / bounded*. `REQ-CLUSTER-019..026`, including `024`, are implemented and validated. Broad `REQ-CLUSTER-017` remains partial; no production-readiness claim follows.
====

.Consensus-owned bounded repair components
image::../../c4/images/RepairComponents.png[Repair Components, width=600]

The implementation adds no C2 container. It composes these responsibilities inside each fixed **Cluster Node Runtime**:

[cols="3,5",options="header"]
|===
|Component |Implemented responsibility

|**Durable Repair Coordinator and Scheduler**
|The coordinator ensures bounded missing/corrupt current-generation work. The scheduler stores no authoritative job data: startup and committed-work signals only trigger bounded consensus queries, and serial dispatch is limited to committed work for the local target. `ClusterNodeRuntime` owns scheduler start/stop with process lifecycle.

|**Fenced Repair Worker**
|Executes only an exact committed `(jobId, claimGeneration)` assignment owned by its stable node and process session. It rechecks currentness, transfers from another named replica, verifies complete length/SHA-256, durably publishes the target, and proposes an idempotent result.

|**Ratis Control Infrastructure**
|Persists repair identity, immutable specification, lifecycle, attempt/deduplication history, claim generations, owner sessions, retry eligibility, and terminal reasons in version-2 snapshots/log state. Its deterministic state machine owns lifecycle but performs no RPC or filesystem side effect.

|**Replica Data Infrastructure**
|Keeps verified whole-object bytes on the existing direct mTLS path. Token-specific staging, fsync, atomic publication/replacement, and directory fsync make repeated exact-content effects safe; bytes never enter the Ratis log or snapshot.
|===

PA-6 `AntiEntropyPlanner` remains pure planning policy; its `HealingTask` is neither durable ownership nor proof of copied data. `RebalancePlanner` is not used. The repair target must already be named by the current committed reference, so this building-block change does not add placement or alter C2 boundaries.

==== Implemented bounded periodic current-reference discovery (accepted ADR 0031)

[NOTE]
====
Status: *implementation-informed / bounded*. `REQ-CLUSTER-027` is implemented and validated by 2 scenarios / 36 steps. Broad `REQ-CLUSTER-017` remains partial; no production-readiness claim follows.
====

This refinement also adds no C2 container or external API. `ClusterAntiEntropyScheduler` is a transport-neutral application component owned by each `ClusterNodeRuntime`; `ReferencePageQuery` and `ReferencePage` bound its read port. `RatisControlPlaneAdapter` maps that port to fail-closed read-only state-machine queries over canonical current references. The scheduler keeps one process-local exclusive cursor, serially probes only references naming the local UUID, and delegates missing/corrupt obligations to the existing durable repair coordinator and worker. `ClusterNodeRuntime` starts/stops both schedulers; cursor reconstruction begins at the first page.

The cursor is absent from consensus/log/snapshot state, page sequences are not claimed as one frozen snapshot, local probes run on `boundedElastic`, and whole-object repair bytes remain on the existing UUID-bound mTLS replica path. Counters expose bounded activity/failure without payloads or credentials. No PA-6 planner, rebalance, cleanup, new repair lifecycle, or ADR 0030 chaos component is introduced.

.Deferred cluster capabilities — deliberately outside bounded repair and discovery
image::../../c4/images/FutureClusterCapabilities.png[Deferred Cluster Capabilities, width=600]

`REQ-CLUSTER-024` is implemented and validated for its bounded seven-point real-filesystem/gRPC interruption catalogue, including independent target-process restart, exact filesystem effects, snapshot/log recovery, fencing, and live leadership transfer. `REQ-CLUSTER-027` separately validates bounded periodic current-reference discovery and existing repair execution on fixed A/B/C. Neither broadens the repair target beyond the current committed generation. `REQ-CLUSTER-017` remains partial because prepared-artifact intents, automated orphan cleanup, rebalance, and wider healing/topology coverage are absent. Local output-only EC decoding exists under ADR 0032, but clustered multipart, conditional/versioned and chunked writes, clustered S3 EC reads, EC self-healing, parameterized EC, consensus-controlled dynamic membership/certificate lifecycle, rolling upgrades, ADR 0030 faults, and broader partitions remain unimplemented or unvalidated.

The ordered next building blocks are distributed shard placement/transfer, the planned bounded ADR 0030 committed-shard test kernel/actions, consensus-owned EC self-healing, shard rebalance, and fenced cleanup; generalized chaos is last. Whole-object replication remains available below the arranged EC minimum, while inter-cluster/NAS-backed replication remains later and NAS-dependent.

=== Level 3 — Code View (Aggregate Root Patterns)

The domain aggregates follow a strict state transition pattern:

==== Bucket Aggregate

`Bucket` is the root aggregate for bucket management. It enforces all state transitions through dedicated `with*()` methods:

```java
public interface Bucket {
    // Lifecycle
    Bucket withVersioningEnabled();
    Bucket withVersioningSuspended();
    Bucket withDeleted();

    // Configuration (one per config type)
    Bucket withCorsConfiguration(CorsConfiguration cors);
    Bucket withoutCorsConfiguration();
    Bucket withLifecycleConfiguration(BucketLifecycleConfiguration lifecycle);
    Bucket withoutLifecycleConfiguration();
    Bucket withEncryptionConfiguration(BucketEncryptionConfiguration encryption);
    Bucket withoutEncryptionConfiguration();
    Bucket withLoggingConfiguration(BucketLoggingConfiguration logging);
    Bucket withoutLoggingConfiguration();
    Bucket withWebsiteConfiguration(BucketWebsiteConfiguration website);
    Bucket withoutWebsiteConfiguration();
    Bucket withNotificationConfiguration(BucketNotificationConfiguration notification);
    Bucket withoutNotificationConfiguration();
    Bucket withReplicationConfiguration(BucketReplicationConfiguration replication);
    Bucket withoutReplicationConfiguration();
    Bucket withPolicy(BucketPolicy policy);
    Bucket withoutPolicy();
    Bucket withPublicAccessBlockConfiguration(PublicAccessBlockConfiguration publicAccessBlock);
    Bucket withoutPublicAccessBlockConfiguration();
    Bucket withOwnershipControls(BucketOwnershipControls ownershipControls);
    Bucket withoutOwnershipControls();
    Bucket withRequestPaymentConfiguration(BucketRequestPaymentConfiguration requestPayment);
    Bucket withoutRequestPaymentConfiguration();
    Bucket withAccelerateConfiguration(BucketAccelerateConfiguration accelerate);
    Bucket withoutAccelerateConfiguration();

    // Multi-instance config (analytics, inventory, metrics, intelligent-tiering)
    Bucket withAnalyticsConfiguration(String id, BucketAnalyticsConfiguration analytics);
    Bucket withoutAnalyticsConfiguration(String id);
    Bucket withInventoryConfiguration(String id, BucketInventoryConfiguration inventory);
    Bucket withoutInventoryConfiguration(String id);
    Bucket withMetricsConfiguration(String id, BucketMetricsConfiguration metrics);
    Bucket withoutMetricsConfiguration(String id);
    Bucket withIntelligentTieringConfiguration(String id, BucketIntelligentTieringConfiguration tiering);
    Bucket withoutIntelligentTieringConfiguration(String id);

    // EP-2: ABAC, bucket metadata, and metadata-table configuration (rules-based,
    // matching the S3 API shape), plus the newer object-lock/inventory-table/
    // journal-table families — all durable in storage-engine mode via BucketStore
    Bucket withAbacConfiguration(AbacConfiguration abac);
    Bucket withMetadataConfiguration(BucketMetadataConfiguration metadata);
    Bucket withMetadataTableConfiguration(BucketMetadataTableConfiguration metadataTable);
    Bucket withBucketConfig(BucketConfig config); // generic aggregate-level config transition

    // Domain event access
    List<ObjectStoreEvent> domainEvents();
    Bucket clearEvents();
```

==== S3Object Aggregate

`S3Object` enforces state transitions for object lifecycle, metadata, ACL, tagging, legal hold, retention, and encryption:

```java
public interface S3Object {
    S3Object withDeleted();
    S3Object withEtag(String etag);
    S3Object withStorageClass(StorageClass storageClass);
    S3Object withAcl(AccessControlPolicy acl);
    S3Object withTagging(Map<String, String> tagging);
    S3Object withLegalHold(LegalHold legalHold);
    S3Object withoutLegalHold();
    S3Object withRetention(RetentionPeriod retention);
    S3Object withoutRetention();
    S3Object withEncryption(EncryptionConfiguration encryption);
    S3Object withoutEncryption();
    List<ObjectStoreEvent> domainEvents();
    S3Object clearEvents();
}
```

**Durability note:** in storage-engine mode, legal hold, object lock configuration, retention, encryption, and restore state are durable via `ObjectConfigMetadataStore` (EP-2). Object tags and ACL currently live on the `S3Object` aggregate but are not yet part of a durable reference document, so they remain `@partial` for restart durability — tracked as an open EP-2 item.

=== Registry-Based Handler Dispatch

The `S3BucketConfigHandler` uses a registry/strategy pattern to dispatch configuration operations:

```java
public class S3BucketConfigHandler {
    private final ConfigHandlerRegistry registry;

    Mono<ServerResponse> handleGet(ServerRequest request, ConfigType type) {
        ConfigHandlerStrategy strategy = registry.lookup(type);
        // strategy performs GET using typed methods
        return strategy.handleGet(request);
    }

    Mono<ServerResponse> handlePut(ServerRequest request, ConfigType type) {
        ConfigHandlerStrategy strategy = registry.lookup(type);
        // strategy performs PUT using typed methods
        return strategy.handlePut(request);
    }

    Mono<ServerResponse> handleDelete(ServerRequest request, ConfigType type) {
        ConfigHandlerStrategy strategy = registry.lookup(type);
        // strategy performs DELETE using typed methods
        return strategy.handleDelete(request);
    }
}
```
