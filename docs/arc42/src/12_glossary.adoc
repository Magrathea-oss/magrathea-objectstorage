[[section-glossary]]
== Glossary

[cols="2,5",options="header"]
|===
|Term |Definition

|Bucket
|S3 object container. Identified by a unique DNS-compliant name (3-63 characters, lowercase)

|S3Object
|S3 object: key, metadata, version, ACL, tagging, attributes, binary content

|ObjectKey
|Unique key of an object within a bucket

|Region
|AWS S3 region (e.g., `us-east-1`, `eu-west-1`)

|StorageClass
|S3 storage class: `STANDARD`, `INTELLIGENT_TIERING`, `PARANOIC_MODE`

|MINIO_STANDARD
|Storage Engine policy use case whose S3-facing storage class is `STANDARD`: dedup disabled, EC planning `4 data / 2 parity`, replication factor `1`, compression disabled, encryption disabled by default

|Admin API
|Internal `/admin/**` JSON API for storage policy, device, and configuration management; not part of S3 API coverage

|Magrathea Product Shell
|Implemented reusable, product-neutral frontend foundation for Magrathea products. It owns frame/navigation/layout, Design Tokens, Shell Primitives, localization infrastructure, extension/route composition, documentation integration, and common UI states, but no product domain/API logic (ADR 0026)

|Design Token
|Product-neutral named visual value, such as color, typography, spacing, elevation, motion, or breakpoint, consumed by the Magrathea Product Shell and Product Extensions

|Shell Primitive
|Reusable product-neutral UI building block supplied by the Magrathea Product Shell, such as a page frame, navigation container, status banner, table, form, dialog, or common UI state

|Product Extension
|Explicit contribution of product-owned navigation, routes, screens, localization, permissions, and API integrations to the Magrathea Product Shell without modifying shell internals

|Object Storage Admin Application
|The implemented Object Storage Product Extension. It owns all Object Storage screens, models, routes, permissions, API clients, and mapping; it is not the reusable shell

|Admin Control Plane
|Operator-facing `/admin/**` management, configuration, status, security, capacity, and reporting surface for capabilities without S3 equivalents. It must not implement object/bucket semantics

|S3 Data Plane
|The exclusive S3-compatible path for object and bucket semantics. Admin-side S3 diagnostics use this plane as an S3 client and receive no storage-engine bypass

|Report provider
|Optional Admin Control Plane adapter that supplies real operational recovery, garbage-collection, scrub, audit, metrics, or traces records. When absent, the validated contract returns HTTP 503 `report-provider-not-configured`; an unavailable contract is not a provider implementation

|ETag
|Entity tag for optimistic concurrency and content verification; computed via a single-pass streaming tee (`UploadDigest`) during PutObject, not a separate materialization pass

|VersionId
|Version identifier for versioned objects

|ACL
|Access Control List: `private`, `public-read`, `public-read-write`, `authenticated-read`

|Tagging
|Key-value pairs for object/bucket categorization

|CORS
|Cross-Origin Resource Sharing: allowed origins, methods, headers

|Multipart Upload
|Upload of large objects in parts. Single-node multipart behavior does not imply clustered multipart support; clustered multipart is not implemented in the fixed EP-10 first slice (`REQ-CLUSTER-006`)

|Storage Engine backend
|Durable persistence backend selected via the `storage-engine` Spring profile; commits bucket registry, multipart upload state, per-object configuration, and object manifest references as crash-safe filesystem documents under a configurable root

|EP-2 / Metadata Durability
|Enterprise Production Readiness phase closing the in-memory-metadata gap for the storage-engine backend: bucket registry, multipart upload state, per-object configuration, and all bucket configuration families are durable and restart-tested

|BucketStore
|Durable JSON store for bucket aggregates (including the full `BucketConfig` document) under `metadata/buckets/`

|MultipartUploadStateStore
|Durable JSON store for multipart upload sessions (upload id, key, initiated timestamp, recorded parts) under `metadata/multipart-uploads/`

|ObjectConfigMetadataStore
|Durable JSON store for per-object legal hold, object lock, retention, encryption, and restore state under `metadata/object-config/`

|S3ObjectManifestReferenceStore
|Durable per-bucket/key index from S3 object to the latest storage-engine manifest, under `metadata/s3-object-references/`

|DurableJson
|Shared Jackson 3 field-visibility mapper and atomic write/read/wipe helper used by the durable storage-engine stores

|RouterFunction
|Functional WebFlux pattern for defining HTTP routes programmatically

|CompletableFuture
|Standard Java async — bridge between pure domain and reactive infrastructure

|Flux<DataBuffer>
|Reactive type for binary streaming in WebFlux

|S3ObjectWrite
|Domain interface: object save contract without reactive types

|S3ObjectContent
|Domain interface: object read contract without reactive types

|Cluster Node Runtime
|Implementation-informed fixed-slice runtime instantiated as nodes A, B, and C. Each JVM co-locates one Ratis metadata voter and one independent whole-object replica server with separate stable-identity, Ratis, object, temporary, and runtime roots (ADR 0028)

|Cluster Control Plane
|Internal Apache Ratis role that orders bucket and object-reference generations in the implemented fixed slice. ADR 0027's wider membership, topology/policy epoch, fencing/tombstone, and durable-job authority remains future scope

|Cluster Data Plane
|Internal grpc-java role that directly transfers checksum-valid immutable whole-object replicas between A/B/C for fixed `N=3/W=2`; it is not an external object API. Chunked, multipart, and EC transfer remain unimplemented cluster capabilities

|Node UUID
|Stable persisted identifier recovered from a node's identity root. The validated fixed slice preserves it across complete JVM restart and seed reordering; it must not be cloned into concurrent live nodes

|Fixed bootstrap manifest
|Identical A/B/C UUID and control/data address declaration used to initialize the first three-voter slice. It cannot add, replace, promote, demote, or remove members and must not rewrite recovered non-empty consensus state

|Static seed
|Operator-configured bootstrap address used to contact the fixed group; seed order is restart-tested but is neither a membership transition nor an availability oracle

|Metadata voter
|Apache Ratis consensus participant storing and voting on the authoritative control log. The fixed slice co-locates exactly one voter with one replica server in each A/B/C JVM

|Control log
|Consensus-ordered history that contains bucket and object-reference generations in the fixed slice; whole-object payload bytes are excluded. The broader ADR 0027 authority remains a target, not an implemented-scope claim

|Object-reference generation
|Consensus-committed record naming verified immutable replicas for one object generation. In the fixed slice it is proposed only after `W=2` checksum-valid durable acknowledgements and is the authoritative basis for reads

|Fencing
|Rejection of stale node incarnations, coordinators, operations, or topology/policy epochs so they cannot publish conflicting or resurrected state

|Topology epoch
|Consensus-controlled version of the parent-linked `zone → rack → host → disk-set → device` placement snapshot

|Write quorum (`W`)
|Minimum number of selected targets that must durably acknowledge checksum-valid immutable data before reference publication; the implemented fixed whole-object slice uses `W=2` of `N=3`

|Valid-data read quorum
|Number of checksum/length-valid sources required after consensus resolves the committed generation. The fixed failover slice validates exact bytes from a referenced whole-object replica; alternate-source repair behavior remains future work

|Degraded write
|A write acknowledged below its declared placement/durability threshold. The fixed EP-10 slice rejects publication below `W=2` or without control quorum

|CycloneDX SBOM
|Planned release inventory of application components emitted and schema-validated in both JSON and XML

|SPDX-normalized license inventory
|Planned dependency-license report preserving package/version identity, detected and concluded licenses, source, and unresolved/manual-review results

|Enterprise Production Readiness Plan (EP-0..EP-11)
|`PLAN.md` roadmap sequencing governance, security, durability, streaming, space management, operability, performance, admin panel, cluster architecture, and optional gateways before broader production claims

|Killer-App Track (KA-1..KA-6)
|`PLAN.md` / `docs/positioning.md` roadmap for competitive differentiation: positioning and MIT licensing, ecosystem conformance, data-lake readiness (S3 Tables), eventing/tiering/federation, distribution, and public proof

|ARC42
|Architecture documentation template in 12 sections

|ADR
|Architecture Decision Record (Michael Nygard format)

|C4 Model
|Architecture description method with 4 levels (Context, Container, Component, Code)

|Structurizr DSL
|Domain-specific language for modeling C4 diagrams
|===
