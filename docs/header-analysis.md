## 4. Handler-Level Anti-Patterns (Architecture Violations)

### 4.1 Handler-local state stores (ConcurrentHashMap)

| Handler | Store | What it holds | Problem |
|---------|-------|---------------|---------|
| `S3ObjectMetadataHandler` | `objectAclStore` | ACL per object | ACL should be persisted in domain/repository |
| `S3ObjectMetadataHandler` | `objectTagStore` | Tags per object | Tags should be persisted in domain/repository |
| `S3BucketMetadataHandler` | `bucketAclStore` | ACL per bucket | ACL should be persisted in domain/repository |
| `S3BucketMetadataHandler` | `bucketTagStore` | Tags per bucket | Tags should be persisted in domain/repository |

### 4.2 Handler-level ETag computation (S3MultipartHandler)

| Line | Code | Problem |
|------|------|---------|
| `uploadPartCopy` | `DigestUtils.md5DigestAsHex(...)` | Handler generates ETag — should be repository |
| `uploadPart` | `DigestUtils.md5DigestAsHex(...)` | Handler generates ETag — should be repository |
| `completeMultipartUpload` | `DigestUtils.md5DigestAsHex(...)` | Handler generates ETag — should be repository |

### 4.3 Handler-level bucket existence checks

Already partially addressed in S3ObjectOperationsHandler (uses service), but some handlers still do direct bucket lookups followed by manual error handling instead of letting the service return domain errors.

### 4.4 Handler-level header parsing duplication

The same SSE header parsing logic is duplicated in:
- `S3ObjectOperationsHandler.parseEncryptionConfiguration()` (lines 516-522)
- `S3ObjectMetadataHandler.updateObjectEncryption()` (lines 154-160)

Both parse the same 7 SSE headers independently. Should be unified into a single utility.

## 5. Proposed Refactoring Plan (Revised Phase 1)

Instead of the original monolithic Phase 1, break into focused sub-phases:

### Phase 1a — Header Enum Foundation
- ✅ **DONE** — S3Header enum created with 92 headers in 22 categories
- Next: Create `HeaderExtractor` utility class that uses `S3Header` to extract headers from requests

### Phase 1b — Remove Handler-Local State (ConcurrentHashMap)
- Move ACL and tagging storage from handler ConcurrentHashMap to domain/repository
- `S3ObjectMetadataHandler.objectAclStore` → domain ACL on S3Object
- `S3ObjectMetadataHandler.objectTagStore` → domain tagging on S3Object
- `S3BucketMetadataHandler.bucketAclStore` → domain ACL on Bucket
- `S3BucketMetadataHandler.bucketTagStore` → domain tagging on Bucket

### Phase 1c — Remove Handler-Level ETag Computation
- Move ETag computation from `S3MultipartHandler` to repository
- Repository computes ETag from content hash on store

### Phase 1d — Unify SSE Header Extraction
- Create single `HeaderExtractor.parseSseHeaders()` used by all handlers
- Remove duplicated SSE parsing in S3ObjectOperationsHandler and S3ObjectMetadataHandler

### Phase 1e — Add Missing Copy Object Headers
- Add x-amz-copy-source-range, x-amz-copy-source-version-id, x-amz-copy-source-if-*, x-amz-metadata-directive, x-amz-tagging-directive, x-amz-object-lock-directive to CopyObject and UploadPartCopy

### Phase 1f — Add Missing Conditional Headers (GetObject, HeadObject)
- Add Range, If-Match, If-None-Match, If-Unmodified-Since, If-Modified-Since handling

### Phase 1g — Add Missing ACL/Grant Headers
- Add x-amz-grant-* header extraction to PutObjectAcl, PutBucketAcl, PutObject, CopyObject

### Phase 1h — Add Missing Object Lock Headers
- Add x-amz-object-lock-legal-hold, x-amz-object-lock-mode, x-amz-object-lock-retain-until-on to PutObject, CopyObject, CreateMultipartUpload

## 6. Header Extractor Utility (Next Step)

The enum is ready. The next step is to create a `HeaderExtractor` utility class that:
1. Takes a `ServerRequest` and extracts all known S3 headers into typed DTOs
2. Uses `S3Header.fromHeaderName()` for lookup
3. Provides methods like `parseSseHeaders(request)`, `parseChecksums(request)`, `parseAclHeaders(request)`, `parseCopyHeaders(request)`, `parseObjectLockHeaders(request)`, `parseConditionalHeaders(request)`
4. Each method returns a typed result object that handlers can consume directly

This utility eliminates header parsing duplication across handlers and ensures every header is extracted exactly once.
